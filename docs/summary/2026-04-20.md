## 주요 변경사항

### [추가] [Planner-Executor] 아키텍처 설계 + Phase 1~4 완성
- **설계 (Design Doc)**: 현재 단순 Intent classifier + 1-hop HANDOFF 구조를 진정한 오케스트레이션으로 업그레이드. Planner(요청→Task DAG 분해) → Executor(병렬/순차 실행) → Synthesizer(결과 합성) 3단계 분리. Feature flag `PLANNER_ENABLED`로 기존 경로와 공존.
- **Phase 1 (인프라)**: `Task`, `Plan`, `TaskStatus` dataclass + `Blackboard` 공유 저장소 추가. 기존 코드 영향 0 (순수 additive).
- **Phase 2 (Planner)**: `planner.py` 신규. Sonnet 기반 Task DAG 분해, few-shot 5개, JSON 파싱/검증/fallback. Shadow 모드로 기존 경로와 병렬 실행. 유닛 8/8 PASS.
- **Phase 3 (Executor+Synthesizer)**: `executor.py` (DAG 위상정렬 + asyncio.gather 병렬, MAX_PARALLEL=10, TIMEOUT=300s) + `synthesizer.py` (Haiku 기반 응답 합성, is_trivial passthrough). 전 경로 orchestrator 통합. 유닛 13/13 PASS.
- **Phase 4 (실 Bedrock 검증)**: 10개 시나리오 실 호출 검증. **10/10 PASS** — 프롬프트 이터레이션 불필요. 복합 7-task DAG 정확히 분해. 토큰 26,713 / 비용 ~$0.11 / 평균 지연 4.9초.

### [추가] [XlsxWorker/xlsx_simple] 단일 합성 MCP 도구 `create_xlsx`
- **근본 원인**: `excel-mcp-server`의 2-step workflow(`create_workbook` → `write_data_to_excel`)가 Sonnet 4.6의 multi-call 불안정성과 결합
- **방향 전환**: 방어 대신 **단순화**. `backend/app/mcp_servers/xlsx_simple/server.py` 신규 (≈110줄, openpyxl 직접 호출)
- **도구 1개** `create_xlsx(filepath, headers, rows, sheet_name)` — 파일 생성+데이터 쓰기+저장을 single-call로 완결
- **프롬프트 최상단에 "신규 생성 = create_xlsx 하나만" 원칙 명시**
- **5차 회고**: Circuit Breaker 추가 — create_xlsx 성공 후 모든 xlsx 쓰기 도구 호출을 short-circuit하여 LLM의 강박적 재호출 무력화
