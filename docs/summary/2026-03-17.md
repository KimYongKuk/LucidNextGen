## 기술적 세부사항

### 토큰 최적화 아키텍처
**문제점 분석**:
- MailWorker inbox(50건) ≈ 15,000자 → 일괄 6K truncation 시 ~18건만 전달 → 검색 오답 발생
- ApprovalWorker doc_body HTML: 76KB 중 CSS/태그가 3~5K 소비 → 유효 콘텐츠 부족

**해결 방식**:
```
이전 결과 → compact_keep_recent_pairs(최근 N쌍 원본, 나머지 200자 압축)
         → 도구별 차등 truncation(목록 16K / 상세 6K / HTML 10K)
         → LLM 호출
```

### HANDOFF 메커니즘 흐름
```
워커 A 응답 → <!--HANDOFF:intent_B--> 마커 감지
           → Orchestrator: 선행 워커 B 실행
           → 결과를 [이전 단계 데이터]로 히스토리 주입
           → 워커 A 재실행 (is_handoff_target=True로 HANDOFF 프롬프트 미주입)
```

**안전장치**:
1. 선행 워커는 HANDOFF 프롬프트 미주입 (무한 루프 방지)
2. Orchestrator `not context.get("is_handoff_target")` 체크
3. `handoff_intent != intent` 검증 (자기 자신 HANDOFF 방지)

## 후속 과제

- **성능 모니터링**: HANDOFF 체이닝 시 응답 시간 약 2배 증가 → 사용자 체감 영향도 측정 필요
- **HTML 파싱 개선**: 현재 regex 기반 태그 제거는 복잡한 중첩 구조에서 불완전 → 필요시 BeautifulSoup 도입 검토
- **HANDOFF 확장**: 현재 최대 1단계 체이닝만 지원 → 2단계 이상 필요 시 재설계 필요
- **테스트 케이스 추가**: 도구별 truncation 경계값(16K/6K/10K)에서의 응답 품질 검증
