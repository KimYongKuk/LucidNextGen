## 기술적 세부사항

### CLARIFY 인텐트 아키텍처
```
모호한 요청 (도메인 키워드 부재)
  ↓
Intent Classifier → CLARIFY 감지
  ↓
Orchestrator: context에 clarify_mode=True 주입
  ↓
DirectResponseWorker: "어디에서 찾아볼까요?" 안내 응답
  ↓
사용자 재입력 → 명확한 인텐트로 재분류
```

### 메모리 압축 알고리즘
```
Key facts 개수 > 100
  ↓
_consolidate_user_facts() 호출
  ↓
Haiku LLM: 신원정보 보존 > 명시적 요청 보존 > 유사 병합 > 모호/일회성 삭제
  ↓
응답 파싱 → 원문 매칭 (메타데이터 보존)
  ↓
실패 시 FIFO 폴백 (안전장치)
```

### JSON 파싱 호환성
- **읽기**: bare list `[{...}]` + dict 래퍼 `{"facts": [...]}` 모두 지원
- **쓰기**: dict 래퍼 형식으로 통일 (향후 일관성 유지)

## 후속 과제

1. **CLARIFY 인텐트 모니터링**: 실제 사용자 요청에서 CLARIFY 감지율 및 정확도 추적, 필요시 프롬프트 미세조정
2. **메모리 압축 효율성 검증**: 장기 사용자(100+ facts)에서 압축 후 메모리 품질 평가, Haiku 프롬프트 최적화
3. **레거시 데이터 마이그레이션**: 기존 bare list 형식 메모리를 dict 래퍼로 일괄 변환 (선택사항)
4. **Fallback 범위 확대**: 현재 제안 범위(전자결재/메일/게시판/사내문서/VOC/웹) 외 추가 도메인 검토
