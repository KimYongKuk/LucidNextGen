# 일일 개발 보고서 — 2026-04-01

## 오늘의 요약
LFON 캘린더 API를 연동한 **CalendarWorker**를 신규 추가하여 일정 조회/등록/삭제 기능을 구현했으며, 권한 기반 접근 제어와 비공개 일정 필터링을 적용했습니다. 동시에 **XlsxWorker**의 `create_workbook` 반복 호출 버그를 워크플로우 지침 강화와 에러 규칙 개선으로 해결했습니다.

## 주요 변경사항

### [추가] CalendarWorker — 캘린더 일정 관리
- **모듈**: `backend/app/agents/workers/calendar_worker.py` (신규)
- **MCP 서버**: `backend/app/mcp_servers/calendar_mcp_server.py` (6개 도구)
  - `get_my_calendars`: 내 캘린더 + 관심 캘린더 목록
  - `get_user_public_calendars`: 특정 사용자의 공개 캘린더 조회
  - `get_calendar_events`: 기간별 일정 조회 (다중 캘린더)
  - `get_event_detail`: 일정 상세 조회
  - `create_event`: 일정 등록 (내 캘린더만)
  - `delete_event`: 일정 삭제 (내 일정만)
- **인텐트 분류**: `intent_classifier.py` 수정 — "캘린더", "일정 조회/등록/삭제", "스케줄" 등 키워드 추가
- **상태 관리**: `state.py` 수정 — `Intent.CALENDAR` 추가, 워커 레지스트리 등록
- **설정**: `mcp_config.json`에 calendar_server 엔트리 추가

### [수정] XlsxWorker — create_workbook 반복 호출 버그 수정
- **모듈**: `backend/app/agents/workers/xlsx_worker.py`
- **개선 사항**:
  1. 시스템 프롬프트에 워크플로우 섹션 추가 (create_workbook → write_data_to_excel 순서 명시, 1회 호출 강조)
  2. 에러 규칙 #7 개선 (Error: 접두사로 시작하는 결과만 에러로 판단)
  3. `_created_workbook_path` 속성으로 중복 호출 감지 및 가드 로직 추가

## 기술적 세부사항

### CalendarWorker 권한 모델
**캘린더 레벨 (visibility 기반)**:
- 내 캘린더: 무조건 접근 가능
- 관심 캘린더: feed 등록 상태 확인 후 접근 (수락 완료)
- 공개 캘린더: 관심 등록 여부 무관하게 접근 가능
- 비공개/수락 전: 접근 불가

**일정 레벨 (event.visibility 기반)**:
- 타인 캘린더의 private 일정: creator 또는 attendee만 상세 조회 가능
- 그 외: 시간만 표시 + "🔒 비공개 일정" 마스킹

### 인증 및 식별
- SSO 서비스 계정 → API 호출
- 사번(employee_number) → GO user.id 매핑 (프로세스 수명 캐싱)
- `prepare_tools()`에서 employee_number 강제 주입 (6개 도구 전부)
- v_user_info_mapping VIEW 재사용

### 캐싱 전략
- 사번 → GO user.id 매핑: 프로세스 수명 캐싱
- 캘린더 피드(목록): 5분 TTL 캐싱 (등록/삭제 시 무효화)

### XlsxWorker 버그 원인 분석
1. **워크플로우 지침 부재**: LLM이 create_workbook 후 다음 단계를 명확히 인식하지 못함
2. **에러 규칙 오작동**: 성공 응답도 "실패"로 오인하여 반복 호출 유발
3. **DEDUP 혼란**: 파일명 변경으로 LLM이 에러로 해석
