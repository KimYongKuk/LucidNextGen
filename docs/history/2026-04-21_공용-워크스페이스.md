# 2026-04-21 공용 워크스페이스 (Public Workspace)

## 개요
운영자(A2304013)가 본인의 워크스페이스를 전 사용자에게 읽기 전용으로 공개할 수 있는 공용 워크스페이스 기능을 추가. 지식베이스 / 시스템 프롬프트 / (향후) 에이전트는 공유되지만, 세션·채팅·메모리는 기존 설계상 사용자별로 자연스럽게 격리된다.

## 상세 내용

### 아키텍처 결정
- **공용 등록 방식**: 별도의 "공용 워크스페이스 생성" UI를 두지 않고, 운영자가 본인 소유 워크스페이스의 설정에서 `is_public` 체크박스를 토글하는 방식. 운영 중 테스트용 on/off 를 쉽게 반복할 수 있음.
- **격리 원칙**: 공유되는 것 = 지식베이스(ChromaDB `workspace_{uuid}`) · 시스템 프롬프트(`instructions`) · (향후) 에이전트. 분리되는 것 = 세션(`chat_sessions.user_id`) · 채팅 로그(`chat_log_new.user_id`) · 워크스페이스 메모리(`workspace_memory (workspace_id, user_id)` UNIQUE).
- **권한 매트릭스**:
  | 작업 | 소유자 | 다른 사용자 (is_public=1) | 다른 사용자 (is_public=0) |
  |------|--------|---------------------------|---------------------------|
  | 목록에 노출 | 내 워크스페이스 섹션 | 공용 워크스페이스 섹션 | — |
  | 단건 조회 (`GET {uuid}`) | ✅ | ✅ | 403 |
  | 파일 목록 조회 | ✅ | ✅ | 403 |
  | 파일 업로드 / 삭제 | ✅ | 403 | 403 |
  | 메타데이터 수정 | ✅ | 403 | 403 |
  | 워크스페이스 삭제 | ✅ | 403 | 403 |
  | `is_public` 전환 | ✅ (운영자만) | 403 | 403 |

### 백엔드 변경
- `OPERATOR_USER_IDS` 환경변수(기본 `A2304013`) 추가. 프론트엔드의 `NEXT_PUBLIC_OPERATOR_USERS`와 동일한 의미. `_is_operator(user_id)` 헬퍼로 워크스페이스 생성/수정 시 `is_public=true` 요청을 **백엔드에서 한 번 더 검증**(방어적 코딩, UI만 믿지 않음).
- `GET /v1/workspaces/public` — 인증 없이 전 사용자에게 공개. 경로 순서 주의: `/public` 이 `/{workspace_uuid}` 보다 앞에 위치해야 함 (FastAPI는 등록 순서대로 매칭).
- `GET /v1/workspaces/{uuid}` 및 `GET /v1/workspaces/{uuid}/files` 에 `workspace["user_id"] != user_id and not workspace.get("is_public")` 조건 적용. 읽기만 완화, 쓰기는 기존 소유자 검증 그대로.

### 프론트엔드 변경
- 사이드바는 두 개의 SWR key(`/api/v1/workspaces?user_id=...`, `/api/v1/workspaces/public`)로 독립 조회. 운영자가 공용으로 만든 워크스페이스는 **양쪽 모두**에 나타남(본인 소유이기 때문). 비운영자는 공용 섹션에서 조회만 가능하며 설정(톱니) 버튼 미노출.
- 워크스페이스 모달의 General 탭에 체크박스 블록. `isOperatorUser()` 로만 렌더되고, 상태 변경 시에만 `update_payload.is_public` 포함(불필요한 전송 방지). 공용 워크스페이스 아이콘은 `Globe` 로 시각화.

### 채팅 경로 (변경 없음)
- 채팅 처리 시 `orchestrator.py` 가 `workspace_service.get_workspace_by_uuid()` 를 호출하는데, 이 함수는 원래 소유권 검증을 하지 않음 → 공용 워크스페이스도 그대로 컨텍스트 주입됨. `search_workspace_docs` 는 UUID 기반 ChromaDB 컬렉션을 직접 조회하므로 사용자와 무관. 결과적으로 채팅 흐름에는 어떤 수정도 필요 없었다.

## 결정 사항 및 주의점
- **이중 검증**: 프론트가 비운영자에게 체크박스를 가리지만, 백엔드에서도 한 번 더 `_is_operator()` 를 확인. 사내 사용자라도 API 직접 호출 시 반드시 걸러야 한다.
- **삭제 시 영향**: 운영자가 공용 워크스페이스를 삭제하면 다른 사용자의 `chat_sessions.workspace_id` 가 고아(orphan)가 된다. 이는 기존 워크스페이스 삭제에도 동일하게 존재하는 문제로, 이번 범위 밖.
- **메모리 격리**: `workspace_memory` 의 `(workspace_id, user_id)` UNIQUE 제약 덕분에 별도 로직 없이 자연 격리. 공용 워크스페이스에서 각 사용자는 자신만의 롤링 요약을 축적한다.
- **`is_public` on/off 시**: `is_public=1 → 0` 으로 내릴 경우, 기존에 해당 워크스페이스로 진입한 비소유자 사용자의 세션은 그대로 남지만 다음 진입 시 단건 조회 403 으로 막힌다. 사이드바에서도 사라진다.
- **운영자 복수화**: 향후 여러 운영자가 필요할 경우 `OPERATOR_USER_IDS=A2304013,B1234567` 처럼 콤마 구분으로 확장. 프론트·백엔드 환경변수 두 곳 모두 업데이트 필요.
- **`workspaces.is_public` 컬럼 위치**: `AFTER instructions` 로 두어 스키마 docs 순서와 일치.

## 후속 과제
- 운영자가 공용 워크스페이스의 채팅 사용 현황(사용자 수, 질문 유형 등)을 볼 수 있는 대시보드
- "이 워크스페이스는 공용입니다" 헤더/뱃지 — 현재는 사이드바 Globe 아이콘만 있음
- 공용 워크스페이스 삭제 시 고아 세션 감지/안내
