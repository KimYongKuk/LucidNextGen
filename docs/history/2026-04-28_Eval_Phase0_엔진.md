# 2026-04-28 Eval Phase 0 — 골든 케이스 회귀 테스트 엔진

## 개요
워커 회귀를 결정적으로(이진 판정) 잡기 위한 골든 케이스 회귀 테스트 엔진을 구축. 운영 백엔드 e2e를 그대로 호출하되 `X-Eval-Auth` 헤더로 JWT를 우회하고, 응답 데이터는 `metadata.is_eval=true`로 마킹되어 운영 통계·메모리에서 자동 분리됨. 케이스의 source of truth는 `backend/tests/eval/cases/*.yaml`(git), DB는 결과/triage 메타만 보관.

## 상세 내용

### 인증·격리 (운영 영향 0 보장)
- 새 헤더 `X-Eval-Auth: <secret>` + `X-Eval-Empno: <empno>` — secret이 backend의 `EVAL_API_KEY` 환경변수와 일치할 때만 우회. 일치하지 않으면 401.
- 인증 dict의 `source: "eval"` 키로 chat 핸들러가 metadata에 `is_eval=true` 추가.
- `_save_chat_log_background`: metadata.is_eval이 True면 워크스페이스 메모리·글로벌 사용자 메모리 트리거 모두 조기 return → 메모리 오염 0.
- `report_service.py`의 모든 운영 통계 SQL은 `_EXCLUDED_USERS_SQL = _EXCLUDED_USERS_SQL_USER + _NOT_EVAL_SQL` 단일 fragment를 그대로 사용하므로 자동으로 eval 트래픽 제외.

### Eval 엔진 (`backend/tests/eval/`)
- **runner.py**: httpx로 `POST /api/v1/chat/message/stream` 호출, SSE 라인을 파싱해 `intent_classified` / `tool_used` / `content` 이벤트만 모은 `CaseResult` 반환.
- **assertions.py**: 9종 assertion (intent, worker, tool_called, tool_not_called, contains, contains_any, not_contains, min_length, max_length, max_time_ms). 실패 키 list와 사람용 detail을 `Verdict`로 반환.
- **reporter.py**: 콘솔(케이스별 1줄 + 요약), JSON(`results/<date>_<runid>.json` + `latest.json`), 옵션 DB 적재.
- **run.py**: CLI. `--all`/`--worker <name>`/`--case <id>` 라우팅, `--persist` 시 DB 적재, `--triggered-by`로 cron/수동 구분.

### 케이스 yaml 형식
- 한 워커당 yaml 파일 1개, 케이스 dict의 list. 필수 키: `id`, `input`. 선택: `description`, `user_id` (`{{SELF_EMP_NO}}` 변수 치환), `chat_mode`, `message_history`, `assertions`.
- `assertions`는 9종 키를 자유 조합. 결정적이지 않은 자연어 유사도 비교는 의도적으로 미지원 (Layer 3에서).

### POC 케이스 (cases/direct.yaml — 3건)
1. `direct_basic_greeting` — "안녕하세요" → DIRECT 라우팅, 도구 미호출, 회피 응답·내부 jargon 금지
2. `direct_self_introduction` — "너는 누구야?" → 루시드AI 아이덴티티 응답, "Claude"/"Anthropic" 노출 금지
3. `direct_simple_math` — "1부터 10까지 더한 값" → 정답 "55" 포함, 검색 도구 미호출

## 결정 사항 및 주의점

### 결정성 우선
- assertion은 intent/worker/tool 호출/명백한 키워드 같은 이진 판정만 사용. "응답이 자연스러운가" 같은 LLM-as-judge는 Layer 3 도입 후. 첫 단계에서 assertion을 너무 강하게 잡으면 false positive로 운영자가 외면하게 됨 — 결정적인 것만 잡고 자주 깨지면 헐겁게 조정.

### Source of truth
- yaml이 source, DB는 캐시·결과·triage. UI에서 케이스 추가/편집은 Phase 3에서 yaml 자동 commit으로 구현 예정. 첫 단계는 IDE에서 yaml 직접 편집.

### cron·알림은 Phase 0.5에서
- 본 Phase 0은 엔진 + POC 케이스까지만. cron 등록(`schtasks`), 어제 대비 신규 실패 알림, 관리자 알림함 누적은 후속 단계. CLI exit code 0/1로 cron이 외부에서 trigger 가능한 형태로 노출만 해둠.

### 사번 의존 워커
- direct/web_search/visualization/url_fetch/youtube/corp_rag(쿼리 안 던지는 한)는 사번 의존 없음 — `EVAL_DEFAULT_EMPNO`(또는 임의 값)으로 동작. 메일/결재/캘린더/예약/메일 워커는 본인 사번 사용 필요(데이터 변동 고려해 키워드 매칭 헐겁게).

### 운영 영향 검증
- chat_log_new에는 eval row가 들어가나 metadata.is_eval=true로 분리됨. `/admin/report`의 모든 통계 SQL은 자동 제외. 검증 방법: 임의 케이스 실행 후 `/admin/report`의 KPI가 1건이라도 늘면 `report_service.py`의 누락 쿼리 색출.

### 다음 단계 (Phase 0.5 / Phase 1)
- Phase 0.5: cron 등록 + 어제 대비 신규 실패 시 메일 알림 + 관리자 알림함 row insert.
- Phase 1: `/admin` 통합 콘솔에 검증 탭 추가(트렌드 차트, 실행 히스토리, 실패 케이스 모달).
- Phase 2 이후: triage 워크플로, 운영 chat → 골든 케이스 promote, 실시간 스트림.
