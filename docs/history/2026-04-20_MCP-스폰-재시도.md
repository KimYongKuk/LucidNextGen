# 2026-04-20 MCP 어댑터 일시 실패 재시도

## 개요

Windows에서 MCP 서버 스폰 경쟁으로 다수 서버가 동시에 transient fail하는 장애가 재발. 특히 캐시 TTL 만료 직후 18개 서버를 동시 스폰하는 시점에 `calendar_server, reservation_server, chart_generator, approval_server, excel_server` 5개가 한꺼번에 실패한 사례(운영 로그 2026-04-20 09:37)가 확인됨. 해당 시점 이후 CalendarWorker가 4개 도구만 받게 되어 회의실 예약/캘린더 등록 기능이 마비되는 연쇄 장애로 이어짐.

기존에는 개별 서버 실패 시 **재시도 없이** 빈 도구 리스트만 리턴하고 캐시 TTL을 60s로 단축하는 완화책만 있었음. 근본 대책이 아니어서 다음 리프레시 때도 같은 경쟁이 재발했음.

이번 수정으로 `_load_server_tools`에 **지수 백오프 재시도**를 추가하여 일시적 스폰 경쟁을 자동 복구한다.

## 변경 파일 요약

| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/adapters/mcp_adapter.py` | 수정 | `MAX_RETRY_ATTEMPTS`, `RETRY_BASE_DELAY` 상수 추가 + `_load_server_tools`에 재시도 루프 도입 |

## 상세 내용

### 새 클래스 상수

```python
# 일시 실패 재시도 — Windows 프로세스 스폰 경쟁은 재시도로 해결되는 경우가 대부분
MAX_RETRY_ATTEMPTS: int = 2  # 총 시도 횟수 (초기 1회 + 재시도 1회)
RETRY_BASE_DELAY: float = 1.0  # 초 단위, 지수 백오프 base
```

### 재시도 동작 흐름

1. `_load_server_tools`가 세마포어(MAX_CONCURRENT_LOADS=4) 안에서 서버별 도구 로드 시도
2. `TimeoutError` 또는 일반 `Exception` 발생 시:
   - 영구 에러(`FileNotFoundError, PermissionError, NotADirectoryError`) → 블랙리스트 직행 (기존 동작 유지)
   - 그 외 → `RETRY_BASE_DELAY * (2 ** attempt)` 백오프 후 재시도 (1s → 2s)
3. `MAX_RETRY_ATTEMPTS` 모두 소진 시 마지막 에러 로깅 후 빈 리스트 리턴
4. 재시도 동안 세마포어 슬롯 점유 유지 → 동시성 초과 방지

### 핵심 설계 결정

- **동시성 제한 유지**: `MAX_CONCURRENT_LOADS=4`는 그대로. 경쟁을 **완전 제거**하기보다 **복구 가능하게** 만드는 방향
- **영구 에러 보존**: 블랙리스트 분류 로직은 기존 유지하여 의미없는 재시도 방지
- **최대 지연**: 1s + 2s = 3s. 스폰 타임아웃(10s) 고려하면 총 최악 23s까지 증가 가능하나, 실제로는 첫 시도 성공이 대부분이라 영향 미미

## 결정 사항 및 주의점

### 왜 동시성 제한을 더 낮추지 않았나

- 이전에 `MAX_CONCURRENT_LOADS`를 무제한 → 4로 제한한 전례(commit faaa194)가 있음
- 4 → 2로 더 낮추면 콜드 스타트 시 로딩 시간이 거의 2배 증가 (운영 재시작 영향)
- **재시도 레이어 추가가 더 저침습적**이며, 정상 경로 성능은 영향 없음
- 재시도로도 해결 안 되는 반복 실패가 관측되면 그때 동시성 하향 검토

### 검증 방법

1. **정상 경로 회귀**: 로컬 `python -c "MCPAdapter.get_tools()"` 호출 → 91개 도구 로드, 0 blacklisted
2. **일시 에러 재시도**: `adapter._client.get_tools`를 1회 실패하도록 몽키패치 → 재시도 후 성공 확인, `fetch` 호출 2회 확인
3. **영구 에러 격리**: `FileNotFoundError` 주입 → 재시도 없이 1회만 호출 후 블랙리스트 진입 확인

모두 통과 (로컬 기준).

### 향후 모니터링

- 로그에서 `retrying in 1.0s...` 메시지 빈도 관찰 — 잦으면 `MAX_CONCURRENT_LOADS` 하향 검토
- 여전히 `servers transiently failed` 경고가 발생하면 재시도로도 해결 안 되는 case → 개별 서버의 스폰 시퀀스 점검 필요

### 관련 후속 작업

- 본 수정은 Planner-Executor 아키텍처 개편의 선결 hotfix (docs/history 별도 Design doc 예정)
- 동시 스폰 실패가 완전히 해소되지 않을 경우, 실패 서버만 **2-pass 순차 재시도**를 추가로 고려 (현재 패치로 충분하면 불필요)
