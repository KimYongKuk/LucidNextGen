# 2026-05-06 워크스페이스 영속 파일 + MISO Workflow 출처 분리

## 개요
MISO Workflow 호출 시 파일 입력 출처를 **채팅 첨부(일회성)** 와 **워크스페이스 영속(고정)** 두 갈래로 명확히 분리. 시나리오:
1. **사용자 직접 요청** — 매번 다른 파일로 호출. 채팅창에 첨부 → `{{latest_file}}`/`{{recent_files}}` 토큰
2. **Cron 자동 호출** (후속 Phase) — 매번 같은 파일로 호출. 워크스페이스 영속 파일 → `{{workspace_files}}` 토큰

기존엔 워크스페이스 업로드 시 ChromaDB 청크/임베딩만 저장되고 원본은 보존되지 않아 외부 워크플로우에 넘길 수단이 없었음 (chromadb_service의 temp 파일은 finally에서 즉시 삭제). 이번에 워크스페이스 업로드 경로에 명시적 원본 보존 단계를 추가.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/services/workspace_service.py | 수정 | `WORKSPACE_UPLOAD_DIR` 신규(`data/workspace_uploads/{ws_uuid}/{file_id}/{filename}`). `upload_file`/`upload_file_from_content`에 원본 디스크 저장 추가. `delete_file`/`delete_workspace`에 원본 정리 추가. `list_persisted_files_by_uuid` staticmethod 신규 (워커가 DB 조회 없이 워크스페이스 영속 파일 목록 가져옴) |

## 상세 내용

### 1. 워크스페이스 영속 파일 보존 인프라
경로: `backend/data/workspace_uploads/{ws_uuid}/{file_id}/{filename}`
- file_id 단위 격리 → 같은 파일명 재업로드 충돌 방지, 삭제 시 디렉토리 통째 제거
- 파일명 sanitize: `/`, `\`, `..` 치환
- 보존 실패해도 ChromaDB 업로드는 성공으로 응답 (RAG는 동작) — non-fatal warning 로그
- 워크스페이스 삭제 시 디렉토리 통째 제거 (워크스페이스 단위 정리)

### 2. MISO Workflow 토큰 출처 매트릭스

| 토큰 | 타입 | 출처 | 사용 케이스 |
|------|-----|-----|-----------|
| `{{message}}` | text/paragraph | 사용자 발화 | 일반 |
| `{{message_lines}}` | list | 사용자 발화 (줄바꿈 split) | 일반 |
| `{{message_number}}` | number | 사용자 발화 (숫자 추출) | 일반 |
| `{{latest_file}}` | file | **채팅 첨부** (user_uploads, mtime 최신 1개) | 사용자 직접, 일회성 |
| `{{file:파일명}}` | file | **채팅 첨부** (명시 파일명) | 사용자 직접, 일회성 |
| `{{workspace_file:파일명}}` | file | **워크스페이스 영속** (명시 파일명) | 사용자 직접, 고정 파일 |
| `{{recent_files}}` | files | **채팅 첨부** (시간 윈도우, 기본 10분) | 사용자 직접, 묶음 |
| `{{recent_files:N}}` | files | **채팅 첨부** (mtime 최신 N개) | 사용자 직접, 묶음 |
| `{{workspace_files}}` | files | **워크스페이스 영속** (전체, 최대 10개) | cron, 사용자 정기 분석 |
| `{{workspace_files:N}}` | files | **워크스페이스 영속** (mtime 최신 N개) | cron, 사용자 정기 분석 |

### 3. context 인자 흐름
- `MisoWorker.stream_response(context)`에서 `workspace_id = context.get("workspace_id")` 추출
- `_build_request_body(.., workspace_id=...)` → `_resolve_mapped_value(workspace_id=...)`로 전파
- 워크스페이스 토큰 사용 시 workspace_id가 없으면 빈 결과 → `FileMappingError`로 사용자 안내
- orchestrator가 이미 workspace_id를 context에 넣어주므로 추가 wiring 불필요

## 결정 사항 및 주의점

- **출처 분리 원칙**: 토큰이 출처를 명시 → 사용자가 호출할 때 채팅 첨부 여부와 무관하게 매니페스트 정의대로 동작. fuzzy fallback(`{{any_file}}`) 도입 안 함 — 모호함은 운용 비용.
- **워크스페이스 단일 파일도 명시 파일명만 지원**: 단일 파일 매핑은 "워크스페이스 영속 파일 중 무작위 1개"가 의미 없으므로 `{{workspace_file:..}}` (명시 파일명)만 제공. 단일 파일에 mtime 최신 1개를 자동 매핑하는 동작은 cron에서도 위험(파일이 바뀌면 결과가 달라짐).
- **다중 파일 워크스페이스 옵션**: 영속 파일 전체(`{{workspace_files}}`) 또는 N개. 기본 안전 상한 10개 (`MISO_RECENT_FILES_MAX`).
- **원본 보존 누락 케이스**: 이번 변경 이전에 업로드된 워크스페이스 파일은 디스크에 원본이 없음 → 외부 워크플로우 호출 불가. 사용자가 재업로드하거나, 일괄 백필 스크립트로 ChromaDB 청크에서 추출 시도 가능하지만 손실 우려. 일단 신규 업로드만 보존, 기존 파일은 사용자가 재업로드 권장.
- **cron 시나리오 후속(Phase 3b)**: 결과 라우팅 = 자동 세션 생성 + 알림 deep-link, 분단위 cron 등록 시 경고 안내, 세션 메타데이터에 `auto_generated`/`cron_job_id` 표시. 본 변경은 cron 호출 시 input 매핑이 동작하는 인프라까지만 갖춤.
