# 2026-03-30 대화 요약 임계치 상향 + 프롬프트 강화

## 개요
Haiku 사전 요약(pre-summarization)의 발동 임계치가 너무 낮아(6메시지/5,000자), 옵션 A/B/C 같은 구조화된 선택지가 조기에 요약으로 대체되어 원문이 소실되는 문제를 수정했다. 임계치 상향과 함께 요약 프롬프트를 강화하여, 요약 시에도 구조화된 데이터를 보존하도록 개선했다.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/agents/workers/base_worker.py | 수정 | 임계치 상향(12msg/15K), 요약 프롬프트 구조 보존 강화, 구조화 데이터 판정 확대 |

## 상세 내용

### 문제 상황
- 사용자에게 옵션 A/B/C를 제시한 후, 사용자가 "옵션 A로 가자"라고 선택
- 3~4턴 대화만으로 임계치(6메시지 + 5,000자) 초과 → Haiku가 이전 대화 전체를 요약
- 요약 과정에서 각 옵션의 구체적 내용이 소실 → AI가 "옵션 A의 내용이 없습니다" 응답

### 변경 1: 임계치 상향 (비용-맥락 균형)
| 상수 | Before | After | 의미 |
|------|--------|-------|------|
| SUMMARIZATION_MESSAGE_THRESHOLD | 6 | 12 | 최소 6턴 대화까지 원문 유지 |
| SUMMARIZATION_CHAR_THRESHOLD | 5,000 | 15,000 | 일반 업무 대화 수준에서 요약 미발동 |

### 변경 2: 요약 프롬프트 강화
- **선택지/옵션 개별 보존**: "옵션 A/B/C"는 라벨+핵심 내용을 각각 보존 (뭉뚱그리기 금지)
- **결정 사항 기록**: 사용자가 선택/확정/거부한 내용을 명시적으로 기록
- **"절대 하지 말 것" 섹션 추가**: 선택지를 "여러 옵션을 논의함"으로 축약하지 않도록 강제
- **최대 단어 수 확대**: 800→1000 (구조 보존을 위한 여유)

### 변경 3: 구조화 데이터 판정 확대
`_has_structured_data()` 판정 기준 추가:
- 기존: 마크다운 테이블(| 시작/끝) 3행 이상
- 추가: 옵션/선택지 패턴 2개 이상 (옵션 A, 방안 1, Option B 등)
- 추가: 번호 매긴 목록 3개 이상 (1. / 1) / (1) 등)
- 효과: 감지 시 truncation limit 2,000→6,000자로 확대 → 원문 더 많이 보존

### 토큰 비용 영향
- ReAct 루프 내부 토큰 관리는 별도 메커니즘(`compact_previous_results`)이 담당하므로 무관
- 히스토리 비용은 기존 대비 약 2배 수준 (6턴 원문 유지)
- Prompt Caching이 시스템 프롬프트 반복 전송 비용을 90% 절감

## 결정 사항 및 주의점
- 20msg/30K안은 헤비 워커(XLSX, PPT, Mail)의 ReAct 루프 비용 폭증 우려로 기각
- 12msg/15K + 프롬프트 강화 조합으로 비용-맥락 균형 확보
- 6턴 이후 요약이 발동해도 옵션/선택지 구조가 보존됨
