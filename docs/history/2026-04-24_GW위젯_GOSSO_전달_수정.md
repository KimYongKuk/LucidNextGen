# 2026-04-24 그룹웨어 위젯 GOSSOcookie 전달 수정

## 개요
그룹웨어(mailgw) 페이지에 임베딩된 Lucid AI 위젯에서 캘린더 일정 등록 시 "서버에 연결할 수 없습니다" 오류가 발생. GOSSOcookie가 브라우저에는 존재하지만 백엔드 CalendarWorker까지 전달되지 않아 LFON API가 서비스 계정 POST를 거부한 것이 원인. `/embed/gw` 페이지에서 URL의 `?gosso=` 파라미터를 lucid 도메인의 `gosso` 쿠키로 이식하도록 수정.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| frontend/app/embed/gw/page.tsx | 수정 | 마운트 시점에 URL `?gosso=` → `document.cookie`에 `gosso` 쿠키 세팅하는 `useEffect` 추가 |

## 상세 내용

### 재현 시나리오 (운영 green, 11:12 전후)
- A2604011 사용자가 그룹웨어 위젯에서 "회의실 예약하고 캘린더에도 등록" 요청
- `create_reservation` ✅ 성공 (예약ID 667301)
- `create_event` ❌ 14.8초 후 "오류: 일정 등록에 실패했습니다. 서버에 연결할 수 없습니다."
- 같은 로그에서 A2304013는 동일 경로(`gosso → 없음`)로 성공 → 단순 gosso 유무 문제로 치부되기 쉬우나 실제로는 LFON 쪽 서비스 계정 권한이 사용자별로 달라서 드러난 이슈. 본질은 사용자 쿠키가 아예 전달되지 않던 것.

### 끊김 지점 분석
1. 위젯 JS(`frontend/widget/lucid-chat-widget-gw.js:125-147`): 그룹웨어 origin의 `document.cookie`에서 `GOSSOcookie` 추출 → iframe src에 `?gosso=<값>` 부착. **정상**
2. Next.js `middleware.ts:81-91`: URL `?gosso=`를 `gosso` 쿠키로 저장하는 로직 존재. **그러나 matcher가 `/embed` 경로 제외** (메모에도 명시) → iframe 로드 시 실행 안 됨
3. `frontend/app/embed/gw/page.tsx`: `token`, `empno`, `sid`만 `URLSearchParams`로 추출. **`gosso` 파라미터를 읽지 않음**
4. `frontend/hooks/use-simple-chat.ts:181-186`: `document.cookie`에서 `GOSSOcookie`(LFON 도메인) 또는 `gosso`(middleware 세팅) 탐색. iframe 오리진은 `lucid.landf.co.kr`이라 그룹웨어 도메인의 `GOSSOcookie` 접근 불가, 2·3번 때문에 `gosso` 쿠키도 없음 → **빈 문자열 → body에 `gosso_cookie` 필드 누락**

결과적으로 백엔드는 항상 `gosso_cookie=""`로 수신 → `CalendarWorker.prepare_tools`에서 `gosso → 없음` 로깅 → `_api_request`가 서비스 계정 폴백(`_sso_login()`)만 시도.

### 수정 내용
`frontend/app/embed/gw/page.tsx`에 마운트 시점 1회 실행되는 `useEffect` 추가:

```tsx
useEffect(() => {
  const params = new URLSearchParams(window.location.search);
  const gosso = params.get("gosso");
  if (gosso) {
    document.cookie = `gosso=${encodeURIComponent(gosso)}; path=/; SameSite=Lax; max-age=86400`;
  }
}, []);
```

- `SameSite=Lax`: iframe 내부에서 동일 origin(lucid.landf.co.kr)으로 fetch할 때 포함됨
- `max-age=86400`: 24시간 (실제 만료는 LFON 세션에 종속되므로 클라이언트 TTL은 상한 역할만)
- `HttpOnly` 미설정: `use-simple-chat`이 JS로 읽어야 하므로 의도적으로 off (middleware와 동일 정책)
- `encodeURIComponent`: 쿠키 값의 특수문자 안전 처리

이후 `use-simple-chat.ts:183`의 `document.cookie.match(/gosso=([^;]+)/)` 폴백이 자동으로 값을 픽업하여 POST body에 `gosso_cookie: <값>` 포함 → 백엔드 chat.py → CalendarWorker 컨텍스트에 전달 → MCP 도구 호출 시 사용자 본인 쿠키로 LFON POST.

## 결정 사항 및 주의점

- **왜 middleware matcher에 `/embed` 추가하지 않았나**: `/embed`는 JWT 인증 의무화에서 명시적으로 제외되어 있고, SSO flow와 혼선 위험이 있어 matcher 수정보다 페이지 레벨에서 처리하는 것이 좁고 명확한 변경임. 메모(chat_endpoint_jwt_auth.md)에도 `middleware matcher가 /embed 제외 → iframe은 middleware 안 거침 (의도됨)`으로 기록되어 있음.
- **위젯 JS 변경 없음**: 위젯은 이미 올바르게 `?gosso=` 파라미터를 붙이고 있음. `deploy.bat`이 위젯 JS도 동기화하지만 이번엔 위젯 파일에 변경 없음.
- **A2304013가 이전에 성공한 이유**: 서비스 계정 폴백(`_sso_login()`)이 특정 사용자의 `creator.id` 조합에 대해서만 POST를 허용. 즉 서비스 계정 우회는 불안정한 경로였고, 사용자 본인 쿠키로 POST하는 것이 정상 경로. 본 수정으로 gosso가 전달되면 LFON이 사용자 본인 세션으로 권한 체크하여 일관되게 동작.
- **빈 값 가드**: URL에 `?gosso=` 키만 있고 값이 빈 문자열인 경우 `if (gosso)` 조건으로 쿠키를 덮어쓰지 않음 → 이전 값 유지.
- **후속 검증 포인트**: 배포 후 운영 로그에서 `CalendarWorker … gosso → 있음` 문자열이 나타나는지, `create_event` POST가 400번대 없이 200으로 반환되는지 확인 필요.
