# 2026-04-24 그룹웨어 위젯 GOSSOcookie 전달 수정

## TL;DR (최종 결론 요약)
**실제 원인**: 그룹웨어 운영/개발 서버가 분리되어 있는데 **개발 그룹웨어에서 테스트하고 있었음**. 조회처럼 쿠키 인증 없이 서비스 계정만으로 가능한 작업은 정상, **쿠키 인증이 필수인 쓰기 작업(일정 등록)만 실패**. 개발 그룹웨어 세션의 GOSSOcookie는 운영 LFON API에서 인증되지 않기 때문.

**해결**: 위젯을 **그룹웨어 운영 서버**에 임베드하니 즉시 정상 동작.

**교훈**: 쿠키 인증이 필요한 작업을 개발 그룹웨어에서 테스트하면 안 됨. 운영 그룹웨어-운영 LFON 쌍에서만 정상 동작. 추후 진짜 dev 환경 분리 시 `.env`에 개발 LFON용 `LFON_BASE_URL/USERNAME/PASSWORD` 세트를 별도 구성 필요.

**코드 변경의 의의**: 환경 구성 이슈였지만 진단 과정에서 추가한 2가지 개선은 그대로 유지 — (1) 쿠키 경유 → prop 경로 전환(iframe 환경에서 더 견고), (2) MCP 서버 파일 로거(LFON API 응답 진단 인프라).

## 개요
그룹웨어(mailgw) 페이지에 임베딩된 Lucid AI 위젯에서 캘린더 일정 등록 시 "서버에 연결할 수 없습니다" 오류가 발생. GOSSOcookie가 브라우저에는 존재하지만 백엔드 CalendarWorker까지 전달되지 않아 LFON API가 서비스 계정 POST를 거부한 것이 원인. `/embed/gw` 페이지에서 URL의 `?gosso=` 파라미터를 lucid 도메인의 `gosso` 쿠키로 이식하도록 수정.

## 상세 내용

### 재현 시나리오 (운영 green, 11:12 전후)
- A2604011 사용자가 그룹웨어 위젯에서 "회의실 예약하고 캘린더에도 등록" 요청
- `create_reservation` ✅ 성공 (예약ID 667301)
- `create_event` ❌ 14.8초 후 "오류: 일정 등록에 실패했습니다. 서버에 연결할 수 없습니다."
- 같은 로그에서 A2304013는 동일 경로(`gosso → 없음`)로 성공 → 단순 gosso 유무 문제로 치부되기 쉬우나 실제로는 LFON 쪽 서비스 계정 권한이 사용자별로 달라서 드러난 이슈. 본질은 사용자 쿠키가 아예 전달되지 않던 것.

### 끊김 지점 분석
1. 위젯 JS(`frontend/widget/lucid-chat-widget-gw.js:125-147`): 그룹웨어 origin의 `document.cookie`에서 `GOSSOcookie` 추출 → iframe src에 `?gosso=<값>` 부착. **정상**
2. Next.js `middleware.ts:81-91`: URL `?gosso=`를 `gosso` 쿠키로 저장하는 로직 존재. **그러나 matcher가 `/embed` 경로 제외** (메모에도 명시) → iframe 로드 시 실행 안 됨
3. `frontend/app/embed/gw/page.tsx`: `token`, `empno`, `sid`만 `URLSearchParams`로 추출. **`gosso` 파라미터를 읽지 않음**
4. `frontend/hooks/use-simple-chat.ts:181-186`: `document.cookie`에서 `GOSSOcookie`(LFON 도메인) 또는 `gosso`(middleware 세팅) 탐색. iframe 오리진은 `lucid.landf.co.kr`이라 그룹웨어 도메인의 `GOSSOcookie` 접근 불가, 2·3번 때문에 `gosso` 쿠키도 없음 → **빈 문자열 → body에 `gosso_cookie` 필드 누락**

결과적으로 백엔드는 항상 `gosso_cookie=""`로 수신 → `CalendarWorker.prepare_tools`에서 `gosso → 없음` 로깅 → `_api_request`가 서비스 계정 폴백(`_sso_login()`)만 시도.

### 수정 내용
`frontend/app/embed/gw/page.tsx`에 마운트 시점 1회 실행되는 `useEffect` 추가:

```tsx
useEffect(() => {
  const params = new URLSearchParams(window.location.search);
  const gosso = params.get("gosso");
  if (gosso) {
    document.cookie = `gosso=${encodeURIComponent(gosso)}; path=/; SameSite=Lax; max-age=86400`;
  }
}, []);
```

- `SameSite=Lax`: iframe 내부에서 동일 origin(lucid.landf.co.kr)으로 fetch할 때 포함됨
- `max-age=86400`: 24시간 (실제 만료는 LFON 세션에 종속되므로 클라이언트 TTL은 상한 역할만)
- `HttpOnly` 미설정: `use-simple-chat`이 JS로 읽어야 하므로 의도적으로 off (middleware와 동일 정책)
- `encodeURIComponent`: 쿠키 값의 특수문자 안전 처리

이후 `use-simple-chat.ts:183`의 `document.cookie.match(/gosso=([^;]+)/)` 폴백이 자동으로 값을 픽업하여 POST body에 `gosso_cookie: <값>` 포함 → 백엔드 chat.py → CalendarWorker 컨텍스트에 전달 → MCP 도구 호출 시 사용자 본인 쿠키로 LFON POST.

## 결정 사항 및 주의점

- **왜 middleware matcher에 `/embed` 추가하지 않았나**: `/embed`는 JWT 인증 의무화에서 명시적으로 제외되어 있고, SSO flow와 혼선 위험이 있어 matcher 수정보다 페이지 레벨에서 처리하는 것이 좁고 명확한 변경임. 메모(chat_endpoint_jwt_auth.md)에도 `middleware matcher가 /embed 제외 → iframe은 middleware 안 거침 (의도됨)`으로 기록되어 있음.
- **위젯 JS 변경 없음**: 위젯은 이미 올바르게 `?gosso=` 파라미터를 붙이고 있음. `deploy.bat`이 위젯 JS도 동기화하지만 이번엔 위젯 파일에 변경 없음.
- **A2304013가 이전에 성공한 이유**: 서비스 계정 폴백(`_sso_login()`)이 특정 사용자의 `creator.id` 조합에 대해서만 POST를 허용. 즉 서비스 계정 우회는 불안정한 경로였고, 사용자 본인 쿠키로 POST하는 것이 정상 경로. 본 수정으로 gosso가 전달되면 LFON이 사용자 본인 세션으로 권한 체크하여 일관되게 동작.
- **빈 값 가드**: URL에 `?gosso=` 키만 있고 값이 빈 문자열인 경우 `if (gosso)` 조건으로 쿠키를 덮어쓰지 않음 → 이전 값 유지.
- **후속 검증 포인트**: 배포 후 운영 로그에서 `CalendarWorker … gosso → 있음` 문자열이 나타나는지, `create_event` POST가 400번대 없이 200으로 반환되는지 확인 필요.

## 2차 수정 (쿠키 경유 → prop 직접 전달)

### 재발 원인
1차 배포 후에도 A2604011 재시도에서 `gosso → 없음` 지속(blue 로그 11:30:22). 점검 결과:
- 사용자 브라우저에 `GOSSOcookie=bf7b6e30-...` 존재 확인
- iframe src에 `&gosso=bf7b6e30-...` 정확히 실림 확인
- 그럼에도 백엔드 수신 값은 빈 문자열

즉 위젯 JS → iframe URL까지는 정상이나, iframe 내부에서 `useEffect` → `document.cookie` 쓰기 → `use-simple-chat` 읽기 경로 어딘가에서 끊김. 유력 원인은 Next.js static prerender된 embed/gw HTML의 브라우저 캐시(hydration 전 구 번들 실행), same-site iframe이지만 특정 브라우저의 third-party cookie 쓰기 제한, 또는 document.cookie 반영 타이밍 경쟁 조건. 쿠키 경유 자체가 신뢰할 수 없는 경로.

### 수정 (쿠키 우회, prop 경로)
URL 파라미터를 React state로 추출하여 컴포넌트 트리로 직접 흘려보내는 방식으로 전환:

- `frontend/app/embed/gw/page.tsx`: 기존 `useEffect` 제거, `useState` 초기화 함수로 `gosso` 파라미터 추출 → `EmbedChat`에 `gossoCookie` prop 전달
- `frontend/components/embed-chat.tsx`: `gossoCookie?: string` prop 선언, `useSimpleChat` options에 전달
- `frontend/hooks/use-simple-chat.ts`:
  - `UseSimpleChatOptions`에 `gossoCookie?: string` 추가
  - fetch body 구성 시 `gossoCookie` prop이 있으면 그 값을 `gosso_cookie`로 사용, 없을 때만 기존 `document.cookie` 폴백

### 보존된 폴백
본 사이트 SSO 경로(lucid.landf.co.kr 직접 접속)에서는 middleware가 `?gosso=` URL 파라미터를 `gosso` 쿠키로 세팅하는 로직이 여전히 유효하므로, prop 없이 쿠키만 있는 경우도 정상 동작. 두 경로가 독립적으로 작동.

### 기대 효과
- 브라우저 캐시/쿠키 정책/SameSite 영향 완전 무관
- iframe hydration 타이밍과 무관 (React state는 마운트 즉시 사용 가능)
- 디버깅 용이 (Network 탭의 request body에 `gosso_cookie` 필드 직접 확인 가능)

## 3차 진단 & 최종 결론 (LFON 서버 환경 불일치)

2차 배포 후 A2203003 재테스트 시 `gosso → 있음`은 정상 찍히지만 여전히 `create_event` 실패(LFON `"서버에 연결할 수 없습니다"`). prop 경로는 동작 확정 → 문제는 **LFON API 호출 구간**으로 이동.

### 진단 로깅 추가
`backend/app/mcp_servers/calendar_mcp_server.py`의 `_api_request_with_cookies`에 파일 로깅(`backend/logs/calendar_mcp.log`) 추가. MCP 서버는 별도 stdio 프로세스라 stderr가 main backend server.log에 캡처되지 않아, LFON 응답 상세 진단이 불가능했던 문제도 이번에 해결.

기록 항목:
- 모든 LFON HTTP 요청의 status code + body preview
- 쿠키 종류(GOSSOcookie vs 서비스 계정 SCOUTER 포함), attempt 번호
- 사용자 쿠키 실패 → 서비스 계정 폴백 진입 여부
- create_event 호출 시 cal/user/go_user_id/gosso_present
- 최종 실패 시 구체 메시지 (LFON 서버 응답 거부 표시)

### 실제 LFON 응답 (2026-04-24 11:53, A2203003 테스트)
```
POST /api/calendar/3858/event/ cookies=[GOSSOcookie] → status=401
  body="인증되지 않았습니다. (common.unauthenticated)"
→ 서비스 계정 폴백
POST /api/calendar/3858/event/ cookies=[GOSSOcookie,SCOUTER] → status=403
  body="com.daou.go.calendar.exception.EventAuthorizationException"
```

- 사용자 GOSSOcookie로 운영 LFON API 찌르니 **401** — LFON이 세션을 모름
- 서비스 계정 폴백(김용국, wg0403)은 GET은 통과하지만 POST에서 **403 EventAuthorizationException**

### 진짜 원인: 개발 그룹웨어 세션 / 운영 LFON API 도메인 불일치
위젯이 **개발 그룹웨어**에 임베드되어 있었음. 사용자가 개발 그룹웨어에 로그인하면 **개발 LFON** 세션용 GOSSOcookie를 발급받는데, 우리 백엔드는 `LFON_BASE_URL=https://lfon.landf.co.kr` (운영)을 바라보므로 개발 쿠키를 운영 LFON에 보내면 당연히 401. 서비스 계정 폴백도 서비스 계정(wg0403=김용국)이 타인 캘린더에 쓰기 권한 없어 403으로 실패.

이전 A2304013 케이스가 성공한 것도 **서비스 계정 본인 캘린더(id=2677)** 였기 때문 — 타인 캘린더였다면 동일하게 403이었을 것.

### 해결
**운영 그룹웨어에 위젯을 임베드**하니 정상 동작 확인. 운영 그룹웨어 세션의 GOSSOcookie는 운영 LFON API에서 정상 인증되어 사용자 본인 쿠키로 POST 수행 → 201/200.

즉 코드는 처음부터 올바르게 동작했고, **환경 구성(위젯 임베드 위치 ↔ 백엔드 LFON 타겟)만 맞추면 되는** 이슈였음. 1·2차 수정(쿠키 → prop 경로 전환, 진단 로깅 추가)은 이 결론에 도달하기 위한 필수 단계였으며, prop 경로는 브라우저 환경 불안정성에도 견고한 구조로 그대로 가치가 있음.

### 운영 주의사항
- **개발 그룹웨어에서 위젯 테스트 시 캘린더 쓰기 불가**. dev 환경에서 제대로 테스트하려면 `LFON_BASE_URL`도 개발 LFON으로 맞춘 별도 dev 백엔드 인스턴스가 필요.
- 향후 dev 환경 분리 시 `.env`에 dev용 `LFON_BASE_URL` / `LFON_SSO_USERNAME` / `LFON_SSO_PASSWORD` 세트를 별도 주입해야 함.
- 서비스 계정(wg0403=김용국)은 타인 캘린더에 쓰기 권한 없음 — 사용자 본인 쿠키 경로가 유일한 방법. 따라서 prop 경로는 필수 인프라.
