# 2026-05-04 위젯 본체 이동 시 SSO 자동 인증

## 개요
그룹웨어 위젯에서 "본체에서 열기" 버튼을 누르면 새 탭이 로그인 화면으로 빠지던 문제를 해결. 위젯이 보유한 X-Widget-Auth 토큰을 본체 SSO empno 토큰으로 변환하는 백엔드 엔드포인트를 추가하고, embed-chat의 본체 진입 흐름이 그 토큰을 받아 `?empno=&gosso=` 쿼리로 부착하도록 변경.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/api/routes/auth.py | 수정 | `POST /api/auth/widget-to-sso` 엔드포인트 추가 |
| frontend/components/embed-chat.tsx | 수정 | `handleOpenInMainApp`이 토큰 변환 후 `?empno=` 부착 |

## 상세 내용

### 문제 원인
- 위젯 페이지 `/embed/gw`는 middleware matcher에서 제외되어 본체 도메인의 `auth_token` JWT 쿠키가 발급된 적이 없는 상태로 동작.
- 기존 `handleOpenInMainApp`이 단순히 `window.open('/chat/{sessionId}')`만 호출 — SSO 파라미터 없이 본체 진입 → 미들웨어가 쿠키 미존재 확인 → `/login` 리다이렉트.
- 위젯의 `X-Widget-Auth` 토큰을 그대로 `?empno=`에 꽂을 수도 없음. 평문 포맷이 다름:
  - 위젯 토큰: `email|timestamp_ms` (WIDGET_SHARED_KEY로 AES)
  - SSO empno 토큰: `사번` 그대로 (AES_KEY로 AES)
- 키 값은 운영에서 우연히 같지만 평문 포맷이 다르므로 미들웨어 복호화 결과가 사번이 아닌 `email|ts` 문자열이 되어 JWT empno에 잘못된 값이 저장됨.

### 해결 방식 — 변환 엔드포인트
`POST /api/auth/widget-to-sso`:
- `Depends(get_current_user_widget)`로 X-Widget-Auth 헤더 자동 검증 (만료/포맷/사번 정규화 모두 기존 로직 재사용).
- 검증된 사번을 `AES_KEY`로 `encrypt_empno()` 재암호화 → `{ "encrypted_empno": "..." }` 반환.

### 본체 진입 흐름
1. 위젯 사용자 클릭 → `handleOpenInMainApp` 실행.
2. `widgetAuthToken`이 있으면 `POST /api/auth/widget-to-sso` 호출 (X-Widget-Auth 헤더로).
3. 응답의 `encrypted_empno`로 `?empno=...&gosso=...` 쿼리 구성 후 `window.open('/chat/{sid}?empno=...&gosso=...')`.
4. 본체 미들웨어가 기존 SSO 흐름 그대로 처리 → `auth_token` HttpOnly JWT 쿠키 발급 → 채팅 화면 정상 진입.
5. 변환 실패 시 fallback — 쿼리 없이 그냥 새 탭 (기존 동작 = 로그인 화면).

### 코드 변경
- `backend/app/api/routes/auth.py`:
  - `from app.utils.crypto import decrypt_empno, encrypt_empno`
  - `from fastapi import APIRouter, Depends, HTTPException`
  - `from app.api.dependencies.widget_auth import get_current_user_widget`
  - `WidgetToSsoResponse(BaseModel)` + `POST /auth/widget-to-sso` 핸들러 추가.
- `frontend/components/embed-chat.tsx`:
  - `handleOpenInMainApp`을 async로 변경.
  - `widgetAuthToken` 있을 때만 변환 호출 — Outline wiki 등 다른 embed 모드에 영향 X (본체 이동 버튼 자체가 `chatMode === "groupware_embed"`에서만 노출).

## 결정 사항 및 주의점
- **변환 엔드포인트 vs 미들웨어 확장**: 미들웨어가 위젯 토큰까지 처리하도록 분기 추가하면 새 탭 URL에 8시간짜리 위젯 토큰이 그대로 노출됨. 변환 방식은 새 탭 URL에 SSO empno만 노출(기존과 동일) → 위젯 토큰 직접 노출 방지.
- **encrypt_empno 키**: `AES_KEY` (기본값 `landf01234567890`) 사용 — 미들웨어의 `/api/auth/decrypt`와 동일 키이므로 별도 환경변수 불필요.
- **gosso 전달**: 그룹웨어 캘린더 등 일부 워커가 GOSSOcookie를 필요로 하므로 변환 시점에 prop으로 가지고 있던 `gossoCookie`를 그대로 동봉.
- **fallback 동작**: 변환 실패 시 사용자는 기존과 동일하게 로그인 화면을 보게 됨 — 회귀 X.
- **재진입 1회 후 쿠키 발급**: 변환 호출 자체는 위젯 토큰만 검증하므로 매 클릭마다 새 SSO empno 발급 → 새 탭마다 fresh JWT 쿠키 발급 가능.
