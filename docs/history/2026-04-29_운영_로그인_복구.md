# 2026-04-29 운영 로그인 흐름 복구

## 개요
운영 서버에서 로그인 버튼을 눌러도 홈으로 리다이렉트되지 않고 로그인 페이지에 머무는 문제, 그리고 비밀번호 설정 메일 링크가 잘못된 도메인(`lucidai.landf.co.kr` — 미존재)으로 발송되던 문제를 함께 수정.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `C:/Services/nginx/conf/conf.d/lfchatbot.conf` | 수정 | `/api/auth/{login,logout,request-setup,setup-password}`를 frontend로 exact-match 라우팅 (HTTP/HTTPS 양쪽) |
| `backend/app/api/routes/auth.py` | 수정 | `SITE_URL` 기본값 `https://lucidai.landf.co.kr` → `https://lucid.landf.co.kr` |
| `C:/Services/LFChatbot_prod/green/backend/.env` | 추가 | `SITE_URL=https://lucid.landf.co.kr` 명시 |
| `C:/Services/LFChatbot_prod/blue/backend/.env` | 추가 | 동일 |

## 상세 내용

### 1) 로그인 후 리다이렉트 누락
- **증상**: 운영서버에서 로그인 버튼을 누르면 홈으로 이동하지 않고 로그인 페이지에 그대로 머무름.
- **원인**: Nginx config의 `location /api/auth/`가 모든 `/api/auth/*` 요청을 FastAPI 백엔드로 직접 프록시. 프론트엔드의 Next.js API route(`frontend/app/api/auth/login/route.ts`)는 백엔드 응답을 받아 `auth_token`/`empno` 쿠키를 세팅해야 하는데, Nginx가 이를 우회하면서 쿠키가 붙지 않음.
- 쿠키 미세팅 → `router.push("/")` 시 middleware가 인증 실패로 판정 → `/login`으로 강제 이동.
- **수정**: Next.js가 처리해야 할 4개 경로(`login`, `logout`, `request-setup`, `setup-password`)를 `location =` exact-match로 frontend(`frontend_active`)에 우선 라우팅. 나머지 `/api/auth/*`(decrypt, resolve-gw-user, register, verify-token, change-password)는 기존대로 backend로 유지.
- HTTP IP 서버블록(192.168.90.30, localhost)과 HTTPS 443 서버블록(lucid.landf.co.kr) 양쪽 모두 동일하게 추가.

### 2) 비밀번호 설정 메일 링크 도메인 오류
- **증상**: 비밀번호 찾기 → 이메일 수신 시 링크 도메인이 `https://lucidai.landf.co.kr`(미존재)로 와서 클릭해도 접속 불가.
- **원인**: `auth.py:273`의 `SITE_URL` 기본값이 `https://lucidai.landf.co.kr`이고, 운영 `.env`에 `SITE_URL`이 미설정 상태였음.
- **수정**:
  - 소스 기본값을 `https://lucid.landf.co.kr`로 정정 (향후 신규 환경에서도 안전).
  - 운영 green/blue `.env`에 `SITE_URL=https://lucid.landf.co.kr` 명시 추가.
- **반영 조건**: 백엔드 프로세스 재기동 필요 (.env는 startup 시 로드).

## 결정 사항 및 주의점
- Next.js API route 4개는 모두 내부에서 `fetch(${BACKEND_URL}/api/auth/...)`로 백엔드를 다시 호출하므로 백엔드 인증 로직 자체는 변경되지 않음. Nginx 변경은 "쿠키 세팅용 Next.js 한 단계를 거쳐 가게 만드는" 라우팅 보정.
- SSO 흐름(`?empno=...` 쿼리)은 middleware.ts가 처리하므로 이번 변경과 무관.
- Nginx reload는 graceful이라 다운타임 없음. 적용 명령:
  ```
  C:\Services\nginx\nginx.exe -t -p C:\Services\nginx\
  C:\Services\nginx\nginx.exe -s reload -p C:\Services\nginx\
  ```
- `.env`의 `SITE_URL` 변경을 반영하려면 active 슬롯(현재 green, 포트 8002)의 backend 프로세스 재기동 필요.
