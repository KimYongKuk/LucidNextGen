# 2026-04-22 SSO 진입 JWT 발급 누락 핫픽스

## 개요

전날(2026-04-21) 도입한 채팅 엔드포인트 JWT 인증이 **SSO 진입 사용자에게 전면 401**을 일으키는 회귀 발견. Next.js `middleware.ts`의 SSO flow가 복호화된 `empno` 쿠키만 set하고 `auth_token` JWT를 발급하지 않던 구조 때문. `jose` 라이브러리의 `SignJWT`로 JWT를 서명·쿠키 set 하는 로직을 middleware에 추가하여 해결. 새벽 배포로 운영 적용 및 dev/prod end-to-end 검증 완료.

## 배경 — 전날 배포 실패 → 롤백 경위

전날 오전 JWT 인증 + 위젯 암호화 토큰을 배포(`92ebaac`) 직후 **모든 사용자 재접속 401 에러** 발생. 즉시 `rollback.bat`으로 blue 복구. 원인 분석 결과:
- 기존 설계 전제: "로그인된 사용자는 `auth_token` 쿠키가 있으므로 투명 이관"
- 실제: **대부분 사용자는 그룹웨어 SSO로 진입** → `/api/auth/login` form 경로를 타지 않음 → `auth_token`이 아예 발급된 적 없음
- 결과: 배포 후 모든 채팅 API 호출이 "인증 토큰 없음" 401

F12 Network에서 401 응답한 요청의 Cookie 헤더를 보니 `empno`, `gosso`만 있고 `auth_token` 부재 — 이게 결정적 단서.

## 상세 내용

### 1. middleware.ts 핵심 추가

```typescript
import { jwtVerify, SignJWT } from 'jose'

async function createAuthToken(empno: string): Promise<string> {
  return await new SignJWT({ empno })
    .setProtectedHeader({ alg: JWT_ALGORITHM })
    .setIssuedAt()
    .setExpirationTime('24h')
    .sign(SECRET_KEY)
}

// SSO flow 내부 — 복호화 성공 후
const authToken = await createAuthToken(decrypted_empno)
redirectResponse.cookies.set('auth_token', authToken, {
  httpOnly: true,
  secure: false,
  sameSite: 'lax',
  path: '/',
  maxAge: 60 * 60 * 24
})
```

payload 구조(`{ empno, iat, exp }`)와 알고리즘(HS256), 만료시간(24시간) 모두 백엔드 `auth.py::_create_token()`과 일치. 같은 `SECRET_KEY`로 서명하므로 백엔드 `get_current_user`가 정상 검증.

### 2. SECRET_KEY 정책 변경

전날까지:
- backend `.env`: `SECRET_KEY=landf01234567890_fastapi_secret_key_change_in_production` (기본값)
- frontend: `SECRET_KEY` 설정 안 됨 → middleware 하드코딩 fallback 사용

이번 배포로 모든 환경 통일:
- backend `.env`: `SECRET_KEY=landf01234567890`
- frontend `.env.local`: `SECRET_KEY=landf01234567890`
- `deploy.bat`이 매 배포마다 backend → frontend로 자동 복사 (findstr로 추출해서 echo)

### 3. WIDGET_TOKEN_VALID_SECONDS 연장

5분(300초) → 1시간(3600초). 사용자가 탭 오래 열어둔 뒤 메시지 보낼 때 401 방지. replay 공격 리스크는 남지만 iframe 컨텍스트에서 토큰 탈취가 어려워 허용 범위.

### 4. deploy.bat 자동화 개선

```bat
set "SECRET_KEY_VAL="
for /f "tokens=1,* delims==" %%a in ('findstr /B "SECRET_KEY=" "%BASE%\%IDLE%\backend\.env"') do set "SECRET_KEY_VAL=%%b"
(
    echo BACKEND_URL=http://192.168.90.30:%IDLE_BACKEND_PORT%
    if defined ADMIN_USERS echo NEXT_PUBLIC_ADMIN_USERS=!ADMIN_USERS!
    if defined SECRET_KEY_VAL echo SECRET_KEY=!SECRET_KEY_VAL!
) > "%BASE%\%IDLE%\frontend\.env.local"
```

매 배포마다 `frontend/.env.local` 재생성 시 `SECRET_KEY`도 자동 포함. 수동 동기화 작업 불필요.

## 결정 사항 및 주의점

### 결정
- **middleware에서 직접 JWT 서명 (jose 활용)**: 백엔드에 SSO-JWT 엔드포인트 추가하지 않음. 서버 간 왕복 불필요, 빠름. 단점은 key가 frontend/backend 양쪽에 있어야 함.
- **SECRET_KEY 짧은 값으로 통일**: 긴 기본값 대신 `landf01234567890`. 관리 편의 우선 (16자라 AES-128 동일 키로 재사용 가능).
- **`rollback.bat` 버그 인지**: state.txt를 자동 업데이트하지 않음 — 수동 `echo "blue" > state.txt` 필요. 추후 수정 대상.

### 주의점
- **SSO 재진입 1회 필수**: 기존 로그인 상태 사용자도 배포 후 그룹웨어 링크 다시 타고 들어와야 `auth_token` 발급됨 (F5 새로고침만으로는 안 됨)
- **SECRET_KEY 변경 시 전원 강제 로그아웃**: 발급된 JWT 모두 서명 검증 실패. 재배포 이후 점진적으로 재로그인 유도.
- **같은 키가 3곳에 하드코딩 (SECRET_KEY/AES_KEY/WIDGET_SHARED_KEY)**: 관리 편의성 vs 키 로테이션 유연성 트레이드오프. 유출 시 모든 flow 영향.
- **전날 도입한 legacy `?empno=` fallback**: 운영 그룹웨어 JSP가 아직 구버전이면 이 경로로 동작. JSP 업데이트 후 제거 예정.
