# 2026-04-22 SSO 진입 JWT 발급 누락 핫픽스

## 개요

전날(2026-04-21) 도입한 채팅 엔드포인트 JWT 인증이 **SSO 진입 사용자에게 전면 401**을 일으키는 회귀 발견. Next.js `middleware.ts`의 SSO flow가 복호화된 `empno` 쿠키만 set하고 `auth_token` JWT를 발급하지 않던 구조 때문. `jose` 라이브러리의 `SignJWT`로 JWT를 서명·쿠키 set 하는 로직을 middleware에 추가하여 해결. 새벽 배포로 운영 적용 및 dev/prod end-to-end 검증 완료.

## 배경 — 전날 배포 실패 → 롤백 경위

전날 오전 JWT 인증 + 위젯 암호화 토큰을 배포(`92ebaac`) 직후 **모든 사용자 재접속 401 에러** 발생. 즉시 `rollback.bat`으로 blue 복구. 원인 분석 결과:
- 기존 설계 전제: "로그인된 사용자는 `auth_token` 쿠키가 있으므로 투명 이관"
- 실제: **대부분 사용자는 그룹웨어 SSO로 진입** → `/api/auth/login` form 경로를 타지 않음 → `auth_token`이 아예 발급된 적 없음
- 결과: 배포 후 모든 채팅 API 호출이 "인증 토큰 없음" 401

F12 Network에서 401 응답한 요청의 Cookie 헤더를 보니 `empno`, `gosso`만 있고 `auth_token` 부재 — 이게 결정적 단서.

## 상세 내용

### 1. middleware.ts 핵심 추가

```typescript
import { jwtVerify, SignJWT } from 'jose'

async function createAuthToken(empno: string): Promise<string> {
  return await new SignJWT({ empno })
    .setProtectedHeader({ alg: JWT_ALGORITHM })
    .setIssuedAt()
    .setExpirationTime('24h')
    .sign(SECRET_KEY)
}

// SSO flow 내부 — 복호화 성공 후
const authToken = await createAuthToken(decrypted_empno)
redirectResponse.cookies.set('auth_token', authToken, {
  httpOnly: true,
  secure: false,
  sameSite: 'lax',
  path: '/',
  maxAge: 60 * 60 * 24
})
```

payload 구조(`{ empno, iat, exp }`)와 알고리즘(HS256), 만료시간(24시간) 모두 백엔드 `auth.py::_create_token()`과 일치. 같은 `SECRET_KEY`로 서명하므로 백엔드 `get_current_user`가 정상 검증.

### 2. SECRET_KEY 정책 변경

전날까지:
- backend `.env`: `SECRET_KEY=landf01234567890_fastapi_secret_key_change_in_production` (기본값)
- frontend: `SECRET_KEY` 설정 안 됨 → middleware 하드코딩 fallback 사용

이번 배포로 모든 환경 통일:
- backend `.env`: `SECRET_KEY=landf01234567890`
- frontend `.env.local`: `SECRET_KEY=landf01234567890`
- `deploy.bat`이 매 배포마다 backend → frontend로 자동 복사 (findstr로 추출해서 echo)

### 3. WIDGET_TOKEN_VALID_SECONDS 연장

5분(300초) → 1시간(3600초). 사용자가 탭 오래 열어둔 뒤 메시지 보낼 때 401 방지. replay 공격 리스크는 남지만 iframe 컨텍스트에서 토큰 탈취가 어려워 허용 범위.

### 4. deploy.bat 자동화 개선

```bat
set "SECRET_KEY_VAL="
for /f "tokens=1,* delims==" %%a in ('findstr /B "SECRET_KEY=" "%BASE%\%IDLE%\backend\.env"') do set "SECRET_KEY_VAL=%%b"
(
    echo BACKEND_URL=http://192.168.90.30:%IDLE_BACKEND_PORT%
    if defined ADMIN_USERS echo NEXT_PUBLIC_ADMIN_USERS=!ADMIN_USERS!
    if defined SECRET_KEY_VAL echo SECRET_KEY=!SECRET_KEY_VAL!
) > "%BASE%\%IDLE%\frontend\.env.local"
```

매 배포마다 `frontend/.env.local` 재생성 시 `SECRET_KEY`도 자동 포함. 수동 동기화 작업 불필요.

## 결정 사항 및 주의점

### 결정
- **middleware에서 직접 JWT 서명 (jose 활용)**: 백엔드에 SSO-JWT 엔드포인트 추가하지 않음. 서버 간 왕복 불필요, 빠름. 단점은 key가 frontend/backend 양쪽에 있어야 함.
- **SECRET_KEY 짧은 값으로 통일**: 긴 기본값 대신 `landf01234567890`. 관리 편의 우선 (16자라 AES-128 동일 키로 재사용 가능).
- **`rollback.bat` 버그 인지**: state.txt를 자동 업데이트하지 않음 — 수동 `echo "blue" > state.txt` 필요. 추후 수정 대상.

### 주의점
- **SSO 재진입 1회 필수**: 기존 로그인 상태 사용자도 배포 후 그룹웨어 링크 다시 타고 들어와야 `auth_token` 발급됨 (F5 새로고침만으로는 안 됨)
- **SECRET_KEY 변경 시 전원 강제 로그아웃**: 발급된 JWT 모두 서명 검증 실패. 재배포 이후 점진적으로 재로그인 유도.
- **같은 키가 3곳에 하드코딩 (SECRET_KEY/AES_KEY/WIDGET_SHARED_KEY)**: 관리 편의성 vs 키 로테이션 유연성 트레이드오프. 유출 시 모든 flow 영향.
- **전날 도입한 legacy `?empno=` fallback**: 운영 그룹웨어 JSP가 아직 구버전이면 이 경로로 동작. JSP 업데이트 후 제거 예정.

## 추가 이슈: 운영 그룹웨어 JSP scriptlet이 `${userId}` EL 값을 못 읽음

### 증상
1차 JSP 적용 후 운영 그룹웨어에서 위젯 401. 원인 분석:
- iframe URL에 `?sid=...&empno=A2304013&gosso=...` (`token=` 없음) → JSP의 `widgetToken` 빈 문자열
- `${userId}` EL 표현은 정상 작동 (원래 JSP가 이렇게 썼고 widget JS의 legacy fallback path에 A2304013 전달됨)
- 그러나 `pageContext.findAttribute("userId")` Java scriptlet에서 `null` 반환

### 원인 추정
다우오피스(TerraceTims) 그룹웨어의 커스텀 EL 리졸버가 `${userId}`를 처리하지만, 이 값이 일반 `pageContext.findAttribute()`가 찾을 수 있는 표준 scope(page/request/session/application) 어디에도 없음.

### 해결
JSTL `<c:set>` 대신 **다중 소스 탐색 버전** 사용 (session의 여러 key 이름 + Map 언패킹):
```java
String empNo = null;
Object attr = pageContext.findAttribute("userId");
if (attr != null) empNo = attr.toString();
// 세션의 여러 key 이름 시도: userId, empNo, empno, USER_ID, EMP_NO, loginUser, user
// Map이면 Map.get("userId")/Map.get("empNo") 접근
// ...
```

### 위젯 JS 운영 동기화 이슈 (부수 발견)
- 기존 `deploy.bat`은 `frontend/widget/*.js`를 nginx 서빙 디렉토리(`C:\Services\LFChatbot_prod\widget\`)로 복사 안 함
- 오늘 위젯 JS에 `buildIframeSrc()` 등 신규 로직 추가했지만 운영에 반영 안 된 상태로 배포
- 즉시 수동 `cp`로 복사 + `deploy.bat`의 Step 2.6에 `xcopy` 로직 추가하여 재발 방지

## 교훈 (Lessons Learned)

1. **EL과 scriptlet의 scope 조회 동등성을 믿지 말 것**: 커스텀 EL 리졸버 환경에서는 `${X}`가 해결되는 위치가 `pageContext.findAttribute("X")`와 다를 수 있음
2. **정적 asset 배포 자동화**: nginx가 서빙하는 파일이 repo의 최신인지 확인. `deploy.bat`에서 누락된 단계 있는지 재점검
3. **iframe 인증은 cookie로 풀 수 없음**: `sameSite=lax` 쿠키는 cross-site iframe에 전송 안 됨. 별도 인증 경로(X-Widget-Auth 헤더) 필수
4. **SSO는 form login과 별개 경로**: 실제 사용자는 대부분 SSO로 들어옴. form login `/api/auth/login` 만 JWT 발급하게 두면 운영 전면 장애
5. **배포 후 실사용자 관점 테스트 필수**: 개발자의 쿠키 상태(`auth_token` 이미 보유)로는 SSO 사용자 회귀를 발견 못함
6. **그룹웨어의 `${userId}`는 사번이 아님**: 다우오피스 JSP의 `${userId}`는 **DaouOffice 내부 user_id** (Shiro principal = 337 형태)이며, LFON/TIMS DB의 사번(A2304013)과 완전 별개. 우연히 TIMS DB에 같은 숫자의 user_id가 다른 사용자를 가리킬 수 있어 **그대로 쓰면 엉뚱한 사용자로 인증됨** (예: 김용국 → 337 → 남세종)

## 추가 이슈: DaouOffice `${userId}` vs LFON 사번 불일치 (해결 완료)

### 발견
- 김용국(A2304013, wg0403)의 DaouOffice `${userId}` = **337**
- TIMS DB `v_user_info_mapping.user_id = 337` = **A1607003 (남세종)**
- 그대로 widget token에 쓰면 김용국이 남세종 계정으로 인증됨 💥

### 원인
DaouOffice는 자체 내부 user_id 체계(Shiro principal)를 씀. TIMS DB는 LFON 별도 체계. 두 체계는 서로 무관하지만 숫자가 겹칠 수 있음.

### 해결
디버그 JSP로 session/request 모든 attribute 덤프 → `email = "wg0403@landf-dev.co.kr"` 속성 발견. email의 local-part(`@` 앞)가 login_id와 동일.

**최종 구조:**
- JSP: `email` 속성(`wg0403@domain`)을 AES 암호화하여 widget token에 포함
- 백엔드 `_resolve_to_sabun()`: email → local-part(login_id) → `v_user_info_mapping.login_id` 조회 → `employee_number` 반환
- 프로세스 수명 캐싱 (login_id → 사번)으로 매 요청 DB 왕복 방지

**입력 허용 범위:**
| 입력 | 처리 |
|------|------|
| `A2304013` (사번) | 그대로 반환 (패턴 `^[A-Z]\d{7}$` 매칭) |
| `wg0403@landf.co.kr` (email) | `@` 앞 추출 → login_id 조회 |
| `wg0403` (login_id) | 바로 login_id 조회 |

향후 다른 그룹웨어나 통합 시에도 사번/email/login_id 어느 것이든 받아서 처리 가능.
