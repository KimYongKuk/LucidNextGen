# 2026-04-06 Outline Wiki 하이브리드 시멘틱 검색

## 개요
Outline Wiki 검색을 기존 키워드 방식에서 **하이브리드 검색**(키워드 + 시멘틱)으로 개선. Haiku로 문서를 요약한 뒤 BGE-m3-ko로 임베딩하여 ChromaDB에 저장하고, 검색 시 Reciprocal Rank Fusion(RRF)으로 양쪽 결과를 병합.

## 변경 파일 요약

| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/services/outline_sync_service.py` | 신규 | 동기화 서비스: Outline→Haiku요약→ChromaDB 임베딩 |
| `backend/app/agents/workers/outline_worker.py` | 수정 | 하이브리드 검색 래핑 (RRF 병합) |
| `backend/app/utils/outline_sync_scheduler.py` | 신규 | APScheduler 증분 동기화 (30분 주기) |
| `backend/app/api/routes/outline_sync.py` | 신규 | 수동 동기화 트리거/상태 API |
| `backend/app/main.py` | 수정 | 스케줄러 등록 + 라우터 추가 |

## 상세 내용

### 아키텍처

```
[사전 준비: 동기화 배치 (30분 주기)]
  Outline API → documents.list (전체 published)
       ↓
  변경 감지: updated_at vs ChromaDB 메타데이터 비교
       ↓
  변경된 문서만 → Haiku 요약 (200~300자, 세마포어 5 동시)
       ↓
  "# {title}\n\n{summary}" → BGE-m3-ko 임베딩
       ↓
  ChromaDB "outline_wiki" 컬렉션에 upsert

[실시간: 사용자 검색]
  사용자 질문
       ↓
  OutlineWorker.prepare_tools()에서 search_documents 래핑
       ↓
  ┌─ 키워드 검색 (MCP → Outline API, 기존 로직)
  ├─ 시멘틱 검색 (ChromaDB 벡터 유사도)
  ↓
  RRF 병합 (양쪽 모두 찾은 문서가 상위)
       ↓
  접근 제어 필터 적용 → 최종 결과 반환
```

### 동기화 서비스 (outline_sync_service.py)

| 항목 | 값 |
|------|-----|
| ChromaDB 컬렉션 | `outline_wiki` (단일 통합) |
| 임베딩 모델 | BGE-m3-ko (기존 공유) |
| 요약 모델 | Haiku |
| 요약 최대 길이 | 300자 |
| 임베딩 대상 | `# {title}\n\n{summary}` (최대 800자) |
| 메타데이터 | document_id, collection_id, title, summary, updated_at, url |
| Haiku 동시성 | 5 (세마포어 제한) |
| API 배치 크기 | 20건씩 페이지네이션 |

### RRF 병합 (outline_worker.py)

```python
RRF_score(doc) = Σ 1/(k + rank_i)  # k=60
```

- 키워드 검색 결과 top-N과 시멘틱 검색 결과 top-10을 document_id로 매칭
- 양쪽 모두 등장한 문서는 높은 RRF 스코어 → 상위 노출
- 한쪽에만 등장한 문서도 결과에 포함 (약점 보완)
- 시멘틱 검색 실패 시 키워드 결과만 반환 (graceful degradation)

### 스케줄러 / API

- **자동**: 30분마다 증분 동기화 (변경분만 처리)
- **수동 (백그라운드)**: `POST /api/v1/admin/outline-sync/trigger`
- **수동 (동기)**: `POST /api/v1/admin/outline-sync/run`
- **상태 조회**: `GET /api/v1/admin/outline-sync/status`

### 환경변수

| 변수 | 기본값 | 설명 |
|------|--------|------|
| `OUTLINE_SYNC_ENABLED` | `true` | 동기화 스케줄러 on/off |
| `OUTLINE_SYNC_INTERVAL_MINUTES` | `30` | 동기화 주기 (분) |
| `OUTLINE_HYBRID_SEARCH` | `true` | 하이브리드 검색 on/off |

## 결정 사항 및 주의점
- MCP 서버는 subprocess(stdio)로 동작하므로 ChromaDB 직접 접근 불가 → OutlineWorker에서 하이브리드 조율
- 문서당 벡터 1개 (제목+요약) → 5,000건이어도 벡터 5,000개, 메모리 ~25MB
- 초기 동기화 시 Haiku API 비용 발생 (5,000건 기준 ~$1.5, 1회성)
- 동기화 중 서버 정상 운영 (기존 키워드 검색으로 동작, 완료 후 하이브리드 활성화)
- `OUTLINE_HYBRID_SEARCH=false`로 즉시 비활성화 가능

### 중요: 초기 동기화는 반드시 서버 API로 실행할 것
- **별도 프로세스(스크립트) 단독 실행 불가**: BGE-m3-ko 임베딩 모델이 GPU 메모리(RTX 3070 8GB)를 점유하므로, 서버 프로세스가 이미 모델을 로드한 상태에서 별도 프로세스가 또 로드하면 `os error 1455` (페이징 파일 부족) 발생
- **올바른 방법**: 서버 재시작 후 API로 트리거 → 서버 내부에서 이미 로드된 임베딩 모델을 공유
  ```
  POST http://localhost:8000/api/v1/admin/outline-sync/trigger
  ```
- `run_outline_sync.py` 스크립트는 서버가 내려간 상태에서만 사용 가능 (실질적으로 사용하지 않음)
