## 개요
Planner-Executor 경로에서 회계 도메인 단일 질의("법인카드 사용 규정")에 Planner LLM이 `corp_rag` task를 헤지로 추가 생성하여 무관한 HR docs(예: 인사팀 급여계좌 변경.txt)가 corp_sources에 노출되던 결함을 수정. 4/29에 IntentClassifier 쪽은 동일 결함을 손봤지만 Planner의 시스템 프롬프트에는 도메인 분기 가드가 누락되어 있어 재발. 두 분류기에 동일 가드를 중복 유지하던 구조의 비효율을 함께 해소하기 위해 도메인 라우팅 가이드를 별도 모듈(`routing_guide.py`)로 추출하여 단일 진실 원천화.

## 결정 사항 및 주의점

- **두 분류기는 별도 모델/역할이라 통합 불가**: IntentClassifier(Haiku, 단일 인텐트)와 Planner(Sonnet, Task DAG 분해)는 사용하는 모델·출력 형식이 달라 단일 분류기로 통합할 수 없음. 단, **도메인-워커 매핑은 동일**해야 하므로 이 부분만 `routing_guide.py`로 추출하여 단일 source 보장.
- **운영의 실제 분기**: `PLANNER_ENABLED=true` 상태에서 모든 요청이 Planner를 거치며 `is_trivial=true`면 워커 직접 실행(DB intent=워커명, 89%), `is_trivial=false`면 Executor+Synthesizer 신 경로(DB intent='planner', 11%), `PlannerFallback` 예외 시만 IntentClassifier로 폴백. 따라서 두 분류기 모두 도메인 가드가 살아있어야 한다.
- **`WORKER_CAPABILITIES`(state.py) 한 줄 가드는 부족**: LLM이 worker_catalog의 짧은 description은 흘려보고 RULE/Example을 더 강하게 따른다. 운영에서 발견된 도메인 헤지 결함은 routing_guide의 표 + 배타성 원칙 + Few-shot Example 추가가 필요.
- **다음 라우팅 룰 추가/변경 시**: `routing_guide.py`의 `DOMAIN_ROUTING_GUIDE`만 수정하면 두 분류기에 자동 반영. 4/29→4/30처럼 한 쪽만 갱신하는 잠복 결함 방지.
- **운영 적용**: Planner 시스템 프롬프트는 프로세스 수명 동안 캐시(`self._system_prompt`)되므로 백엔드 재시작 필요. 다음 deploy.bat 실행 시 반영.
- **잠재 보완점**: 파일명 부스팅(corp-acct에 `8. 전결규정.txt`, `기타(전표)_접대비 관련 규정.txt`가 있는데 BGE-m3 매칭에서 밀림) — 이번 PR 범위 외, 별도 작업.
