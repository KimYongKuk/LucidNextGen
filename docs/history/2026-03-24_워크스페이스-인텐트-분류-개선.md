# 2026-03-24 워크스페이스 인스트럭션 기반 인텐트 분류 개선

## 개요
워크스페이스 인스트럭션이 인텐트 분류에 반영되지 않아 올바른 Worker로 라우팅되지 못하는 문제를 해결.
추가로 Worker 응답에 내부 DB 스키마가 노출되지 않도록 가드레일 추가.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/agents/intent_classifier.py` | 수정 | CLASSIFIER_PROMPT에 workspace_instructions 필드 추가, WORKSPACE RULES 확장, classify()에서 인스트럭션 앞 500자 전달 |
| `backend/app/agents/workers/base_worker.py` | 수정 | build_system_prompt()에 내부 스키마 노출 방지 가드레일 추가 |

## 상세 내용

### 이슈 1: 내부 스키마 노출 방지 (base_worker.py)
- 사용자가 워크스페이스 인스트럭션 생성을 요청할 때, LLM이 시스템 프롬프트에 포함된 DB 뷰/컬럼명을 그대로 응답에 포함하는 문제
- `build_system_prompt()` 마지막에 가드레일 한 줄 추가:
  - "응답에 내부 DB 뷰 이름, 컬럼명, SQL 쿼리, 테이블 구조를 절대 포함하지 마세요"

### 이슈 2: 워크스페이스 인스트럭션 기반 인텐트 분류 (intent_classifier.py)

**배경:**
- 기존: Classifier가 workspace_name, workspace_description만 참고 → instructions 내용은 Worker 단계에서야 주입됨
- 결과: "전자결재 조회용 워크스페이스"에서 "자주보전 활동 정리해줘"라고 하면 approval 키워드가 없어 direct로 분류 → DirectWorker에는 도구 없음 → "조회할 수 없습니다"

**변경 사항:**

1. **CLASSIFIER_PROMPT CONTEXT 섹션**: `workspace_instructions` 필드 추가
2. **WORKSPACE RULES 확장**: 인스트럭션 기반 라우팅 가이드라인 추가
   - 메시지가 워크스페이스 목적과 일치 → 해당 전문 인텐트로 분류
   - 메시지가 워크스페이스 목적과 무관 → 인스트럭션 무시, 메시지 기준 분류
3. **classify() 메서드**: `context["workspace_instructions"]`에서 앞 500자만 잘라서 프롬프트에 전달

**인스트럭션 500자 컷 이유:**
- 평균 인스트럭션 2,124자, 최대 11,128자 — 전문은 토큰 낭비
- 대부분 인스트럭션은 앞 500자 내에 "목적/역할" 기술
- Haiku 분류기에 ~170토큰 추가 비용만 발생

**context 전달 경로 (기존 인프라 활용):**
```
chat.py → workspace_context["instructions"] (DB 전문)
  → a2a_streaming.py → context["workspace_instructions"] (이미 있었음)
    → classifier.classify() → 앞 500자만 프롬프트에 포함 (신규)
```

## 결정 사항 및 주의점
- `default_intent` 필드 추가 방안은 채택하지 않음 — 사용자에게 추가 입력을 요구하는 UX 문제
- GPT/Claude/Gemini의 접근과 동일하게 인스트럭션을 분류 컨텍스트에 포함하는 방식 채택
- 500자 컷은 대부분 커버하지만, 인스트럭션 앞부분에 목적 없이 바로 실행 명령으로 시작하는 경우 분류 정확도 떨어질 수 있음
