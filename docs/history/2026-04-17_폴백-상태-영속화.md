# 2026-04-17 폴백 상태 영속화

## 개요
Bedrock Inference Profile 폴백 상태(`_using_fallback`, `_restore_at`)를 JSON 파일에 영속화하여 프로세스 재시작(배포, 크래시) 후에도 복구 예약이 유지되도록 개선.

## 배경
기존 `RegionFallbackManager`는 폴백 플래그를 싱글톤 인스턴스 속성으로만 보관했다. 폴백이 활성화되면 다음 날 KST 09:00(UTC 00:00)에 자동 복구하도록 예약되지만, 그 전에 프로세스가 재시작되면 플래그가 `False`로 초기화되어 쿼터가 아직 회복되지 않았는데도 primary(us.*)를 시도하게 된다.

실제로 2026-04-16 17:02 Blue 슬롯에서 폴백 활성화 → 2026-04-17 07:21 Green 슬롯 배포로 재시작 → 08:30경 us.*를 재시도하면 스로틀링에 재차 걸릴 위험이 있었다.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/core/region_fallback.py` | 수정 | `_load_persisted_state()`, `_save_persisted_state()` 추가. `__init__`에서 복원, `activate_fallback`/`reset_to_primary`/auto-restore 경로에서 저장 |
| `C:/Services/LFChatbot_prod/blue/backend/.env` | 수정 | `FALLBACK_STATE_FILE` 추가 |
| `C:/Services/LFChatbot_prod/green/backend/.env` | 수정 | `FALLBACK_STATE_FILE` 추가 |
| `C:/Services/LFChatbot_prod/shared/` | 신규 | 공용 상태 파일 디렉터리 |

## 상세 내용

### 저장 포맷 (JSON)
```json
{"using_fallback": true, "restore_at": 1776384000.0}
```

### 파일 경로
- 환경변수 `FALLBACK_STATE_FILE`로 지정
- 운영: `C:/Services/LFChatbot_prod/shared/fallback_state.json` (Blue/Green 공용)
- 개발 기본값: `backend/data/fallback_state.json`

### 원자적 저장
```python
tmp_path = _STATE_FILE + ".tmp"
with open(tmp_path, "w") as f:
    json.dump(...)
os.replace(tmp_path, _STATE_FILE)  # Windows atomic rename
```

### 복원 로직
`__init__`에서 파일 읽고, `restore_at > now()`인 경우에만 `_using_fallback=True`로 복원. 이미 만료된 상태면 무시하고 정상 시작.

### 저장 시점 (3곳)
1. `activate_fallback()` — 폴백 활성화 시 `(True, restore_at)`
2. `reset_to_primary()` — 수동 리셋 시 `(False, 0.0)`
3. `is_fallback_active` 자동 복구 분기 — KST 09:00 경과 시 `(False, 0.0)`

## 결정 사항 및 주의점

### Redis 대신 파일 선택
- 단일 플래그 + 타임스탬프만 저장하는 용도로 Redis는 과잉
- 표준 라이브러리만으로 구현 가능 (의존성 제로)
- 운영자가 `type fallback_state.json`으로 상태 확인 가능
- 수동 리셋은 파일 삭제로 즉시 가능

### Blue/Green 공용 경로 필요
- 두 슬롯이 각자 `backend/data/`에 쓰면 슬롯 전환 시 상태 공유 안 됨
- `C:/Services/LFChatbot_prod/shared/`에 두면 활성 슬롯이 기록한 상태를 대기 슬롯이 배포 후 즉시 복원
- 동시 접근 경합은 사실상 없음(활성 슬롯만 Bedrock 호출)

### 파일 손상 대비
- 로드 실패 시 `(False, 0.0)` 반환하여 정상 상태로 시작
- `logger.warning`으로 기록은 남김

### 관리자 메일 중복 방지
- 영속화 복원 시에는 메일 발송 안 함 (이미 폴백 활성화 메일은 최초 활성화 때 전송됨)
- 로그에 `[PROFILE_FALLBACK] Restored from ... using_fallback=True, restore_at=...` 만 남김

### 한계
- 파일 경로가 잘못 설정되면 영속화 실패 → 기존 동작(인메모리)로 폴백
- 대안으로 파일 대신 MySQL 테이블 사용도 검토 가능하나, 현재 규모에서는 오버엔지니어링
