# 2026-04-21 암호화 파일 업로드 소프트 실패 + 사전 감지

## 개요
비밀번호 보호(암호화)된 PDF/Office 파일 업로드 시 ChromaDB 임베딩이 실패하여 "Upload failed" 에러로 처리되던 문제 해결. 디스크 저장은 이미 성공한 상태였으므로 **VOC 첨부/다운로드/UI 참조 용도로는 사용 가능**했지만 사용자 관점에서는 업로드 실패로 보였음. 임베딩 실패를 소프트 실패로 전환 + 업로드 단계에서 암호화 파일을 사전 감지하여 임베딩 단계 자체를 스킵.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/api/routes/upload.py | 수정 | `_detect_file_encryption()` 헬퍼 신규, `_process_upload_background` 재작성 (사전 감지 → 임베딩 스킵, 임베딩 예외 시 `completed_disk_only`) |
| backend/app/api/routes/workspace.py | 수정 | `_process_workspace_upload_background`에 동일 패턴 적용 (upload.py 헬퍼 재사용) |
| frontend/components/multimodal-input.tsx | 수정 | 폴링에서 `completed_disk_only` 상태를 ready로 처리 + info 토스트로 안내 |
| frontend/lib/api/workspaces.ts | 수정 | `UploadStatus.status` union에 `completed_disk_only` 추가, `uploadFileWithPolling` 결과에 `disk_only` 플래그 주입 |
| frontend/components/workspace-settings-modal.tsx | 수정 | 업로드 결과의 `disk_only` 파일 수집 후 일괄 info 토스트로 안내 |

## 새 Status 값
| status | 의미 | 프론트 처리 |
|--------|------|-------------|
| processing | 진행 중 | 기존 그대로 |
| completed | 임베딩까지 성공, RAG 검색 가능 | success 토스트 |
| **completed_disk_only** | 디스크 저장은 성공, 임베딩 스킵/실패 | ready 상태 + info 토스트 ("검색 인덱싱 건너뜀") |
| failed | 처리 자체 실패 (디스크 저장도 미완) | error 토스트 |

## 사전 감지 로직
`_detect_file_encryption(file_content, filename) -> (bool, reason)`:
- **PDF**: `PyPDF2.PdfReader.is_encrypted` 확인
- **DOCX/XLSX/PPTX/XLS**: 매직 바이트로 OLE2 Compound Document 판별 (`\xd0\xcf\x11\xe0`). 일반 OOXML은 ZIP(`PK`) 형식이므로 OLE2면 암호화된 Office 파일로 간주
- 감지 실패/예외 시 False 반환 (정상 임베딩 경로로 진행)

## 동작 플로우
```
업로드 요청
  ├─ 디스크 저장 (기존, 이미 soft-fail)
  ├─ 백그라운드 처리 시작
  │   ├─ _detect_file_encryption() — 암호화 감지
  │   │     → True면 status=completed_disk_only, 임베딩 스킵, 종료
  │   │     → False면 계속
  │   └─ ChromaDB.upload_file() 시도
  │         → 성공: status=completed
  │         → 예외: status=completed_disk_only (디스크엔 이미 있음)
  └─ 프론트 폴링
      ├─ completed → success 토스트, 첨부 가능
      ├─ completed_disk_only → info 토스트, 첨부 가능 (RAG만 불가)
      └─ failed → error 토스트 (이제 거의 안 뜸)
```

## 영향 범위
- **VOC 자동 등록 + 첨부**: 암호화 파일도 `user_uploads/`에 저장되므로 `register_works_voc`의 attachments 파라미터로 정상 첨부 가능 — 이번 수정으로 회복된 핵심 가치
- **채팅 multimodal**: `completed_disk_only` 파일도 `ready` 상태라 메시지 전송 가능
- **RAG 검색**: 암호화 파일은 ChromaDB에 없으므로 `search_user_files`/`search_workspace_docs` 결과에서는 누락됨. 사용자에게 info 토스트로 안내
- **has_session_files()**: ChromaDB + 디스크 OR 검사이므로 `completed_disk_only` 파일도 디스크 체크로 감지됨 (2026-04-20 수정 덕분)

## 결정 사항 및 주의점
- **Sniffing 범위 제한**: PyPDF2의 `is_encrypted` + OLE2 매직 바이트 정도만 확인. 완벽하지 않지만 대다수 엔터프라이즈 암호화 파일(사내 DRM, PDF 비밀번호 보호) 커버
- **예외 처리 관대함**: 감지 자체에서 예외 발생 시 False 반환해 정상 경로 진행 — 감지 로직이 업로드를 깨뜨리지 않도록
- **에러 메시지 단축**: 임베딩 실패 메시지는 120자(`message`) / 200자(`warning`)로 제한해 UI 깨짐 방지
- **`failed` 상태 유지**: 파일 크기 초과, 초기 디스크 저장 실패 등 "정말 실패한" 경우는 여전히 `failed`로 유지 — 구분 가능
- **복호화 시도 없음**: 사용자가 비밀번호를 입력받아 복호화하는 UX는 복잡도 높아 제외. 향후 요구 시 별도 flow로 설계
