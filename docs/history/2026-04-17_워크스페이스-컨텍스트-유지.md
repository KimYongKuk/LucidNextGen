# 2026-04-17 워크스페이스 컨텍스트 유지 (사이드바)

## 개요
워크스페이스 내 대화 아이템을 클릭하면 `/chat/[id]`로 이동하면서 URL의 `?workspace_id=…` 쿼리가 사라져, 사이드바가 워크스페이스 필터를 잃고 전체 채팅 리스트로 되돌아가던 버그를 수정.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| frontend/components/sidebar-history-item.tsx | 수정 | `Link` href에 `workspace_id` 쿼리 포함, `workspaceId` prop 추가 및 memo 비교에 반영 |
| frontend/components/sidebar-history.tsx | 수정 | `ChatItem`에 현재 `workspaceId` 전달 (pinned/today/past 3곳) |
| frontend/components/chat.tsx | 수정 | 세션 마운트 시 `effectiveWorkspaceId`를 URL 쿼리에 `replaceState`로 반영, `query` 처리 시에도 workspace_id 유지 |

## 상세 내용

### 원인
- `SidebarHistory` / `SidebarWorkspaces`는 `useSearchParams().get("workspace_id")`로 현재 워크스페이스 컨텍스트를 읽어 SWR 키에 포함시킴.
- 워크스페이스 안에서 채팅 아이템을 클릭하면 `<Link href={\`/chat/${chat.id}\`}>`로 이동 → 쿼리가 비어 필터 해제.
- 새로고침/직접 URL 접근 시에도 `chat/[id]/page.tsx`가 백엔드에서 세션의 `workspace_id`를 받아 `Chat`에 prop으로만 넘기고 URL엔 반영하지 않아 같은 문제가 발생.

### 해결
1. `sidebar-history-item.tsx`: `workspaceId`가 있으면 `href = \`/chat/${id}?workspace_id=${ws}\``로 구성. memo 비교에서도 `workspaceId` 변화 감지.
2. `sidebar-history.tsx`: 3개 그룹 매핑 모두에 `workspaceId={workspaceId}` 전달.
3. `chat.tsx`: `effectiveWorkspaceId`가 있고 현재 URL 쿼리가 다르면 `window.history.replaceState`로 `workspace_id` 보장. 기존의 `query` 처리 useEffect가 쿼리스트링을 통째로 지우던 부분도 `workspace_id`는 보존하도록 수정.

Next.js 14.1+ 이후 `window.history.replaceState`는 `useSearchParams`와 동기화되므로 서버 라운드트립 없이 사이드바의 SWR 키가 즉시 갱신됨.

## 결정 사항 및 주의점
- `router.replace`가 아닌 `window.history.replaceState`를 사용한 이유: 서버 컴포넌트 재실행으로 인한 메시지 재조회를 피하기 위함. 기존 코드도 동일 패턴(`window.history.replaceState({}, "", \`/chat/${id}\`)`)을 쓰고 있었음.
- `Chat`의 `stableWorkspaceId`는 초기값으로 한 번만 고정되므로, 세션이 속한 워크스페이스와 URL이 불일치해도 세션 소속이 우선 — 사이드바도 세션 소속 워크스페이스 기준으로 표시됨.
- `onFinish`의 `mutate` 키 직렬화는 여전히 `workspace_id` 없이 생성 중이나, `SidebarHistory`는 `workspace_id`가 포함된 키로 캐싱되므로 별개 이슈(워크스페이스 내 새 메시지 전송 후 사이드바 자동 갱신)가 남아 있을 수 있음 — 필요 시 후속 작업.
