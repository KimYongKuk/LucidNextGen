### `tool_called_any` assertion 추가
IT/회계 워커처럼 규정과 VOC 도구를 모두 가진 경우, "둘 중 하나라도 호출되면 PASS"가 더 합리적. AND 매칭(`tool_called`)과 별개로 OR 매칭이 가능해짐.

```yaml
tool_called_any: ["search_it_docs", "execute_it_voc_query"]
# 둘 중 하나만 호출돼도 통과 — LLM 판단에 따라 둘 다 호출하거나 하나만 호출하는 경우
```

## 결정 사항 및 주의점

### 발견된 라우팅 결함 (별도 후속 필요)

intent_classifier가 **"규정"** 키워드를 corp_rag의 강한 신호로 잡고 도메인(IT/회계)을 보지 않음:

| 입력 | 분류 | 문제 |
|---|---|---|
| "사내 정보보안 관리 규정" | corp_rag → CorpRAGWorker | search_it_docs 없음 → hr/safety로 폴백 → "찾지 못함" 응답 |
| "사내 경비처리 규정" | corp_rag → CorpRAGWorker | search_ac_docs 없음 → hr만 → "찾지 못함" 응답 |

→ 사용자가 "사내 IT 보안 규정 알려줘" 같은 자연스러운 질문을 하면 운영에서 "찾지 못함" 응답이 나오는 UX 결함. 후속 작업으로 분리:
- intent_classifier에 "PC/네트워크/VPN/AD/보안/소프트웨어/라이선스" + "규정" → it_support 룰 추가
- "법인카드/경비/전표/세금계산서/회계/결산" + "규정" → acct_support 룰 추가
- 또는 CLARIFY 분기로 처리

### Eval-driven dev 가치 입증
이번 케이스 작성 → 실행 → FAIL 추적 사이클에서:
