# 2026-03-31 회의실/자산 예약 Worker

## 개요
그룹웨어(LFON) 자산예약 REST API를 연동하여, 루시드AI 챗봇에서 자연어로 회의실 조회/예약 등록/취소가 가능한 ReservationWorker 모듈을 추가했다.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/mcp_servers/reservation_mcp_server.py` | 신규 | MCP 서버 (6개 도구: 사업장/회의실 조회, 예약 현황, 내 예약, 등록, 취소) |
| `backend/app/agents/workers/reservation_worker.py` | 신규 | Sonnet Worker (보안 래핑, 시스템 프롬프트) |
| `backend/app/agents/state.py` | 수정 | `Intent.RESERVATION` 추가, `INTENT_TO_WORKER` 매핑, `WORKER_CAPABILITIES` |
| `backend/app/agents/workers/__init__.py` | 수정 | `ReservationWorker` import 및 레지스트리 등록 |
| `backend/app/agents/intent_classifier.py` | 수정 | RESERVATION 인텐트 분류 (LLM + quick_classify regex) |
| `backend/mcp_config.json` | 수정 | `reservation_server` 항목 추가 |

## 상세 내용

### 아키텍처
```
사용자: "내일 본사 빈 회의실 있어?"
    ↓
IntentClassifier → Intent.RESERVATION (quick_classify regex)
    ↓
ReservationWorker (Sonnet)
    ↓ prepare_tools(): employee_number 강제 주입
MCP Tools (reservation_mcp_server.py)
    ↓ SSO 쿠키 자동 로그인 + API 호출
LFON REST API (https://lfon.landf.co.kr/api/asset/...)
    ↓
스트리밍 응답
```

### MCP 도구 (6개)
| 도구 | 용도 | API |
|------|------|-----|
| `get_sites` | 사업장 목록 | `GET /api/asset` |
| `get_rooms` | 회의실 목록 | `GET /api/asset/{id}/item/` |
| `get_daily_reservations` | 날짜별 예약 현황 | `GET /api/asset/{id}/items/daily` |
| `get_my_reservations` | 내 예약 조회 (전 사업장 병렬) | daily API × 전 사업장 → userId 필터 |
| `create_reservation` | 예약 등록 | `POST /api/asset/{id}/item/{id}/reserve` |
| `cancel_reservation` | 예약 취소 | `DELETE /api/asset/item/reservation` |

### 인증 방식
- **서비스 계정 SSO 로그인**: `LFON_SSO_USERNAME`/`LFON_SSO_PASSWORD`로 `/api/login` → 쿠키 캐싱
- API가 payload의 `user.id`를 신뢰 → 서비스 계정 세션으로 다른 사용자 명의 예약 가능
- 401/403 시 자동 재로그인 (1회 재시도)
- POST/DELETE 요청에 `Origin`/`Referer` 헤더 필수 (CSRF 보호)
- **서비스 계정 권한 요구**: 조회는 아무 계정이나 가능하지만, 예약 등록/삭제는 **자산예약 관리 권한**이 있는 계정 필요 (일반 계정은 403)

### 사번 → GO user.id 매핑
- 기존 VIEW 재사용: `v_user_info_mapping` (별도 VIEW 불필요)
- `v_user_info_mapping.user_id` = GO 내부 user.id = 예약 API의 `user.id`
- 사번 체계: "A2304013" 형태 (숫자가 아님)
- 프로세스 수명 캐싱 (`_user_mapping_cache`)

### "내 예약 조회" 구현 방식
`/api/asset/my/reservation/remnant`는 세션 기반이라 서비스 계정 대행 불가.
대신 **전 사업장 daily API 병렬 조회 + userId 필터** 방식으로 구현:
- 기본 7일, 사용자 요청 시 최대 30일 (`days` 파라미터)
- daily API는 2일치 반환 → `ceil(days/2)` 스텝 × 15 사업장 병렬
- `asyncio.Semaphore(20)`으로 동시성 제한
- 예약ID로 중복 제거 (`seen_ids` set)

### 보안 안전장치
1. `prepare_tools()`에서 `employee_number`를 SSO 세션 사번으로 강제 주입 (LLM 변경 불가)
2. 시스템 프롬프트에 "등록/취소 전 반드시 사용자 확인" 규칙
3. `create_reservation` 전 `get_daily_reservations`로 충돌 확인 필수
4. `item_id`는 반드시 `get_rooms` 결과에서 가져온 값만 사용 (추측/기억 의존 금지)

### 에러 처리
- API non-200 응답도 JSON 파싱 → 실제 에러 메시지(예: "이미 예약된 시간") 사용자에게 전달
- `_api_request`에서 status code + response body 상세 로깅 (stderr)
- `create_reservation` 호출 전 파라미터 로깅 (asset, item, user)

### On/Off
- `RESERVATION_WORKER_ENABLED` 환경변수 (기본값: `true`)
- intent_classifier의 quick_classify에서 체크

### 인텐트 분류
- **quick_classify regex**: `회의실|회의룸|C/?R\d|예약 (현황|등록|취소|조회|목록|내역)|빈 회의실|내 예약|예약 잡아|예약 해줘`
- **LLM fallback**: CLASSIFIER_PROMPT에 reservation 인텐트 설명 + 예시 추가

## 결정 사항 및 주의점
- **API 기반** (DB VIEW가 아닌 REST API): 쓰기 작업이 포함되어 있어 그룹웨어 비즈니스 로직(충돌 체크, 알림 등)을 타기 위함
- **서비스 계정 필요**: 예약 등록/삭제에는 자산예약 관리 권한이 있는 계정이 필요. 일반 계정(예: dikim)은 403 반환
- **서비스 계정 쿠키 만료**: SSO 쿠키의 수명은 그룹웨어 설정에 의존. 만료 시 자동 재로그인하지만, 비밀번호 변경 시 .env 업데이트 필요
- **공용차량 예약**: 사업장 중 `[공용차량] *` 카테고리도 동일 API로 조회/예약 가능 (회의실과 동일 구조)
- **사업장 약칭 매핑**: "본사"→id=70, "성서"→id=100 등은 Worker 시스템 프롬프트에 하드코딩
- **v_reservation_user_mapping VIEW**: DBA가 생성했으나 실제로는 기존 `v_user_info_mapping` 재사용으로 불필요. 삭제 가능
