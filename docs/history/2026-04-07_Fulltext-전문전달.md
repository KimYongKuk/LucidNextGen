# 2026-04-07 파일 전문 전달 기능 (Fulltext)

## 개요
파일 업로드 시 추출된 전체 텍스트를 디스크에 보관하고, "요약해줘", "번역해줘" 같은 전문 처리 요청 시 임베딩 검색 대신 전체 텍스트를 LLM에 직접 전달하는 기능을 추가했다. 기존 청크 검색(3개 × 1,000자 = ~3,000자)으로는 불가능했던 전문 요약/번역이 가능해진다.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/services/chromadb_service.py` | 수정 | `_save_fulltext()`, `get_fulltext()`, `get_file_id_by_filename()` 메서드 추가, `upload_file()`에서 텍스트 추출 직후 fulltext 저장 |
| `backend/app/mcp_servers/rag_server.py` | 수정 | `get_uploaded_file_content` MCP 도구 추가 (50,000자 제한) |
| `backend/app/agents/workers/user_files_worker.py` | 수정 | tool_names에 추가, 시스템 프롬프트에 전문형/검색형 분기 규칙 |

## 상세 내용

### 아키텍처 (기존 vs 개선)
```
기존: 파일 → 청킹 → 임베딩 → ChromaDB → 검색 → 청크 3개 → LLM (부분 분석)
개선: 파일 → 전체 텍스트 → data/fulltext/{file_id}.txt 저장 (병렬)
                        └→ 청킹 → 임베딩 → ChromaDB (기존 유지)
      
      요청 시: "요약해줘" → get_uploaded_file_content → fulltext 직접 전달 → LLM (전문 분석)
              "OO 찾아줘" → search_user_files → ChromaDB 검색 → LLM (기존 동작)
```

### Fulltext 저장 (`chromadb_service.py`)
- 저장 경로: `data/fulltext/{file_id}.txt`
- 저장 시점: `upload_file()` 내 텍스트 추출 직후 (567줄)
- Excel: `_chunk_excel_sheets()` 결과를 합산하여 저장
- 일반 파일: `extract_text()` 반환값 그대로 저장
- `file_id` 생성을 텍스트 추출 전으로 이동 (fulltext 저장에 필요)

### MCP 도구 (`rag_server.py`)
- `get_uploaded_file_content(session_id, filename, user_id)`
- ChromaDB 메타데이터에서 `filename` → `file_id` 매핑 조회
- `data/fulltext/{file_id}.txt` 읽기
- 50,000자 초과 시 트렁케이트 + 안내 메시지
- fulltext 미존재 시 (기능 추가 이전 업로드) 안내 메시지

### UserFilesWorker 프롬프트 분기
- LLM이 요청 유형을 판단하여 도구 선택:
  - 전문형 (요약/번역/전체분석) → `get_uploaded_file_content`
  - 검색형 (키워드/특정내용) → `search_user_files` / `search_workspace_docs`

## 결정 사항 및 주의점
- **50,000자 제한**: Claude Sonnet 200K 컨텍스트의 ~32% 사용, 약 33페이지 커버
- **기존 동작 유지**: 청크 검색 경로는 변경 없음, fulltext는 추가 경로
- **기능 추가 이전 파일**: fulltext.txt가 없으므로 안내 메시지 반환 (search_user_files 유도)
- **하위 호환**: file_id 생성 위치만 이동, 기존 ChromaDB 저장 로직 변경 없음
