# 2026-04-22 LFON 그룹웨어 계정 관리 연동

## 개요

LFON 그룹웨어(다우오피스/TerraceTims)의 계정 관리 API 3종(OTP 초기화, 비밀번호 초기화, 메일 용량 증설)을 챗봇에 통합. 사용자가 자연어로 "내 OTP 초기화해줘" / "비밀번호 리셋" / "메일 용량 늘려줘" 요청 시 ITSupportWorker가 **2-step confirm/execute 패턴**으로 안전하게 처리한다.

본인 계정만 조작 가능하도록 사번(`employee_number`)을 Worker의 `prepare_tools()`에서 강제 주입하며, 사번 → 그룹웨어 user_id 변환은 MCP 서버 내부에서 수행(LLM이 임의 user_id 주입 불가). LFON API는 `Authorization` 토큰 헤더로 인증되며, 이력은 LFON 측 DB에 자동 기록되므로 챗봇 측 별도 감사 테이블은 두지 않는다.

## 아키텍처

### 2-step Confirm/Execute 패턴

```
[1단계] 사용자: "내 그룹웨어 비밀번호 초기화해줘"
           ↓
      ITSupportWorker (LLM)
           ↓ 도구 호출
      confirm_reset_password (employee_number 자동 주입)
           ↓ MCP 서버 내부
      - 사번 → v_user_info_mapping → user_id (1367)
      - UUID 토큰 발급, 메모리에 (token → {empno, action, user_id, issued_at}) 저장
      - 토큰 60초 유효
           ↓
      응답: {status, token, action_label, detail, next_step}
           ↓
      LLM → 사용자에게 확인 질문
      "비밀번호 초기화를 진행할까요? '예'라고 답해주세요."

[2단계] 사용자: "예"
           ↓
      LLM → execute_reset_password(token="abc123")
           ↓ MCP 서버 내부
      - 메모리 조회: 토큰 존재? 60초 내? empno 일치? action 일치?
      - 모두 통과 → LFON API PUT 호출
        (Authorization: <토큰>, body: {ids:[1367], updateDataFlag:"password"})
      - 응답 파싱 (평문 "success"/"fail" OR JSON object)
      - logger.info 기록
           ↓
      응답: "✅ 초기화 완료" 등 사용자 친화 메시지
```

### 6개 도구 매트릭스

| 작업 | 1단계 (confirm) | 2단계 (execute) | LFON Endpoint | updateDataFlag |
|------|----------------|----------------|--------------|--------------|
| OTP 초기화 | `confirm_reset_otp` | `execute_reset_otp` | `/secure/lfon/management/OTP` | `OTP` |
| 비밀번호 초기화 | `confirm_reset_password` | `execute_reset_password` | `/secure/lfon/management/password` | `password` |
| 메일 용량 증설 | `confirm_increase_mail_quota` | `execute_increase_mail_quota` | `/secure/lfon/management/userInfo` | `mail` |

## 상세 내용

### 1. MCP 서버 (`lfon_mgmt/server.py`)

**핵심 함수:**
- `_lookup_user_id_by_sabun(employee_number)` — 사번 → v_user_info_mapping.user_id 조회, 프로세스 수명 캐싱
- `_issue_confirm_token(empno, action, user_id)` — UUID 토큰 발급, 만료된 토큰 청소
- `_consume_confirm_token(token, expected_empno, expected_action)` — 1회성 검증 + 소비
  - 토큰 존재 / 만료 / empno mismatch / action mismatch 모두 체크
- `_call_lfon_api(action, user_id)` — 실제 PUT 호출, 평문/JSON 응답 모두 파싱

**토큰 검증 매트릭스:**

| 공격 시나리오 | 방어 |
|-------------|------|
| 만료된 토큰 사용 | `issued_at` 60초 검증 |
| 같은 토큰 2회 사용 | `_pending_confirms.pop()`으로 1회성 보장 |
| 다른 사용자 토큰 탈취 | `info["empno"] != expected_empno` 차단 |
| confirm_password로 execute_otp 트릭 | `info["action"] != expected_action` 차단 |

### 2. ITSupportWorker 통합

**`tool_names`에 6개 추가** — 기존 `register_works_voc`, `reset_sap_password`와 함께 관리.

**프롬프트 섹션 추가 (`## 그룹웨어 계정 관리`):**
- 사용자 요청 패턴별 도구 매핑 (OTP/비밀번호/메일용량)
- SAP 비밀번호와의 구분 (LFON vs SAP 혼동 방지)
- 2-step 패턴 엄격 준수 규칙
  - confirm 응답 받자마자 같은 턴 execute 호출 **금지**
  - "예/네/진행" 명확한 동의 확인 후에만 execute
  - token은 다른 action에 전달 금지
- 메일 용량 증설 결과 `code` 값별 분기 (`success` / `full` / `end`)
- 본인 한정 — 타인 대상 요청은 거절하고 WORKS 등록 안내

**`prepare_tools()` 확장:**
기존 `register_works_voc`, `reset_sap_password`만 래핑했던 것을 `SECURED_TOOLS` 집합으로 재구성하고 6개 신규 도구 추가. LLM이 `employee_number`를 임의 조작해도 `prepare_tools()`에서 인증된 사번으로 덮어씀.

## 감사 로그 정책

**챗봇 측에는 별도 감사 테이블 없음.** 이유:
- LFON 서버가 원본 API 호출 이력을 자체 DB에 기록 (호출자 IP, 토큰, 대상 사용자, 시각, 결과 포함)
- 챗봇 측 중복 기록은 관리 부담만 증가, 진실의 원천은 LFON 측
- 챗봇 `logger.info`로 action/empno/user_id/success/code는 남김 — 디버깅 및 사용자 클레임 대응용

## 결정 사항 및 주의점

### 결정
- **2-step 패턴은 프롬프트 + 서버 토큰 검증 이중화** — 프롬프트 가이드가 실패해도 서버 토큰 검증이 최후 보루
- **본인만 조작**: SECURED_TOOLS 집합으로 `prepare_tools()`에서 강제 주입. 향후 IT 관리자용 확장은 별도 worker로 분리 예정
- **쓰로틀링 미적용**: 2-step이 실수 방지 역할, 본인 한정이라 남용 시나리오 제한적. 필요 시 향후 추가.
- **감사 테이블 미도입**: LFON 측 이력이 진실의 원천. 중복 기록은 관리 부담.

### 주의점
- **confirm 토큰은 프로세스 메모리 저장** — blue/green 전환 중 토큰 증발 가능 (사용자가 재확인 필요, 허용 가능한 빈도)
- **API Key 유출 시** `.env`의 `LFON_MGMT_API_KEY` 교체 + 개발자에게 토큰 재발급 요청 필요
- **LFON API 응답 포맷 불일치** — OTP/비밀번호는 평문 `"success"/"fail"`, 메일 증설은 JSON 객체. `_call_lfon_api()`에서 분기 처리됨
- **SAP vs LFON 비밀번호 혼동 주의** — 프롬프트에 명시했지만 LLM이 애매하게 판단할 경우 재질문하도록 가이드
- **쓰로틀링 없으므로** 사용자가 악의적으로 반복 초기화 시도하면 제한 없이 호출 가능. 모니터링 후 필요 시 `_pending_confirms` 레이트리밋 추가.

## Dev 검증 중 발견한 2가지 버그 및 수정

### 버그 1: MCP Subprocess 재생성으로 메모리 토큰 증발

**증상:** confirm_* 호출 후 즉시 execute_* 호출해도 "토큰 만료" 에러.

**원인:** `langchain-mcp-adapters`가 매 tool call마다 `create_session()`으로 MCP 서브프로세스를 새로 spawn. 초기 구현은 `_pending_confirms: dict` 메모리 저장소에 토큰을 저장했는데, confirm이 서브프로세스 A에 저장 → execute가 서브프로세스 B에서 조회 → 빈 dict → 만료 판정.

**수정:** Stateless AES 암호화 토큰으로 전환. 토큰 자체에 payload(`{"e": empno, "a": action, "u": user_id, "t": issued_at}`)를 AES-ECB + PKCS7 + base64 + URL 인코딩으로 담음. 검증 시 복호화해서 empno/action/TTL 체크. 메모리 저장 불필요.
- 키: `WIDGET_SHARED_KEY` 재사용 (기존 위젯 토큰과 동일 키)
- TTL: 300초(5분) → 사용자 응답 지연 여유

**트레이드오프:** 1회성 보장 손실(같은 토큰 재사용 가능). 단, 짧은 TTL + LLM은 재사용할 이유 없음 + 본인 한정이라 남용 리스크 낮음 → 허용.

### 버그 2: LFON API 응답 `{"result":"success"}` 구조 미처리 → AttributeError → LLM "오류" 응답

**증상:** 실제 증설은 완료됐는데 챗봇 UI에는 "죄송합니다, 처리 중 오류가 발생했습니다" 표시.

**원인:** LFON API가 증설 **성공 시** `{"result":"success"}` (JSON 내 문자열)을 반환. 초기 구현은 항상 dict로 가정:
```python
result = data.get("result", {})  # result = "success" (str!)
result.get("success", False)     # AttributeError: str has no .get
```
예외가 MCP tool error로 변환되어 LLM에 전달됨 → LLM이 "오류 발생" 응답.

**수정:** JSON result 파싱을 두 구조 모두 처리:
```python
if isinstance(result, str):
    # {"result": "success"} / {"result": "fail"}
    is_success = result.strip().lower() == "success"
    ...
elif isinstance(result, dict):
    # {"result": {"success": ..., "message": ..., "code": ...}}
    ...
```
catch-all `except Exception`도 추가 — 예외가 MCP error로 올라가지 않도록 친화적 메시지 반환.

### 추가 개선: 메시지 톤 중립화

`code: "full"` (이미 증설됨) 응답을 LLM이 "실패"로 해석해 "죄송합니다"로 시작. 메시지에서 "STATUS_NOT_ERROR" 태그 제거하고 순수 정보 톤으로 전환:
```
ℹ️ 확인 결과: 메일 용량이 이미 증설된 상태입니다.

추가 증설이 필요하시면 기안 상신을 통해 요청해주세요. 
현재 설정으로 충분하다면 별도 조치는 필요 없습니다.
```
"실패"/"오류"/"NOT_ERROR" 부정 표현 모두 제거 → LLM이 그대로 pass-through.
