# 2026-04-20 WORKS VOC 자동 등록 — 첨부파일 기능

## 개요
IT VOC 자동 등록 시(`register_works_voc`) 사용자가 업로드한 파일을 함께 첨부할 수 있도록 기능 추가. WORKS 내부 API를 리버스 엔지니어링한 2단계 플로우(`/api/file` → VOC 등록 body에 embed)로 구현. 공개 OpenAPI는 첨부를 지원하지 않아 배제.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/mcp_servers/works_it_mcp_server.py | 수정 | `_upload_file_to_works()` / `_resolve_attachment_path()` 추가, `register_works_voc`에 `attachments` 파라미터 + `_14v07o8vj` embed |
| backend/app/agents/workers/it_support_worker.py | 수정 | `build_system_prompt()` 오버라이드 — 업로드 파일 목록 주입 + 첨부 지시 규칙 |
| backend/app/api/routes/upload.py | 수정 | MAX_FILE_SIZE 10MB→50MB (3곳) |
| backend/app/api/routes/workspace.py | 수정 | MAX_FILE_SIZE 10MB→50MB |
| frontend/components/multimodal-input.tsx | 수정 | 프론트 MAX_FILE_SIZE 10MB→50MB |

## 상세 내용

### 플로우 (등록 시점 한 호출로 완결)
```
ITSupportWorker (LLM)
  ├─ 시스템 프롬프트에 "업로드된 파일 목록" 노출
  ├─ 사용자에게 "이 파일들도 첨부할까요?" 확인
  └─ register_works_voc(title, details, system_name, attachments=["error.log"])

works_it MCP 서버
  ├─ _resolve_attachment_path(fname, employee_number)
  │   → data/user_uploads/{date}/{employee_number}/{fname} 스캔
  │   → 경로 탈출(.., /, \) 차단, 범위 검증
  ├─ _sso_login() → GOSSOcookie 획득 (캐싱됨)
  ├─ _upload_file_to_works(cookies, path)
  │   → POST /api/file?GOSSOcookie=... (multipart/form-data)
  │   → {id, path, name, hostId} 수집
  ├─ values["_14v07o8vj"] = [meta1, meta2, ...]
  ├─ payload["_14v07o8vj"] = [meta1, meta2, ...]  (최상위 중복)
  └─ POST /api/works/applets/{WORKS_APPLET_ID}/docs (JSON)
```

### 결정사항

**Q1 → (b) 파일명 리스트 + 백엔드 resolve**
- LLM은 파일 경로가 아닌 **파일명**만 넘김 (`attachments=["error.log"]`).
- MCP 서버가 `employee_number`로 `user_uploads/{date}/{employee_number}/` 하위에서 exact filename 매칭으로 resolve.
- 여러 날짜에 동일 파일명 존재 시 최근 mtime 우선.

**Q2 → 명시 첨부 (사용자 확인 후)**
- ITSupportWorker `build_system_prompt()`이 턴마다 업로드 디렉터리를 스캔, 파일 목록을 프롬프트에 주입.
- LLM은 VOC 등록 제안 시 "아래 파일도 함께 첨부할까요?"로 사용자 확인 후 attachments 포함 호출.
- 매 턴 프롬프트 재생성이므로 대화 중간에 업로드된 파일도 다음 턴에서 자동 인식.

**Q3 → 등록 성공 후 원본 유지**
- Works 등록 성공해도 Lucid 디스크(`user_uploads/`) 원본은 그대로.
- ChromaDB 임베딩 결과도 그대로 유지 (LLM의 본문 자동 작성에 도움되므로).

### 보안 (`_resolve_attachment_path`)
- 파일명에 `/`, `\`, `..` 포함 거부
- `user_id`(=employee_number)는 `prepare_tools()`에서 SSO 세션 기준으로 강제 주입되므로 타 사용자 파일 접근 불가
- resolve된 경로가 `USER_UPLOAD_DIR` 하위인지 최종 검증
- 파일 미발견 / 경로 거부 시 해당 파일만 스킵, VOC 등록은 계속 진행

### 환경변수 / 상수
| 변수 | 기본값 | 설명 |
|------|--------|------|
| WORKS_ATTACHMENT_FIELD | `_14v07o8vj` | Works applet의 첨부 필드 ID. 934/1445 동일 구조 확인됨. 다를 경우 env로 override |
| WORKS_APPLET_ID | `934` | 앱릿 ID. 테스트 시 `1445`로 임시 변경 권장 |
| MAX_FILE_SIZE | 50MB | 업로드 크기 상한 (상향 전 10MB) |

### 파일 업로드 응답 형식 (Works `/api/file`)
```json
{
  "id": null,
  "path": "/202616/19FE9FF2AEFD46038C50495362EC997F",
  "name": "스크린샷.png",
  "hostId": "TMUzAwMDAwMEEtOTEw"
}
```
응답이 리스트로 감싸져 오는 경우도 있어 `isinstance(data, list)` 분기 처리.

### 에러 핸들링
- 파일 resolve 실패 / 업로드 실패: `attachment_warnings` 리스트에 기록, VOC 등록은 계속
- 성공 응답 본문에 "첨부파일: ... / 첨부 제외: ..." 분리 표시
- SSO 실패 폴백(OpenAPI) 경로는 첨부 미지원 → 로그로 경고 후 본문만 등록

## 테스트 방법
1. `.env`에서 `WORKS_APPLET_ID=1445`로 임시 전환 (기본 934는 운영 applet)
2. backend 재시작
3. Lucid 채팅에서 파일 업로드 (스크린샷/로그 등, 50MB 이하)
4. "IT 문제 있는데 VOC 등록해줘" 대화
5. LLM이 "아래 파일도 첨부할까요?" 확인 시 동의
6. 응답에 "첨부파일: xxx (1개)" 표시 확인
7. Works 1445 applet에서 생성된 게시글 상세 열어 첨부파일 확인
8. 검증 완료 시 `WORKS_APPLET_ID=934`로 복구

## 결정 사항 및 주의점
- **리버스 엔지니어링 기반** — Daou Works 업데이트 시 필드 ID/응답 구조 변경 가능성. 필드 ID는 `WORKS_ATTACHMENT_FIELD` env로 오버라이드 가능하도록 외부화.
- **공식 OpenAPI(`/public/v1/works`)는 첨부 미지원** — SSO 실패 폴백 시 첨부는 반영 안 됨. 이 경우 로그로 명시.
- **applet 934의 첨부 필드 ID 검증은 실제 운영 배포 전 필수** — 1445(테스트앱)와 구조 동일하다고 확인받았으나, 필드 ID 자체가 applet별 랜덤 부여이므로 실제 적용 시 재확인 권장.
- **파일 크기 50MB 상향** — nginx는 이미 50MB 허용 중이라 서버 설정 변경 불필요. 100MB 이상으로 가려면 FastAPI 스트리밍 리팩터링 + nginx 수정 필요.
- **ChromaDB 임베딩 유지** — VOC 첨부된 파일이 ChromaDB에 임베딩된 상태로 남는 것이 이점(LLM이 파일 내용을 이해한 상태로 VOC 본문 자동 작성).

## 후속 수정 (같은 날, 테스트 발견 이슈)

1차 테스트에서 "캡처/paste 이미지를 포함해 VOC 등록 요청" 시 첨부가 아예 반영되지 않던 버그를 수정.

### 증상
- 사용자가 이미지 paste/drag 후 "IT VOC 등록해줘" 요청 → Planner가 `clarify`로 라우팅 → DirectResponseWorker가 "IT VOC는 이미지 첨부 지원 안 함" **환각** 응답
- 이후 "ㅇㅇ" 턴에서 ITSupportWorker가 불려도 `register_works_voc`의 `args_keys`에 `attachments` 자체가 없음 → 첨부 플로우 미실행

### 근본 원인
`chromadb_service.has_session_files(session_id)`가 **ChromaDB 세션 컬렉션 count만** 체크. `/api/v1/upload/image` 엔드포인트는 base64 반환 구조로 **디스크에만** 저장하고 ChromaDB에는 안 넣음 → paste/drag 이미지가 `has_files=False`로 잡힘 → Planner가 "파일 없음" 판단 → clarify 분기.

### 수정 내용
| 파일 | 변경 |
|------|------|
| backend/app/services/chromadb_service.py | `has_session_files(session_id, user_id=None)` 시그니처 확장. ChromaDB 체크 후 fallback으로 `user_uploads/{date}/{user_id}/` 디스크 체크 추가 |
| backend/app/api/routes/chat.py | `has_session_files` 호출 시 `user_id` 함께 전달 |
| backend/app/agents/workers/it_support_worker.py | `_list_uploaded_files`에 업로드 경과 시간 표시(`N분 전`), 최근 10분 이내 파일에 `🆕` 마커. 프롬프트 규칙 6개로 강화(UUID 파일명 인정, "방금 올린 파일" 명시 시 자동 첨부, "첨부 지원 안 함"류 거짓 응답 금지 등) |

### 프롬프트 규칙 핵심
- 🆕 마크 붙은 파일은 현재 대화 주제 관련 가능성 매우 높음 — UUID여도 무시 금지
- 사용자가 "업로드한 이미지/첨부한 스크린샷" 등 명시 → 🆕 파일 자동 attachments 포함, 되묻기 금지
- "이미지 첨부 미지원" 응답은 거짓, 절대 금지

### 아직 남은 과제 (별개 이슈)
- Planner/DirectResponseWorker 수준의 환각 응답 방지 (IT VOC 도메인 지식 없이 추측 답변) — 이번 수정은 has_files 신호 복구로 대부분 우회 가능하지만 근본 해결은 별도 필요

## 후속 수정 2 (같은 날, 2차 테스트 발견 이슈)

1차 후속 수정(has_files 복구) 후 재테스트 시 Planner가 **"스크린샷 분석(user_files) → VOC 등록(it_support)"** 2-task DAG로 분해 → user_files의 `search_user_files`가 ChromaDB 텍스트 검색이라 이미지에는 결과 0 → "파일 찾지 못함" 응답 후 AWAITING_CONFIRM 블록.

### 원인
Planner가 "첨부파일 있음 + 등록 요청" 시 **분석 사전 단계를 과잉으로 끼워넣음**. 실제로는 `register_works_voc`의 `attachments` 파라미터로 MCP가 직접 Works에 파일을 forward하므로 내용 분석 불필요. `search_user_files`는 텍스트 문서 RAG 용으로 이미지에 쓸 수 없음.

### 수정 내용
| 파일 | 변경 |
|------|------|
| backend/app/agents/planner.py | CORE RULE 8 추가 — "쓰기 작업 + 파일 첨부" 조합은 쓰기 워커에게 attachments 파라미터로 직접 위임 (user_files 사전 태스크 금지). Example 6(신규) IT VOC+스크린샷 케이스 few-shot 추가, 기존 Example 6은 7로 재번호 |

### 규칙 핵심
- 사용자가 "등록해줘/업로드해줘/첨부해줘" + 파일 첨부 → 분석 태스크 없이 해당 쓰기 워커 단일 태스크로 라우팅
- 예외: "내 파일 내용 읽고 요약해서 본문에 반영해줘" 같이 **명시적 분석 요청**만 user_files 사전 태스크 허용
- `search_user_files`가 이미지에 동작하지 않는다는 사실을 few-shot에 명시
