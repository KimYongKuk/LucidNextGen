# 2026-04-21 채팅 엔드포인트 JWT 인증 의무화

## 개요

채팅 관련 6개 엔드포인트에 HttpOnly 쿠키 기반 JWT 인증을 의무화하여, 클라이언트가 body/query로 임의 `user_id`를 주입해 타인 데이터에 접근하는 취약점을 차단한다. 이 한 번의 변경으로 `user_id`에 의존하는 모든 Worker(Mail/Approval/Calendar/Reservation/Outline/NAS/ITSupport)가 자동으로 본인 데이터로만 격리된다.

## 배경 / 발견된 취약점

- `ChatRequest.user_id`가 body 필드로 존재하며, 서버가 별도 인증 검증 없이 그 값을 그대로 context에 주입 → Worker들이 "본인 사번"으로 신뢰하고 동작
- 공격자는 브라우저 DevTools로 body의 `user_id`만 바꿔 재전송하면 타인 메일/결재/일정 조회, 타인 세션 삭제 등이 가능했음
- JWT 발급 로직·HttpOnly 쿠키 저장은 이미 존재하지만(`auth.py:_create_token()`, `app/api/auth/login/route.ts`), chat 엔드포인트에 적용되지 않은 상태
- 이 문제는 그룹웨어 계정 관리 API(LFON, 패스워드/OTP 초기화) 연동 사전 논의 중 발견됨 — destructive 액션을 붙이기 전에 반드시 선결해야 할 사안

## 상세 내용

### 1. 신규 Dependency

`backend/app/api/dependencies/auth_jwt.py`

```python
async def get_current_user(auth_token: str | None = Cookie(None)) -> dict:
    # HttpOnly 쿠키 auth_token 에서 JWT 읽음 (JS 접근 불가)
    # jwt.decode 서명/만료 검증
    # 실패 시 401 반환
    return {"empno": payload["empno"], "name": payload.get("name", "")}
```

- 환경변수: `SECRET_KEY`, `ALGORITHM`(기본 HS256) — 기존 auth.py와 동일 키 사용
- 실패 케이스 분기: 토큰 없음 / 만료 / 변조 / 사번 누락 → 각기 적절한 401 메시지

### 2. 적용된 엔드포인트 (6개)

| Method | Path | 인증 전 | 인증 후 |
|--------|------|--------|--------|
| POST | `/v1/chat/message/stream` | body `user_id` 신뢰 | JWT 사번 사용, body `user_id` 무시 |
| GET | `/v1/chat/sessions` | Query `user_id` 신뢰 | JWT 사번 사용 |
| GET | `/v1/chat/sessions/search` | Query `user_id` 신뢰 | JWT 사번 사용 |
| GET | `/v1/chat/sessions/{id}/messages` | Query `user_id` 신뢰 | JWT 사번 사용 |
| DELETE | `/v1/chat/sessions/{id}` | Query `user_id` 신뢰, 소유권 검증 無 | JWT 사번 + **소유권 검증(403)** |
| PATCH | `/v1/chat/sessions/{id}` | 인증 없음 | JWT + 소유권 검증 |
| POST | `/v1/chat/sessions/{id}/pin` | 인증 없음 | JWT + 소유권 검증 |

### 3. 점진적 제거 방침

`request.user_id` 및 Query `user_id` 필드는 **삭제하지 않고 `Optional`로 남김**. 서버 로직에서는 읽지 않는다. 이유:
- 기존 프론트엔드 코드가 필드를 여전히 전송하는 상태에서 Pydantic validation 에러 방지
- 배포 순서 불일치(백엔드 선배포/프론트엔드 후배포) 상황에서 장애 방지
- 1~2주 운영 후 필드 자체 제거 예정

### 4. 소유권 검증 (delete/patch/pin)

DELETE/PATCH/PIN 엔드포인트는 JWT 인증뿐 아니라 `chat_sessions.user_id == authenticated_empno` 검증을 추가:
```python
session_info = chat_log.get_session(session_id)
if session_info.get("user_id") != authenticated_empno:
    raise HTTPException(status_code=403, detail="본인 세션만 수정/삭제할 수 있습니다.")
```

### 5. 프론트엔드 Cookie Forwarding

Next.js API Route는 **서버사이드 fetch**이므로 브라우저 쿠키가 자동 전달되지 않음. 각 route.ts에서 원본 요청의 `cookie` 헤더를 읽어 백엔드 fetch에 명시적으로 forward:

```typescript
const cookieHeader = request.headers.get("cookie") || "";
const res = await fetch(backendUrl, {
  headers: { "Content-Type": "application/json", Cookie: cookieHeader },
});
```

상대경로 `/api/v1/*` 호출(sidebar-history.tsx 등)은 `next.config.js` rewrite 경유 same-origin이므로 브라우저 쿠키 자동 포함 → 수정 불필요.

브라우저에서 백엔드 직접 호출(`use-simple-chat.ts`)은 개발 모드에서 cross-origin이 될 수 있어 `credentials: 'include'` 추가.

## 방어 메커니즘 요약

| 공격 유형 | 방어 |
|-----------|------|
| body/query `user_id` 위조 | 서버에서 무시, JWT 사번만 사용 |
| JWT 쿠키 XSS 탈취 | HttpOnly 플래그로 JS 접근 차단 |
| JWT 변조 | 서명 검증 실패 시 401 |
| JWT 만료 사용 | 만료 검증 후 401 |
| 타인 세션 조작 | 소유권 검증으로 403 |

## 결정 사항 및 주의점

### 결정
- **단계적 필드 제거**: `user_id` 필드 즉시 삭제 대신 서버에서 무시. 1~2주 관찰 후 삭제.
- **HttpOnly 쿠키 방식 고수**: Authorization 헤더 대비 XSS 노출 리스크 낮음. 기존 로그인 플로우와 정합.
- **소유권 검증은 destructive 엔드포인트에만**: 조회 엔드포인트는 JWT 사번을 조회 조건으로 사용해 자동 격리됨.

### 주의점
- `.env` 의 `SECRET_KEY`가 여전히 기본값(`"landf01234567890_..."`)으로 설정되어 있으면 JWT 위조 가능 → 운영 배포 전 강력한 랜덤 키로 교체 필수
- 운영 배포 후 24시간 내 `auth_token` 쿠키 없는 브라우저는 401 받고 재로그인 필요 → 공지 권장
- CORS `allow_origins=["*"]` + `allow_credentials=True` 조합은 브라우저가 일부 상황에서 거부 가능 → 향후 구체적 origin으로 변경 검토
- 본 변경은 **채팅 관련 엔드포인트만** 적용. workspace/upload/image 계열은 별도 작업 필요.
