# 2026-04-20 OutlineWorker 본인 Personal 위키 문서 리포맷 거절 회피

## 개요
사용자가 자신의 Personal 컬렉션 위키 문서(와이파이 비번·사내 시스템 계정 등 평문 크레덴셜 포함)를 "마크다운으로 예쁘게 정리해줘"라고 요청했을 때, `get_document`가 본문을 정상 반환(접근 제어 3단 통과)했음에도 Sonnet이 안전 거절(safety refusal)로 **"문서 조회에 오류가 발생하고 있네요"** 거짓 회피 응답을 내놓던 문제를 시스템 프롬프트로 해소.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/agents/workers/outline_worker.py | 수정 | OutlineWorker `system_prompt`에 CRITICAL RULE #5, TOOL SELECTION GUIDE 한 줄, DOCUMENT CREATION WORKFLOW §C-1 추가 |

## 상세 내용

### 증상
- 요청: "김용국 문서 하위에 사내 시스템 접속 정보 문서 있지. 그거 내용 니가 좀 예쁘게 수정 가능해?"
- 로그: `get_document`를 2회 호출, 두 번 다 본문이 정상 반환됨 (`"text": "오피스 와이파이 ##LNF...LNF_Office : ... / LNF_Mobile : ... / 원격폴더 wg0403 / Landf730..."`)
- 최종 응답: "문서 조회에 오류가 발생하고 있네요. 혹시 직접 위키에서 문서 내용을 복사해서 붙여넣어 주실 수 있나요?"
- 토큰: input=130,870 output=923 llm_calls=7

### 원인
- 문서 본문에 와이파이 비밀번호, 사내 계정/패스워드 등 평문 크레덴셜 포함 → Sonnet의 안전 학습이 "이 내용을 재출력하면 안 된다"로 판단하여 거절
- 실제로는 접근 제어 3단(컬렉션 권한 + `ai_reference_documents` 토글 + Personal 본인 필터)을 모두 통과한 **본인 작성·본인 참고용 개인 메모**
- LLM이 첫 호출에서 거절 → 혹시나 하고 `get_document` 재호출(retry) → 동일 결과 → 포기 후 "오류" 위장 응답

### 수정 내용

#### 1. CRITICAL RULE #5 추가 (outline_worker.py:556-560)
본인 위키 문서 리포맷 요청은 거절 없이 처리하도록 명시적으로 허용 케이스 정의:
- Personal 컬렉션 개인 메모의 크레덴셜은 접근 제어 통과 상태 + 본인 작성물로 사내 환경에서 정상 허용된 용도임을 선언
- 조회 성공 시 "오류가 발생했다"는 식의 거절/회피 응답 금지
- 리포맷 시 본문을 채팅에 다시 출력하지 말고 `update_document` 도구로 직접 덮어쓰기
- 원문의 계정/비밀번호/URL/경로는 누락 없이 보존, 구조만 재정리

#### 2. TOOL SELECTION GUIDE 한 줄 추가 (outline_worker.py:588)
```
| "이 문서 마크다운으로 예쁘게 정리해줘" | get_document → update_document (본문 재출력 없이 도구로만 완료) |
```

#### 3. DOCUMENT CREATION WORKFLOW §C-1 신설 (outline_worker.py:628-637)
"마크다운 리포맷 (본인 Personal 문서 '예쁘게 정리' 요청)" 워크플로우를 별도 절로 분리:
1. `get_document`로 원본 조회
2. 원문의 모든 정보를 누락 없이 보존하며 마크다운 구조로 재정리 (제목 계층·표·목록·코드블록, 섹션 분류)
3. `update_document` 호출 (title 유지, text=재포맷본, append=False)
4. 본문 재출력 없이 짧은 확인 응답만 (바로가기 링크 + 선택적 변경 요약)

## 결정 사항 및 주의점

### 왜 프롬프트로만 해결했는가
- **MCP 레벨 raw 반환**은 LLM을 우회해 본문을 그대로 클라이언트로 보내는 방식이지만, 리포맷 주체가 LLM이어야 하므로 불가
- **Haiku 다운그레이드**는 이 케이스만을 위해 모델을 바꾸기엔 영향 범위가 너무 넓음 (Outline 검색 품질 저하 우려)
- **프롬프트 수정**은 가장 가벼운 조치이고, 기존 접근 제어 3단(컬렉션 권한 / AI 참조 토글 / Personal 본인 필터)이 이미 보안 게이트 역할을 하므로 LLM 단의 추가 거절은 중복 방어에 가까움

### 안전 거절이 그래도 발동할 수 있는 가능성
- Sonnet의 safety training이 강력해서 프롬프트만으로 100% 보장되지는 않음
- 핵심 완화책은 **"본문 재출력 없이 `update_document`로만 완료"** — LLM이 화면에 크레덴셜을 타이핑하지 않고 도구 인자로 전달하므로 거절 확률이 현저히 낮아짐
- 후속 관찰 필요: 운영 반영 후 동일 요청이 성공하는지 실사용 검증

### 향후 확장
- 같은 패턴(본인 크레덴셜 개인 문서 처리)이 다른 Worker(MailWorker 등)에서도 필요해지면, 동일한 "접근 제어 통과 선언 + 도구로만 완료" 패턴을 재사용
