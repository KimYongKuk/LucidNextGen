# 2026-04-15 예약 item_id 할루시네이션 방지

## 개요
그룹웨어 위젯(groupware_embed)에서 회의실 예약 등록이 실패하는 문제를 수정.
LLM이 존재하지 않는 item_id를 할루시네이션하여 LFON API가 500 에러를 반환함.

## 원인 분석

### 증상
- 위젯: `create_reservation(item_id=337)` → LFON 500 "일시적인 오류입니다"
- 본 화면: `create_reservation(item_id=256)` → 성공

### 근본 원인
1. `get_daily_reservations` 출력에 **room ID가 포함되지 않았음** (회의실 이름만 표시)
2. `message_history`에 tool result가 포함되지 않아, 다음 턴에서 LLM이 이전 `find_available_rooms` 결과의 정확한 item_id를 참조 불가
3. LLM이 item_id를 추론하다 **존재하지 않는 337을 생성** (할루시네이션)
4. 성서(asset_id=100) 실제 회의실 ID: [254, 255, 256, 258, 294, 360] — 337은 없음
5. 본 화면에서는 LLM이 item_id를 모르는 것을 인지하고 `get_rooms`를 재호출하여 정확한 ID 확보

### 비결정성
위젯/본 화면 코드 차이가 아닌 LLM 판단 차이. 동일 시나리오에서도 할루시네이션 발생 가능.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/mcp_servers/reservation_mcp_server.py` | 수정 | get_daily_reservations에 item_id 포함 + create_reservation에 item_id 유효성 검증 |

## 상세 내용

### 1. get_daily_reservations 출력에 item_id 추가
- 변경 전: `### 성공-C/R3 - 6명`
- 변경 후: `### 성공-C/R3 - 6명 (item_id=256)`
- LLM이 대화 텍스트에서 정확한 item_id를 참조할 수 있도록 함

### 2. create_reservation에 item_id 유효성 검증
- 예약 실행 전 해당 사업장의 실제 회의실 목록과 대조
- 존재하지 않는 item_id 시 에러 메시지와 함께 올바른 회의실 목록 반환
- `_rooms_cache` 활용 (추가 API 호출 최소화)

## 결정 사항 및 주의점
- `v_user_info_mapping.user_id`와 `go_users.id`는 다른 ID 체계이나, 현재 예약 API에서는 `v_user_info_mapping.user_id`가 동작함 (추가 모니터링 필요)
- `v_reservation_user_mapping` VIEW는 DBA 문서상 생성 완료라 되어있으나 실제로 존재하지 않음 — DBA 확인 필요
