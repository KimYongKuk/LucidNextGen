# 2026-05-06 Outline 위키 임베드 챗 두 번째 메시지 401 fix

## 개요
Outline 위키에 띄운 "출장나온 루시드" 임베드 위젯에서 첫 메시지는 정상 응답, 두 번째 메시지부터 `401 {"detail":"인증 토큰이 없습니다. 로그인이 필요합니다."}` 가 발생하던 버그 수정.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| frontend/app/embed/wiki/page.tsx | 수정 | `widgetAuthToken`/`userId`를 `useMemo([searchParams])` → `useState(() => …)` 1회 캡처로 변경 |

## 상세 내용

### 증상
- 위키 위젯 iframe `src=/embed/wiki?token=…&sid=…` 정상 발급/로드
- 첫 메시지 전송 → **200 OK**, 응답 정상
- 두 번째 메시지 전송 → **401 Unauthorized**
- 새로고침 후 첫 메시지만 다시 정상 → 두 번째부터 같은 증상 반복

### 진단 흐름
1. 백엔드 server.log에 `[WIDGET_AUTH] …` 경고 0건 → `X-Widget-Auth` 헤더가 백엔드에 도달하지 않음을 확인 (widget_auth.py 코드에 진입조차 안 함)
2. 브라우저 Network 탭의 401 fetch 헤더 확인:
   - `Cookie:` 없음 (임베드 컨텍스트라 JWT 쿠키 없음 — 정상)
   - `X-Widget-Auth:` **없음** ← 비정상
   - `Referer: https://lucid.landf.co.kr/chat/541bafd5-13ae-…` ← `/embed/wiki?token=…` 가 아님
   - `chat_mode: "outline_embed"` (위키 임베드에서 보낸 게 맞음)
3. iframe `src` 자체는 정상(`/embed/wiki?token=…&sid=…`)임을 사용자가 DOM 검사로 확인
4. → 임베드 페이지가 첫 메시지 전송 후 **자기 URL을 `/chat/{sessionId}`로 바꿔치기** 했다는 결론

### 근본 원인
[`components/multimodal-input.tsx:149`](../../frontend/components/multimodal-input.tsx#L149) 의 `submitForm` 첫 줄:
```js
window.history.pushState({}, "", `/chat/${chatId}`);
```
([`suggested-actions.tsx:90`](../../frontend/components/suggested-actions.tsx#L90), [`chat.tsx:157`](../../frontend/components/chat.tsx#L157) 도 동일 패턴)

본체 채팅 UI에서는 새 세션 ID를 URL에 노출하기 위한 정상 동작이지만, 임베드 페이지에서는 부작용 발생:

1. iframe 첫 로드: URL = `/embed/wiki?token=…&sid=…`
2. 첫 메시지 입력 → `pushState` 호출 → URL = `/chat/{sessionId}` (token 파라미터 사라짐)
3. 첫 fetch는 이미 호출된 상태라 헤더에 토큰 정상 → 200 OK
4. 응답 후 React 리렌더 → `useSearchParams()`가 새 URL 기준으로 갱신 → `useMemo([searchParams])` 재계산 → `searchParams.get("token") = null` → `widgetAuthToken = undefined`
5. 두 번째 메시지 fetch → `X-Widget-Auth` 헤더 빠짐 → JWT 쿠키도 없으므로 401

### 수정
[`frontend/app/embed/wiki/page.tsx`](../../frontend/app/embed/wiki/page.tsx):

```diff
-import { useEffect, useMemo } from "react";
+import { useEffect, useState } from "react";
 ...
-  const widgetAuthToken = useMemo(() => {
-    return searchParams.get("token") || undefined;
-  }, [searchParams]);
+  const [widgetAuthToken] = useState<string | undefined>(
+    () => searchParams.get("token") || undefined
+  );
 ...
-  const userId = useMemo(() => {
-    return searchParams.get("empno") || getUserId() || "wiki_user";
-  }, [searchParams]);
+  const [userId] = useState<string>(
+    () => searchParams.get("empno") || getUserId() || "wiki_user"
+  );
```

`useState(() => …)`의 init function은 마운트 시 1회만 호출되므로, 이후 `searchParams`가 어떻게 바뀌어도 첫 진입 시점의 token/empno가 영구 보존됨.

## 결정 사항 및 주의점
- **그룹웨어 임베드는 이미 같은 패턴**: [`embed/gw/page.tsx:9-15`](../../frontend/app/embed/gw/page.tsx#L9-L15) 가 `useState(() => …)` 로 작성돼 있어 이 버그를 안 겪음. 위키 임베드만 `useMemo` 패턴으로 작성된 게 원인이었음 → 두 임베드 페이지의 토큰 캡처 패턴이 일치하게 됨.
- **근본 fix는 별도 작업으로 미룸**: `MultimodalInput`/`SuggestedActions`/`Chat`이 임베드 컨텍스트에서 `pushState`를 호출하는 것 자체가 부자연스럽다. `chatMode` prop을 받아 `outline_embed`/`groupware_embed`일 때 `pushState` 스킵하는 가드를 추가하면 더 깔끔하지만, 이번 변경의 영향 범위를 최소화하기 위해 현재는 캡처 시점 변경만으로 마무리.
- **최근 lfon/AD 인증 변경과 무관**: `pushState` 동작은 이전부터 있었고, 위키 임베드의 `useMemo` 패턴도 이전부터 있었음. 단지 사용자가 두 번째 메시지를 보내는 빈도가 낮아서 이번에 처음 발견된 것으로 추정.
