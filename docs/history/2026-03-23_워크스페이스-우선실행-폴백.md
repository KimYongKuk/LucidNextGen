# 2026-03-23 워크스페이스 우선 실행 + 전문 워커 폴백

## 개요
워크스페이스에서 전문 워커(corp_rag, it_support 등)로 분류된 요청이 강제로 user_files로 오버라이드되어 전문 워커가 아예 호출되지 않던 문제를 해결. user_files를 1순위로 실행하되, 워크스페이스 문서에서 결과를 찾지 못하면(NO_RESULTS) 원래 분류된 전문 워커로 자동 폴백하는 구조로 변경.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/agents/intent_classifier.py | 수정 | `_apply_overrides`에서 corp_rag/it_support/acct_support → user_files 강제 오버라이드 제거 |
| backend/app/agents/orchestrator.py | 수정 | Phase 1.8 (workspace-first 라우팅) + Phase 4.5 (workspace 폴백) 추가 |

## 상세 내용

### 변경 전 (기존)
```
워크스페이스(파일有) + intent=CORP_RAG
  → _apply_overrides에서 user_files로 강제 변환
  → UserFilesWorker만 실행
  → NO_RESULTS여도 CORP_RAG 워커 호출 불가
```

### 변경 후 (신규)
```
워크스페이스(파일有) + intent=CORP_RAG
  → Phase 1.8: user_files로 교체 (원래 인텐트 보존)
  → Phase 4: UserFilesWorker 실행 (스트리밍)
  → Phase 4.5: NO_RESULTS 감지 시 CorpRAGWorker 폴백
  → Phase 6: CorpRAGWorker도 NO_RESULTS면 fallback_intent 워커 추가 실행
```

| IT_SUPPORT | O | 〃 |
| ACCT_SUPPORT | O | 〃 |
| WEB_SEARCH | O | 〃 |
| BOARD | O | 〃 |

### Orchestrator 새 Phase 흐름
```
Phase 0: Memory 로드
Phase 1: Intent Classification
Phase 1.5: CLARIFY 처리
Phase 1.8: Workspace-first 라우팅 (NEW)
  → workspace_has_files + 전문 인텐트 → user_files로 교체, 원래 인텐트 보존
Phase 2: Worker Dispatch
Phase 3: Build Messages
Phase 4: Worker Streaming (user_files 실행)
Phase 4.5: Workspace fallback (NEW)
  → NO_RESULTS → 원래 전문 워커로 폴백, intent/worker_name 복원
Phase 5: HANDOFF Check
Phase 6: Fallback Check (기존)
```

### 타이밍 이벤트
`orchestrator_timing`에 `workspace_fallback_ms` 필드 추가 (workspace 폴백 실행 시에만).

## 결정 사항 및 주의점
- `direct` → `user_files` 오버라이드는 intent_classifier에서 유지 (워크스페이스에 파일이 있으면 문서 기반 응답이 맞음)
- 최악의 경우 3단계 순차 실행(user_files → 전문 워커 → fallback)이 가능하나, 대부분 1단계에서 종료
- MAIL/APPROVAL 등 외부 시스템 워커는 workspace-first 대상에서 제외 (워크스페이스 문서와 무관)
- quick_classify 경로는 workspace_has_files=True일 때 LLM 위임으로 넘어가므로 fallback_intent=None 문제는 발생하지 않음
