# 2026-03-31 자체 로그인 인증 시스템

## 개요
기존 SSO(암호화 사번 URL 파라미터) 방식과 병행하여, ID/PW 기반 자체 로그인 인증 시스템을 추가했다. 그룹웨어 비밀번호와 독립적인 자체 비밀번호를 사용하며, JWT 토큰으로 세션을 관리한다. 보안을 위해 이메일 인증 기반 셀프 비밀번호 설정 방식을 채택하고, 로그아웃 기능까지 완성했다.

## 상세 내용

### 전체 인증 플로우
```
[신규 사용자 — 셀프 비밀번호 설정]
  1. /login → "처음이신가요?" 클릭
  2. 사내 이메일 입력 → POST /api/auth/request-setup
  3. TIMS DB에서 사원 정보 조회 (v_user_info_mapping: login_id → empno, name)
  4. setup_tokens 테이블에 1회용 토큰 저장 (24시간 유효)
  5. SMTP로 비밀번호 설정 링크 이메일 발송
  6. 사용자가 이메일 링크 클릭 → /setup?token=xxx
  7. 비밀번호 입력 → POST /api/auth/setup-password
  8. users 테이블에 계정 생성 + 토큰 소멸
  9. /login에서 로그인

[기존 사용자 — 로그인]
  POST /api/auth/login (login_id, password)
    → bcrypt 검증 → JWT 발급
    → 쿠키 4개 설정: empno, login_id, user_name, auth_token(httpOnly)

[로그아웃]
  POST /api/auth/logout → 쿠키 4개 삭제 → /login 리다이렉트

[SSO 로그인] (기존 유지)
  ?empno=<encrypted> → 복호화 → empno 쿠키 → 기존 동작
```

### Middleware 인증 우선순위
1. SSO empno URL 파라미터 (최우선)
2. JWT auth_token 쿠키 (jose 라이브러리로 서버사이드 검증)
3. empno 쿠키 (SSO 레거시 호환)
4. 인증 없음 → /login 리다이렉트 (/unauthorized 대신)
5. /login, /setup, /unauthorized 페이지는 인증 스킵
6. logo.png, logo.svg 정적 리소스는 matcher에서 제외

### Backend API
| 엔드포인트 | 메서드 | 용도 |
|------------|--------|------|
| `/api/auth/login` | POST | 로그인 (login_id + password → JWT + empno/login_id/name 반환) |
| `/api/auth/register` | POST | 사용자 등록 (관리자용) |
| `/api/auth/change-password` | POST | 비밀번호 변경 |
| `/api/auth/verify-token` | POST | JWT 토큰 검증 |
| `/api/auth/request-setup` | POST | 이메일 인증 요청 (TIMS DB 조회 + 토큰 생성 + 메일 발송) |
| `/api/auth/setup-password` | POST | 토큰으로 비밀번호 설정 (계정 생성) |
| `/api/auth/decrypt` | POST | SSO 사번 복호화 (기존) |

### 쿠키 구성
| 쿠키 | httpOnly | 용도 |
|------|----------|------|
| `empno` | false | 사번 (기존 시스템 호환, 클라이언트에서 getUserId() 등) |
| `login_id` | false | 로그인 ID (사이드바 표시용) |
| `user_name` | false | 사용자 이름 (사이드바 표시용) |
| `auth_token` | true | JWT 토큰 (middleware 검증용, XSS 방어) |

### 사이드바 사용자 표시
- `app/(chat)/layout.tsx`에서 쿠키 3개(empno, login_id, user_name) 읽어서 `AppSidebar user={...}` 전달
- `SidebarUserNav`에 login_id 표시, 드롭다운 메뉴에 테마 토글 + 로그아웃 버튼
- 기존 `user={undefined}` → 실제 쿠키 기반 사용자 정보로 변경

### DB 테이블
```sql
-- 사용자
CREATE TABLE users (
    empno VARCHAR(20) PRIMARY KEY,
    login_id VARCHAR(50) NOT NULL UNIQUE,
    name VARCHAR(50) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,  -- bcrypt (rounds=12)
    is_active TINYINT(1) DEFAULT 1,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 1회용 설정 토큰
CREATE TABLE setup_tokens (
    id INT AUTO_INCREMENT PRIMARY KEY,
    token VARCHAR(64) NOT NULL UNIQUE,
    empno VARCHAR(20) NOT NULL,
    login_id VARCHAR(50) NOT NULL,
    name VARCHAR(50) NOT NULL,
    email VARCHAR(100) NOT NULL,
    used TINYINT(1) DEFAULT 0,
    expires_at DATETIME NOT NULL,  -- 24시간
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### JWT 설정
- 알고리즘: HS256
- 만료: 24시간 (ACCESS_TOKEN_EXPIRE_MINUTES=1440)
- Payload: empno, name, exp, iat
- 검증: Frontend middleware에서 `jose` 라이브러리로 직접 검증 (백엔드 호출 없음)

### 이메일 발송
- 기존 `EmailService` (SMTP) 재사용
- 발신: intmail@landf.co.kr (Lucid AI)
- HTML 템플릿: 비밀번호 설정 버튼 + 24시간 유효 안내
- TIMS DB 조회: `v_user_info_mapping` (login_id → employee_number, name)

## 결정 사항 및 주의점
- **그룹웨어 연동 제외**: OTP 요구 + 비밀번호 초기화 사용 시나리오 충돌로 자체 비밀번호 관리 결정
- **셀프 가입**: 사내 이메일 인증 기반 — TIMS DB(v_user_info_mapping)에 등록된 임직원만 가능
- **보안**: 관리자가 비밀번호를 알 수 없음 (이메일 토큰 → 본인이 직접 설정)
- **토큰 1회용**: setup_tokens.used 플래그로 재사용 방지
- **쿠키 4개**: empno(기존 호환) + login_id(UI) + user_name(UI) + auth_token(보안)
- **로그아웃**: 쿠키 4개 전부 삭제 + /login 리다이렉트
- **SITE_URL**: .env에 설정 필요 (개발: `http://192.168.90.30:3099`, 운영: `https://lucid.landf.co.kr:9090`)
- **정적 리소스**: middleware matcher에서 logo.png, logo.svg 제외 (로그인 페이지 로고 표시)
