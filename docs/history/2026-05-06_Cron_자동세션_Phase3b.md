# 2026-05-06 Cron 자동 호출 인프라 (Phase 3b — 백엔드 + 등록 폼 + 사이드바)

## 개요
워크스페이스에 부착된 외부 Agent(MISO Workflow 등)를 cron 스케줄로 자동 호출하고, 결과를 **새 채팅 세션 + 알림 deep-link**로 사용자에게 통보하는 인프라 구축. 사용 케이스: "매일 9시 인사이트 리포트 자동 생성".

## 상세 내용

### 1. 마이그레이션 (수동 실행 필요)
```sql
-- backend/migrations/add_cron_session_metadata.sql
ALTER TABLE chat_sessions
    ADD COLUMN auto_generated TINYINT(1) NOT NULL DEFAULT 0,
    ADD COLUMN source_agent_id VARCHAR(36) NULL,
    ADD INDEX idx_auto_user (auto_generated, user_id),
    ADD INDEX idx_source_agent (source_agent_id);
```
- 기존 세션은 자연스럽게 `auto_generated=0`, `source_agent_id=NULL`이 되어 영향 없음.
- DBeaver 등으로 1회 실행 필요 (서버 시작 전).

### 2. CronRunner — 단일 자동 호출 처리기
입력: `(workspace_uuid, agent_id)` 두 개만 — self-contained.

흐름:
1. workspace + workspace_agents에서 등록자(`attached_by_user_id`) 조회 → cron 실행 user_id로 사용
2. agents 매니페스트 조회 + `triggers[0].prompt` 읽기 (없으면 default `"🔄 자동 실행: 정기 워크플로우를 실행해주세요."`)
3. 새 `session_id`(`cron_<uuid24>`) 발급, `chat_sessions` INSERT (`auto_generated=1`, `source_agent_id`)
4. `external_agent_router.instantiate_worker_for_agent(agent)` → MisoWorker (또는 후속 Runner/Webhook)
5. `worker.stream_response(messages=[HumanMessage(prompt)], context={user_id, workspace_id, session_id, auto_generated:true})` 실행, `on_chat_model_stream` chunk 수집
6. `chat_log_service.save_chat_log` — user/assistant 메시지 1쌍 저장 (`metadata.auto_generated=true`, `metadata.source_agent_id`, `metadata.elapsed_ms`)
7. `user_notifications` INSERT — `type='schedule_done'`, `link_url=/chat/{session_id}` (알림 클릭 시 자동 라우팅)
8. 실패 케이스도 동일 알림(⚠ 자동 실행 실패) → 사용자가 cron 깨졌음을 인지

### 3. AgentCronScheduler — APScheduler 매니저
- 싱글턴(`get_agent_cron_scheduler()`).
- `AsyncIOScheduler` 백엔드 사용 (다른 스케줄러와 동일).
- Job ID: `cron_{workspace_uuid}_{agent_id}` — `(workspace, agent)` 쌍 1개 유일.
- Hook 4개:
  - `on_workspace_attach(ws, agent)`: 매니페스트 schedule trigger 있으면 등록
  - `on_workspace_detach(ws, agent)`: job 제거
  - `on_agent_manifest_changed(agent)`: 부착된 모든 워크스페이스 job 갱신 (status≠active이면 모두 해제)
  - `start()` / `reload_all()`: 서버 시작 시 active agents × workspace_agents 전체 일괄 등록
- `misfire_grace_time=600` (10분) — 잠깐 서버 죽었다 살아도 누락 없이 실행.
- 환경변수 `AGENT_CRON_ENABLED=false`로 즉시 차단 가능.

### 4. agent_service hook 통합
- `attach_to_workspace()` 끝부분에 `on_workspace_attach` 호출
- `detach_from_workspace()`에 `on_workspace_detach` 호출
- `update_agent()`에 `on_agent_manifest_changed` 호출 (PATCH 시 status가 pending_review로 떨어지므로 자동 해제됨)
- `change_status()`에 `on_agent_manifest_changed` 호출 (운영자 승인 → active 시 cron 등록)
- `soft_delete_agent()`에 `on_agent_manifest_changed` 호출 (모든 부착의 cron 해제)
- 모두 `non-fatal` — cron 등록 실패해도 본 작업(attach 등)은 성공.

### 5. 매니페스트 schedulePrompt 필드
폼에 cron 프리셋이 `none`이 아닐 때만 노출되는 입력란 추가. 매니페스트의 `triggers[0].prompt`로 직렬화.

```ts
// triggers 구조
[{ type: "schedule", cron: "0 9 * * *", timezone: "Asia/Seoul", prompt: "어제 매출 데이터를 분석..." }]
```

분단위 cron(`*` 또는 `*/N`로 시작) 등록 시 주황색 경고 박스 노출 — "채팅 세션이 빠르게 늘어납니다".

## 결정 사항 및 주의점

- **단일 schedule 가정**: 매니페스트 `triggers` 배열에서 첫 번째 `type=schedule`만 사용. 후속에 다중 schedule 필요시 확장.
- **cron user_id = attached_by_user_id**: 워크스페이스에 agent를 부착한 사람이 cron 결과 수신자. 워크스페이스 owner와 다를 수 있음 (멀티 사용자 워크스페이스).
- **다중 워크스페이스 다른 cron 미지원 (현재)**: 한 agent가 여러 워크스페이스에 부착되어도 모두 매니페스트의 같은 cron 사용. 후속에 `workspace_agents.cron_override` 컬럼 검토.
- **PATCH → 자동 cron 해제**: 매니페스트 수정 시 status가 `pending_review`로 떨어지므로 `on_agent_manifest_changed`가 호출되어 cron 해제됨. 운영자가 active로 승인하면 다시 `change_status` → `on_agent_manifest_changed`로 재등록. 연속성 보장.
- **알림 라우팅**: notification-inbox는 이미 `item.link` → `router.push(/chat/{id})` 패턴 ([inbox-drawer.tsx:74-77](frontend/components/notification-inbox/inbox-drawer.tsx#L74-L77)). 추가 frontend 변경 없이 deep-link 동작.
- **사이드바 시각 구분 완료**: cron 자동 세션은 사이드바 "🔄 자동 실행" 별도 그룹 + 시계 아이콘. 시간(Today/Past) 그룹과 섞이지 않아 일반 대화와 구분됨. `auto_generated` 컬럼이 백엔드 SELECT → API forward → `groupChatsByDate` → ChatItem prop까지 일관되게 전달.
- **분단위 cron 차단 안 함**: 사용자가 정말 필요할 수 있어(임상/모니터링) 경고만 노출 + 실 등록 허용. 운영 부담은 사용자 자기 책임.

## 서버 재시작 후 동작 확인 체크리스트
1. 마이그레이션 SQL 실행 (DBeaver) — `chat_sessions.auto_generated/source_agent_id` 컬럼 생성
2. 백엔드 재시작 → `[STARTUP] Agent Cron Scheduler starting...` 로그 확인
3. `[AgentCronScheduler] reload_all: N job(s) registered` 출력 — 기존 active agents 중 schedule 있는 N개 등록
4. 새 cron Agent 등록 (스케줄 프리셋 + 자동 실행 프롬프트) → 워크스페이스 attach
5. 등록한 cron 시각 도달 시 `[AgentCronScheduler] dispatch done ws=... agent=... status=success session=cron_...` 로그
6. 사용자 채팅 사이드바에 새 세션(`🔄 {agent_name} 자동 실행`) 출현 + 알림함에 통보 → 클릭 시 해당 세션으로 이동
