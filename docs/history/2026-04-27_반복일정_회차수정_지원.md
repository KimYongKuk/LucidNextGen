# 2026-04-27 반복 일정 회차 수정 지원

## TL;DR
반복 일정의 한 회차 수정 시 LFON 캘린더 API가 `500 internal.error` 반환하던 이슈 수정. `update_event`에 `recur_change_type` 파라미터 추가 + event_id에 `_타임스탬프`가 포함된 인스턴스 ID인 경우 PUT URL에 `?recurChangeType=...` 쿼리 자동 부착. `delete_event`는 이미 동일 패턴이었으나 update에만 빠져 있던 것.

`event_id`의 underscore 뒤 `1777374000000`은 ms 타임스탬프(2026-04-29 18:00 KST 인스턴스 시작 시각). DaouOffice 캘린더 반복 일정의 단일 회차 식별자 형식.

## 수정 상세

### MCP 서버 (`calendar_mcp_server.py`)
- `update_event` signature에 `recur_change_type: str = "this"` 추가
  - "this" (이 회차만, 기본 — 가장 안전)
  - "all" (전체 회차)
  - "following" (이 회차부터 이후 전체)
- PUT URL 구성:
  ```python
  is_recurrence_instance = "_" in event_id
  path = f"/api/calendar/{calendar_id}/event/{event_id}"
  if is_recurrence_instance:
      path += f"?recurChangeType={recur_change_type}"
  ```
- 단일 일정(`_` 없음)은 기존 동작 유지하므로 회귀 위험 없음
- `_log_to_file`로 인스턴스 여부 + recur_change_type 기록 → 향후 진단 용이

### CalendarWorker 프롬프트
"일정 수정" 섹션에 4번 항목 추가:
> **반복 일정 주의**: event_id에 "_타임스탬프"가 포함되면 반복 일정의 한 회차임 (예: "403063_1777374000000").
> 사용자에게 "이번 회차만/전체/이후 전체" 확인 후 `recur_change_type` 명시.
> 기본 "this"가 안전하지만 사용자 의도가 모호하면 반드시 확인할 것.

"일정 삭제"에도 동일한 회차 범위 명시 안내 추가 (delete_event는 이미 `delete_type` 파라미터 보유).

## 결정 사항 및 주의점

- **기본값 "this"**: Google/Outlook 캘린더 UX 관례상 사용자가 "수정"이라 말할 때는 보통 단일 회차 의도. 데이터 손상 위험도 가장 적음. delete_event는 docstring상 기본값이 "all"인데 그건 추후 일관성 정리 검토 대상(이번 변경 범위 밖).
- **`is_own` 검증과 creator 검증은 그대로**: 사용자 본인 캘린더 + 본인이 등록한 일정만 수정 허용 정책 유지.
- **GET으로 받은 event_data를 그대로 PUT 보내는 구조도 유지**: LFON이 인스턴스 ID + recurChangeType으로 알아서 분기 처리하므로 payload 변환 불필요.
- **단일 일정 회귀 가능성**: `_`가 일반 event_id에 들어갈 일은 없음(LFON event_id는 숫자 ID). 따라서 가드는 안전. 만약 미래에 LFON ID 포맷이 바뀌면 검토 필요.
- **검증 포인트**: 배포 후 운영 로그에서 `update_event 요청: ... recur_instance=True recur_change_type=...` 라인이 찍히는지, PUT이 200으로 끝나는지 확인.

## 2차 보강 (요일 변경 + 마스터 ID 가드)

### 재현 시나리오 (08:09 ~ 08:10, A2304013)
사용자 요청: **"'테스트' 반복 일정을 매주 화요일 → 매주 수요일 20:00~21:00으로 변경 + 장욱진 참석자 추가"** (Planner 로그에서 확인)

**1차 호출 (08:09:10) — 200 OK인데 사용자가 보기엔 요일 변경 실패**
- `event_id='403081_1777374000000'` (4/28 화요일 20시 인스턴스 = 첫 회차)
- `start_time='2026-04-29T20:00:00.000+09:00'` (4/29 수요일 20시 — 정확)
- `end_time='2026-04-29T21:00:00.000+09:00'`
- `add_attendee_names='장욱진'`
- `recur_change_type='all'`
- args_keys에 **`recurrence` 없음** ← 핵심
- LFON 응답: 200 OK

`update_event` 코드는 GET으로 받은 `event_data` 전체를 복사한 뒤 변경 필드만 덮어쓰는 구조:
```python
updated = {**event_data}  # recurrence(매주 화요일) 포함
if start_time: updated["startTime"] = start_time  # 4/29 수요일로
if recurrence: updated["recurrence"] = recurrence  # 안 들어와서 미실행
```
LFON이 시리즈 update 시 RRULE을 우선 적용 → "매주 화요일 20시"로 시간만 변경됨. 사용자 입장: 요일 변경 실패로 보임.

**2차 호출 (08:10:30) — 마스터 ID로 호출 → LFON 500**
- `event_id='403081'` (마스터 ID, underscore 없음)
- `recurrence` 포함 (LLM이 1차 실패 후 학습)
- LFON 응답: 500 "일시적인 오류입니다." (internal.error)

LFON은 반복 일정의 마스터 ID에 대한 직접 PUT을 거부함. 단일 일정 update와 마스터 ID update가 모두 underscore 없는 ID라 코드만으로는 구분 불가하던 상태.

### 수정 1 — recurrence 파라미터 사용 안내 (docstring + 시스템 프롬프트)

`backend/app/mcp_servers/calendar_mcp_server.py` `update_event` docstring의 `recurrence` 항목:
```
**반복 일정의 요일/주기 자체를 변경할 때는 startTime만 옮기지 말고
이 파라미터에 새 RRULE을 함께 전달해야 함**.
예: 매주 화요일 → 매주 수요일로 변경 시
    recurrence="FREQ=WEEKLY;BYDAY=WE;UNTIL=..." + start_time=<수요일 ISO>
startTime만 수요일로 바꾸고 recurrence를 안 보내면 기존 RRULE(매주 화요일)이
그대로 적용되어 시간만 바뀐 듯한 결과가 됩니다.
```

`backend/app/agents/workers/calendar_worker.py` 시스템 프롬프트 "일정 수정" 섹션:
- 4번에 마스터 ID 직접 호출 금지 규칙 추가
- 5번 신설: 요일/주기 변경 시 RRULE 동반 전달 + 구체 예시 + 시간만 변경 시 recurrence 불필요 구분

### 수정 2 — 마스터 ID 직접 호출 가드 (회귀 위험 0)

`update_event`에 GET 응답 기반 분기 추가:
```python
event_has_recurrence = bool(event_data.get("recurrence"))
is_recurrence_instance = "_" in event_id
if event_has_recurrence and not is_recurrence_instance:
    return ("오류: 반복 일정의 마스터 ID로는 직접 수정할 수 없습니다. "
            "get_calendar_events로 변경할 회차의 인스턴스 ID(예: '403081_<타임스탬프>')를 "
            "확인한 뒤 해당 ID로 다시 호출하세요. ...")
```

3가지 경로별 동작:
| 케이스 | event_id | recurrence | 동작 |
|---|---|---|---|
| 단일 일정 | `403080` | 없음 | 기존대로 PUT (회귀 0) |
| 반복 인스턴스 | `403081_177...` | 있음 (참고용) | 기존대로 ?recurChangeType 부착 |
| 반복 마스터 | `403081` | 있음 | 안내 메시지 반환 → LLM이 인스턴스 ID로 재시도 |

**필드명 검증**: 코드 내 `event.get("recurrence", "")` (line 777), `payload["recurrence"]` (line 971), `updated["recurrence"]` (line 1124) 등 다수 위치에서 LFON 응답·요청에 `recurrence` 키를 사용 중. 필드명 확정으로 임시 진단 로그 없이 즉시 가드 가능.

`_log_to_file`에 가드 발동 시 `"update_event 가드: 반복 일정 마스터 ID 직접 호출 차단"` 기록 → 향후 LLM이 가드에 걸리는 빈도 모니터 가능.

### 회귀 위험 분석
- 단일 일정 update: `event_data.get("recurrence")`가 None/빈문자열/빈dict → falsy → 가드 통과 → 영향 0
- 반복 인스턴스 update: `is_recurrence_instance=True` → 가드 통과 → 기존 동작
- 반복 마스터 update: 기존엔 LFON 500이었는데 이제 명확한 안내 메시지로 LLM 재시도 유도 → 사용자 응답 품질 개선

### 검증 포인트
1. 단일 일정 수정 (예: 시간만 변경) — 정상 200 응답 유지되는지
2. 반복 일정의 한 회차 수정 — 기존 동작 유지되는지
3. 사용자가 "매주 X요일 → 매주 Y요일" 자연어 요청 시 LLM이 `recurrence`+`start_time` 둘 다 보내는지 (`args_keys`에 recurrence 포함되는지)
4. LLM이 마스터 ID로 호출하는 케이스 발생 시 가드 메시지가 LLM에 정확히 반환되어 인스턴스 ID 재호출로 이어지는지

## 3차 보강 (recur_change_type 기본값 "this" → "all", LFON 검증값 정렬)

### 재현 (16:54, A2304013)
2차 보강 배포 직후 사용자 테스트에서 회귀 확인. LLM이 5개 인스턴스에 대해 단건 회차 수정 시도(`recur_change_type="this"`) — calendar_mcp.log 발췌:
```
PUT .../event/403446_1777770000000?recurChangeType=this → 500 "일시적인 오류입니다."
PUT .../event/403446_1778979600000?recurChangeType=this → 500
PUT .../event/403446_1780189200000?recurChangeType=this → 500
PUT .../event/403446_1778374800000?recurChangeType=this → 500
PUT .../event/403446_1779584400000?recurChangeType=this → 500
PUT .../event/403446_1779584400000?recurChangeType=all  → 200 OK
```
**즉 LFON DaouOffice는 `recurChangeType=this`/`following` enum을 거부하고 `all`만 인정**. 2차 보강에서 default를 "this"로 잡은 게 LFON 실제 정책과 안 맞았던 것. delete_event는 처음부터 default="all"이라 정상 동작 중이었는데, update_event에서 단건 수정 의도를 보존하려고 잡은 default가 LFON 정책 미확인 상태에서 결정된 게 원인.

### 부작용
1. 사용자가 "한 회차만 옮겨줘" 의도로 호출했을 때 5번 모두 500 응답 → LLM이 다른 우회(삭제 후 재등록) 시도하거나 사용자에게 "1건만 성공"이라는 잘못된 메시지 출력.
2. 마지막 1건이 `all`로 200 OK 받았으나 사용자 화면에 미반영 (LFON 캐시/UI 문제로 추정, 별개 이슈).

### 수정
1. **`recur_change_type` default를 `"this"` → `"all"`로 변경**: delete_event와 일관, LFON 검증값.
2. **docstring 보강**: "this/following은 LFON 미지원 (500 internal.error 거부). 'all'만 사용. 단건 변경 의도여도 'all'로 호출하거나 그룹웨어 직접 처리 안내".
3. **CalendarWorker 시스템 프롬프트 보강**: 4번 항목 전체 재작성 — `recur_change_type="all"` 강제, "this" 사용 금지, 단건 의도면 그룹웨어 직접 안내, 회차마다 반복 호출 X(시리즈 한 번 호출).
4. **명확한 LFON 500 처리**: PUT 응답이 500 + recur_change_type이 this/following이면 일반 "일시적인 오류" 메시지 대신 LLM에게 정확한 대안 안내 반환:
   ```
   "LFON 그룹웨어가 반복 일정 단건 수정(recur_change_type='this')을 지원하지 않습니다.
    (a) recur_change_type='all'로 전체 회차에 동일 변경 적용,
    (b) 사용자에게 그룹웨어 캘린더에서 해당 회차만 직접 수정하도록 안내."
   ```
   LLM이 헛된 우회(삭제후재등록 등) 시도하지 않고 사용자에게 명확히 안내하도록 유도. calendar_mcp.log에도 `update_event 단건 수정 거부` 라인 기록.

### 회귀 위험 분석
- **단일 일정 수정**: `is_recurrence_instance=False`이므로 recurChangeType 쿼리 자체가 안 붙음. 영향 0.
- **반복 인스턴스 update_event 호출 (recur_change_type 미지정)**: default "all"이 적용되어 LFON 검증값과 일치 → 200 OK 정상 (이전 default "this"는 500이었으니 오히려 회복).
- **반복 인스턴스 update_event 호출 (recur_change_type="this" 명시 시)**: LFON 500 → 친절한 안내 메시지 반환 → LLM 재시도 또는 사용자 안내. 사용자 입장 응답 품질 개선.
- **delete_event**: 변경 없음. 기존 default "all" 유지.

### 미해결 (별개 이슈)
- 마지막 1건의 `recurChangeType=all` PUT이 LFON 200 OK 받았으나 사용자가 그룹웨어 화면에서 변경 미반영 확인. LFON 내부 캐시/지연/UI refresh 문제로 추정. 우리 측에서 detect/우회 어려움. 추후 LFON 담당자에게 확인 필요.
