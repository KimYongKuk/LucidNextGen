# 2026-04-27 반복 일정 회차 수정 지원

## TL;DR
반복 일정의 한 회차 수정 시 LFON 캘린더 API가 `500 internal.error` 반환하던 이슈 수정. `update_event`에 `recur_change_type` 파라미터 추가 + event_id에 `_타임스탬프`가 포함된 인스턴스 ID인 경우 PUT URL에 `?recurChangeType=...` 쿼리 자동 부착. `delete_event`는 이미 동일 패턴이었으나 update에만 빠져 있던 것.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/mcp_servers/calendar_mcp_server.py | 수정 | `update_event`에 `recur_change_type` 파라미터 추가, 인스턴스 ID 감지 시 `?recurChangeType=...` 쿼리 부착, `_log_to_file`로 호출 컨텍스트 기록 |
| backend/app/agents/workers/calendar_worker.py | 수정 | 시스템 프롬프트 "일정 수정/삭제" 섹션에 반복 일정 인스턴스 식별·확인 가이드 추가 |

`event_id`의 underscore 뒤 `1777374000000`은 ms 타임스탬프(2026-04-29 18:00 KST 인스턴스 시작 시각). DaouOffice 캘린더 반복 일정의 단일 회차 식별자 형식.

## 수정 상세

### MCP 서버 (`calendar_mcp_server.py`)
- `update_event` signature에 `recur_change_type: str = "this"` 추가
  - "this" (이 회차만, 기본 — 가장 안전)
  - "all" (전체 회차)
  - "following" (이 회차부터 이후 전체)
- PUT URL 구성:
  ```python
  is_recurrence_instance = "_" in event_id
  path = f"/api/calendar/{calendar_id}/event/{event_id}"
  if is_recurrence_instance:
      path += f"?recurChangeType={recur_change_type}"
  ```
- 단일 일정(`_` 없음)은 기존 동작 유지하므로 회귀 위험 없음
- `_log_to_file`로 인스턴스 여부 + recur_change_type 기록 → 향후 진단 용이

### CalendarWorker 프롬프트
"일정 수정" 섹션에 4번 항목 추가:
> **반복 일정 주의**: event_id에 "_타임스탬프"가 포함되면 반복 일정의 한 회차임 (예: "403063_1777374000000").
> 사용자에게 "이번 회차만/전체/이후 전체" 확인 후 `recur_change_type` 명시.
> 기본 "this"가 안전하지만 사용자 의도가 모호하면 반드시 확인할 것.

"일정 삭제"에도 동일한 회차 범위 명시 안내 추가 (delete_event는 이미 `delete_type` 파라미터 보유).

## 결정 사항 및 주의점

- **기본값 "this"**: Google/Outlook 캘린더 UX 관례상 사용자가 "수정"이라 말할 때는 보통 단일 회차 의도. 데이터 손상 위험도 가장 적음. delete_event는 docstring상 기본값이 "all"인데 그건 추후 일관성 정리 검토 대상(이번 변경 범위 밖).
- **`is_own` 검증과 creator 검증은 그대로**: 사용자 본인 캘린더 + 본인이 등록한 일정만 수정 허용 정책 유지.
- **GET으로 받은 event_data를 그대로 PUT 보내는 구조도 유지**: LFON이 인스턴스 ID + recurChangeType으로 알아서 분기 처리하므로 payload 변환 불필요.
- **단일 일정 회귀 가능성**: `_`가 일반 event_id에 들어갈 일은 없음(LFON event_id는 숫자 ID). 따라서 가드는 안전. 만약 미래에 LFON ID 포맷이 바뀌면 검토 필요.
- **검증 포인트**: 배포 후 운영 로그에서 `update_event 요청: ... recur_instance=True recur_change_type=...` 라인이 찍히는지, PUT이 200으로 끝나는지 확인.
