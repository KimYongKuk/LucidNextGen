# 2026-04-21 그룹웨어 위젯 암호화 토큰 인증

## 개요

그룹웨어 iframe 임베드 위젯(`lucid-chat-widget-gw.js`)이 URL 쿼리로 평문 사번(`?empno=A2304013`)을 전달하던 구조를, AES 암호화 토큰(`?token=<encrypted>`) 방식으로 전환. JWT 쿠키 인증 도입으로 위젯이 동작 불능 상태가 되는 것을 방지하면서, 위젯 경로도 본 웹 UI와 동등한 수준의 보안을 제공한다.

## 배경

오늘 도입한 채팅 엔드포인트 JWT 인증([2026-04-21_채팅_엔드포인트_JWT_인증.md](2026-04-21_채팅_엔드포인트_JWT_인증.md))은 HttpOnly 쿠키 기반이라, 그룹웨어 iframe에서 사용하는 위젯에는 쿠키가 없어 동작 불가. 위젯은 반드시 동작해야 한다는 요구사항에 따라 iframe 경로 전용 인증 체계가 필요.

기존 위젯 구조의 취약점: iframe URL `?empno=A2304013`만 F12로 바꾸면 타인 사번으로 위젯 열기 가능 (본 웹 UI와 동일한 수준의 취약점).

## 아키텍처

```
[1] 그룹웨어 사용자 접속
      ↓
[2] JSP 서버(Java)가 session.getAttribute("userId")에서 사번 얻음
      ↓
[3] JSP가 AES-ECB 암호화: "empNo|timestamp_ms" → "fK8ks2Lm..."
      ↓
[4] 렌더링된 HTML:
    <script>
      LucidChat.init({ authToken: 'fK8ks2Lm...', ... });
    </script>
      ↓
[5] 위젯 JS가 iframe src 생성:
    https://lucid.landf.co.kr/embed/gw?token=fK8ks2Lm...&sid=...&gosso=...
      ↓
[6] iframe 내부 GroupwareEmbedPage:
    - URL에서 token 추출
    - EmbedChat에 widgetAuthToken prop 전달
      ↓
[7] useSimpleChat이 백엔드 호출 시 X-Widget-Auth 헤더로 전송
      ↓
[8] backend get_chat_user가 X-Widget-Auth 헤더 감지 →
    widget_auth.get_current_user_widget()이 복호화:
    - AES 복호화 → "empNo|timestamp_ms"
    - 타임스탬프 5분 이내 검증
    - 실패 시 401
      ↓
[9] 인증된 사번으로 Worker 동작 (타인 사번 조작 불가)
```

## 상세 내용

### 1. 신규 Dependency — `widget_auth.py`

기존 `backend/app/utils/crypto.py`의 `decrypt_empno()` 유틸을 재사용(AES ECB + PKCS7). 평문 포맷만 `"empNo|timestamp_ms"`로 확장. 타임스탬프로 replay 방어(기본 5분 유효).

환경변수:
- `WIDGET_SHARED_KEY` — JSP와 동일한 AES 키 (16자 AES-128 권장)
- `WIDGET_TOKEN_VALID_SECONDS` — 기본 300초

### 2. 통합 인증 — `get_chat_user`

```python
async def get_chat_user(
    x_widget_auth: Optional[str] = Header(None),
    auth_token: Optional[str] = Cookie(None),
) -> dict:
    if x_widget_auth:
        return await get_current_user_widget(x_widget_auth)
    user = await get_current_user(auth_token)
    user["source"] = "jwt"
    return user
```

`chat.py`의 7개 엔드포인트 모두 `Depends(get_chat_user)`로 변경. Cookie JWT(본 웹 UI) 또는 Widget Token(iframe 위젯) 둘 다 허용.

### 3. Widget JS — legacy 호환

`buildIframeSrc()` 헬퍼로 URL 생성 로직 일원화. `config.authToken`이 있으면 `?token=` 포함, 없으면 `?empno=` fallback(JSP 업데이트 전 과도기). JSP 업데이트 후엔 항상 `token` 경로 사용.

### 4. JSP 측 Java 코드 (사용자 직접 적용 필요)

```jsp
<%@ page contentType="text/html;charset=UTF-8" language="java" %>
<%@ page import="javax.crypto.Cipher" %>
<%@ page import="javax.crypto.spec.SecretKeySpec" %>
<%@ page import="java.util.Base64" %>
<%@ page import="java.net.URLEncoder" %>
<%!
    private static final String WIDGET_KEY = "landf01234567890";  // .env WIDGET_SHARED_KEY와 동일

    private String generateWidgetToken(String empNo) throws Exception {
        String payload = empNo + "|" + System.currentTimeMillis();
        SecretKeySpec keySpec = new SecretKeySpec(WIDGET_KEY.getBytes("UTF-8"), "AES");
        Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");
        cipher.init(Cipher.ENCRYPT_MODE, keySpec);
        byte[] encrypted = cipher.doFinal(payload.getBytes("UTF-8"));
        return URLEncoder.encode(Base64.getEncoder().encodeToString(encrypted), "UTF-8");
    }
%>
<%
    String empNo = (String) pageContext.findAttribute("userId");
    String widgetToken = (empNo != null) ? generateWidgetToken(empNo) : "";
%>
<script src="https://lucid.landf.co.kr/widget/lucid-chat-widget-gw.js"></script>
<script src="https://lucid.landf.co.kr/widget/lucid-service-menu.js"></script>
<script>
LucidChat.init({
    apiUrl: 'https://lucid.landf.co.kr',
    authToken: '<%= widgetToken %>',
    position: 'bottom-right'
});
LucidServiceMenu.init({
    apiUrl: 'https://lucid.landf.co.kr'
});
</script>
```

파일 위치: `/opt/TerraceTims/web/webmail/webapps/ROOT/WEB-INF/views/webapp/custom_index_header.jsp`

## 방어 메커니즘 요약

| 공격 | 방어 |
|------|------|
| iframe URL `?token=` 복호화 시도 | AES-128 키 없이는 복호화 불가 |
| 탈취한 토큰 재전송 (replay) | 타임스탬프 5분 유효, 만료 시 401 |
| `?empno=` 직접 주입 시도 (legacy) | WIDGET_SHARED_KEY 설정 후엔 무시됨, token 필수 |
| 다른 사번 위조 | Java JSP가 session에서 직접 읽기 — 클라이언트 조작 불가 |

## 호환성 / 배포 순서

과도기 전략: 백엔드는 `token` + `cookie JWT` 둘 다 허용. JSP가 아직 `userId` 평문 방식이어도 legacy `?empno=` fallback으로 동작.

**안전한 배포 순서:**
1. 백엔드 `.env`에 `WIDGET_SHARED_KEY=landf01234567890` 추가 후 재시작
2. 백엔드 배포 (과도기: cookie JWT + widget token + legacy empno 모두 허용)
3. 프론트엔드 배포 (`lucid-chat-widget-gw.js`, embed 페이지 업데이트)
4. 그룹웨어 JSP 업데이트 (선생님 작업)
5. 동작 검증 후 legacy `?empno=` fallback 제거 (1~2주 후)

## 결정 사항 및 주의점

### 결정
- **암호화 유틸 재사용**: 기존 `crypto.py`의 `decrypt_empno()` 그대로 활용. 평문 포맷만 `"empNo|ts"`로 확장.
- **AES-128 ECB 모드**: 기존 Java 그룹웨어 시스템의 암호화 방식과 호환. GCM/CBC로 강화는 향후 과제.
- **타임스탬프 5분 유효**: JSP가 페이지 로드마다 새 토큰 발급하므로 5분이면 충분. 긴 세션은 JSP 리프레시 필요.

### 주의점
- **AES 키 공유**: JSP 소스코드 + 서버 `.env` 모두에 동일 키 하드코딩. 키 변경 시 두 시스템 동시 배포 필요.
- **AES-ECB는 이론적으로 덜 안전**하지만 짧은 단일 블록 평문(사번+타임스탬프 ≈ 24자)이라 패턴 분석 불가.
- **`WIDGET_KEY` 변경 시**: 이전에 발급된 위젯 토큰 모두 무효 → 사용자 페이지 새로고침 필요.
- **Service Menu는 이번 범위에서 제외**. 필요 시 동일 패턴 별도 적용.
