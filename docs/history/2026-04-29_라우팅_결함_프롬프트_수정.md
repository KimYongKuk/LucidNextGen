# 2026-04-29 IT/회계 규정 라우팅 결함 — 프롬프트 두 줄로 수정

## 개요
Eval cron에서 발견된 라우팅 결함("규정"+IT/회계 키워드 결합어가 corp_rag로 잘못 분류) 해결. 키워드 룰 추가 0줄, **두 곳의 LLM 가이드만 수정**해서 분류 책임을 LLM에 그대로 맡기는 방식. Haiku 직접 호출 검증으로 6/6 케이스 의도대로 분류됨을 확인.

## 상세 내용

### 결함 진단

eval cron 로그에서 두 케이스가 corp_rag로 잘못 분류됨을 확인:
- "사내 정보보안 관리 규정 알려줘" → corp_rag (의도: it_support)
- "사내 경비처리 규정 알려줘" → corp_rag (의도: acct_support)

→ CorpRAGWorker는 `search_it_docs`/`search_ac_docs`를 보유하지 않으므로 hr/safety로 폴백 → "찾지 못함" 응답이 사용자에게 노출되는 UX 결함.

### 원인

키워드 룰의 문제가 아니라 **LLM 가이드의 문제**였음:

1. **`state.py:WORKER_CAPABILITIES[Intent.CORP_RAG]`**: `"사내 문서 검색 (인사, 회계, IT, 안전 규정)"` — Planner LLM에 "corp_rag가 IT/회계 규정도 처리한다"고 잘못 알려줌. 실제 코드는 IT/회계는 it_support/acct_support가 통합 보유.
2. **`intent_classifier.py:CLASSIFIER_PROMPT`** corp_rag 줄: `"HR policies, safety guidelines"`로 적혀 있었으나, "policies/regulations" 표현이 광범위해 IT/회계 도메인까지 포함되는 것처럼 LLM이 해석.

### 수정 내용

**state.py**:
```python
# 변경 전
Intent.CORP_RAG: "사내 문서 검색 (인사, 회계, IT, 안전 규정)"

# 변경 후
Intent.CORP_RAG: "사내 인사(HR)·안전환경 규정 검색 — IT 규정은 it_support, 회계 규정은 acct_support가 담당 (corp_rag로 보내지 말 것)"
Intent.IT_SUPPORT: "IT 도메인 통합 (IT 규정 문서 + IT VOC 사례 + 계정/패스워드/메일용량 등 운영) — IT 관련이면 규정·운영 무관 모두 여기로"
Intent.ACCT_SUPPORT: "회계/재경 도메인 통합 (회계 규정 문서 + 회계 VOC 사례) — 회계 관련이면 규정·운영 무관 모두 여기로"
```

**intent_classifier.py**:
```
- corp_rag: HR (인사) and Safety/Environment (안전환경) regulations/policies ONLY, organization chart, ...
  CRITICAL: corp_rag handles **HR + Safety only**. Other domains have dedicated workers:
    - IT/Security regulations → "it_support"
    - Accounting/Finance regulations → "acct_support"
    - Approval form templates → "approval"
    - Bulletin board posts/notices → "board"
  IMPORTANT: "규정/정책/지침" 키워드 단독으로 corp_rag로 보내지 말 것 — 도메인을 먼저 판단하고 매핑.
```

### 검증

`backend/tests/eval/`와는 별도로 Haiku 직접 호출 스크립트로 6 케이스 검증:

| 입력 | 결과 |
|---|---|
| "사내 정보보안 관리 규정" | it_support (+ corp_rag fallback) ✅ |
| "사내 경비처리 규정" | acct_support ✅ |
| "사내 IT 보안 정책" | it_support (+ corp_rag fallback) ✅ |
| "사내 법인카드 사용 규정" | acct_support ✅ |
| "사내 안전관리 규정" | corp_rag ✅ (회귀 없음) |
| "연차 휴가 규정" | corp_rag ✅ (회귀 없음) |

primary intent + fallback 동작도 합리적 (it_support에 IT 규정 문서가 없으면 corp_rag로 자연 폴백).

## 결정 사항 및 주의점

### 키워드 룰 추가 안 함 — LLM 가이드만으로 충분
사용자가 제기한 핵심 인사이트: "규정에 대한 keyword classifier가 꼭 있어야 해? intent를 LLM에 맡길 순 없나?" — 정확한 진단. 키워드 룰은 새 표현 나올 때마다 추가해야 하는 부채. LLM은 이미 도메인을 구분할 능력이 있고, **명확한 가이드**만 주면 됨.

### 적용 시점
운영 슬롯 .env에 EVAL_API_KEY 들어갈 때처럼 즉시 재시작 안 함. 다음 deploy.bat 실행 시 자연 반영. 긴급 변경 아님.

### 복구된 eval 케이스
`it_support_security_regulation`, `acct_support_regulation` 두 케이스를 어제 삭제했다 다시 살림 — 이번 라우팅 수정의 회귀 보호. 다음 cron(2026-04-30 03:00)부터 자동 검증.

### 후속 모니터링 포인트
- corp_rag로 분류되어야 할 입력(HR/안전)이 it_support/acct_support로 잘못 가는지 (반대 방향 회귀)
- "정책" / "지침" / "방침" 등 유사어도 정상 분류되는지
- Planner 경로의 분류도 동일하게 정정됐는지 (state.py WORKER_CAPABILITIES가 Planner의 worker_catalog 소스라 자연 반영됨)
