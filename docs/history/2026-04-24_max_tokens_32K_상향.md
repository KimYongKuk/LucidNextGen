# 2026-04-24 max_tokens 32K 상향

## 개요
A2203003 사용자의 "엘앤에프 전체 부서 조직도" 요청이 Sonnet Worker의 `max_tokens=8192`에 걸려 마크다운 계층 트리 출력이 중간에 잘리는 현상이 운영 로그에서 확인됨 (2026-04-22 17:44, CorpRAGWorker `output=8,488` 토큰으로 절단). 150~200개 부서를 계층 마크다운으로 출력하면 8K로 부족. Sonnet 4.5/4.6 API 상한인 64K 중 표준값인 **32K(32768)**로 일괄 상향.

## 상세 내용

### 상향 기준
- **Sonnet 4.5/4.6**: Bedrock API 기준 output 최대 64K, 표준 운영 권장치 32K 선택
- **Haiku 4.5**: output 최대 8K (모델 상한), 4096 → 8192로 상향
- Synthesizer는 Haiku 기반이라 8K가 한계

### 부작용 관리
1. **응답 시간**: output이 늘면 스트리밍 시간 선형 증가. 다만 Bedrock read_timeout은 **청크 간 간격**이므로 전체 시간엔 영향 없음. 클라이언트 측(nginx/ALB/브라우저) 타임아웃은 streaming SSE라 무관.
2. **비용**: Sonnet output 단가가 높아 장문 요청 시 건당 비용 최대 4배 증가 가능. 모니터링 대상은 `[TOKEN_LOG]` / `[CHAT_STREAM] Token usage`.
3. **Synthesizer 절단**: Worker가 32K를 뱉어도 Synthesizer가 8K로 재합성하는 구조. 긴 표 passthrough 시 여전히 잘릴 수 있음 → Synthesizer 프롬프트가 "압축 재정리" 지향이라 일반적으로 8K에 수렴하지만, 추후 longrun 케이스 발견 시 재검토.

### 전파 경로
- `ModelConfig.max_tokens` → [base_worker.py:1200](../../backend/app/agents/workers/base_worker.py#L1200) `ChatBedrockConverse(max_tokens=config.max_tokens)` → 모든 Worker 공통
- `openapi_bedrock_service.py` → OpenAI-compatible `/v1/chat/completions` 엔드포인트 (외부 API 호출용, 별도 IAM)
- `synthesizer.py` → A2A 파이프라인 최종 응답 합성

## 결정 사항 및 주의점
- **단계적 접근**이었으나 사용자 요청으로 16K 스킵하고 32K로 직행.
- Haiku(fallback/synthesizer)는 모델 상한이 8K라 32K 적용 불가. Worker가 Sonnet→Haiku로 폴백되는 케이스에서는 **암묵적으로 8K로 다시 축소됨** (리전 폴백 상황이 발생하면 긴 응답은 재절단 가능).
- 64K로 추가 상향 시에는 Bedrock 클라이언트 `read_timeout` (base_worker의 `BEDROCK_CONFIG.read_timeout=120`)도 함께 상향 필요.
- **근본 해결책은 여전히 Planner 단계에서 대형 리스트를 xlsx 경로로 유도하는 것.** 이번 상향은 안전망 강화 성격.
- 배포 후 검증: A2203003의 부서 조직도 질의 재현, `output=8,192`로 정확히 잘리는 패턴이 사라지는지 확인.
