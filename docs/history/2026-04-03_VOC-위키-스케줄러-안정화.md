# 2026-04-03 VOC 위키 스케줄러 안정화

## 개요
APScheduler로 등록된 VOC→L&F Wiki 자동 동기화 작업이 서버 시작 시 이벤트 루프를 블로킹하여 전체 서버가 프리즈되는 문제를 수정하고, 동기화 누락 방지 로직을 개선했다.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/services/voc_wiki_service.py | 수정 | boto3/MySQL 동기 호출을 ThreadPoolExecutor로 격리, since 로직 오버랩 적용 |
| backend/app/utils/voc_wiki_scheduler.py | 수정 | misfire_grace_time 축소, _execute try/except 추가 |

## 상세 내용

### 1. 이벤트 루프 블로킹 해결

**원인**: `generate_text_haiku`가 `async def`이지만 내부에서 동기 boto3 `invoke_model()`을 호출. 일반 채팅에서는 FastAPI 워커 체인에서 실행되어 문제없지만, APScheduler의 async job은 이벤트 루프에서 직접 실행 → boto3 HTTP 요청이 이벤트 루프 전체를 블로킹 → 서버 프리즈 (Ctrl+C도 불가).

**수정**:
- `_call_haiku_safe()`: boto3 호출을 `ThreadPoolExecutor`에서 실행하는 래퍼 메서드 추가
- `_get_last_sync_date()` / `_save_sync_log()`: 동기 MySQL PooledDB 호출도 `run_in_executor`로 격리
- `asyncpg.create_pool()`: `asyncio.wait_for(timeout=15)` 추가 (DB 연결 실패 시 무한대기 방지)
- `ThreadPoolExecutor(max_workers=3, thread_name_prefix="voc_wiki")`: 전용 스레드 풀

### 2. APScheduler misfire 방지

- `misfire_grace_time`: 3600 → 60 (서버 재시작 시 놓친 스케줄 실행 방지)
- `_execute()`에 try/except 추가 (예외 발생해도 서버 프로세스 보호)

### 3. since 로직 개선 (누락 방지)

**기존**: `since = last_sync_date + 1일` → 마지막 동기화 다음날부터만 조회
- 문제: 동기화 중간 크래시 시 당일 미처리 VOC 영구 누락

**변경**: `since = last_sync_date` (1일 오버랩)
- 마지막 동기화 당일부터 재조회
- LLM 병합 프롬프트가 중복 자동 처리 ("동일 증상+해결 → 참고 날짜만 추가")
- 서버가 며칠 죽었다가 살아나도 자동 복구

### 4. 과거 누락분 백필

- 3/1~4/3 전량 447건 재처리 완료
- 결과: 4건 신규 생성 + 66건 업데이트 (기존 초기 적재에서 누락된 건 복구 확인)

## 결정 사항 및 주의점
- `_call_haiku_safe`는 VocWikiService 전용. 다른 서비스의 `generate_text_haiku`는 FastAPI 워커 체인에서 실행되므로 별도 처리 불필요.
- 개발서버에서 스케줄러를 끄려면 `.env`에 `VOC_WIKI_SYNC_ENABLED=false` 설정.
- 1일 오버랩으로 인한 LLM 호출 증가는 미미 (일 10~30건 재처리, Haiku 비용).
