# 2026-04-14 캘린더 사용자별 SSO 인증

## 개요
캘린더 일정 등록/수정/삭제 시 서비스 계정(wg0403) 대신 사용자 본인의 LFON GOSSOcookie를 사용하도록 전환.
기존에는 모든 사용자의 캘린더 API 호출이 단일 서비스 계정으로 이루어져, 타 사용자의 캘린더에 일정 등록 시 권한 오류가 발생했음.

## 상세 내용

### 데이터 흐름
```
[LFON JSP] GOSSOcookie 읽기 → AES 암호화
  → lucid.landf.co.kr/?empno=<enc>&gosso=<enc>
  → [middleware.ts] gosso 복호화 → gosso 쿠키 저장
  → [route.ts] gosso 쿠키 → backend body
  → [chat.py] ChatRequest → RequestContext
  → [a2a_streaming.py] context["gosso_cookie"]
  → [CalendarWorker.prepare_tools()] 도구 args에 주입
  → [calendar_mcp_server.py] 사용자 쿠키로 API 호출, 실패 시 서비스 계정 폴백
```

### _api_request 리팩토링
기존 단일 `_api_request` 함수를 2계층으로 분리:
- `_api_request_with_cookies()`: 지정 쿠키로 API 호출 (저수준)
- `_api_request()`: gosso_cookie 우선 사용, 401/403 시 서비스 계정 폴백 (고수준)

### 폴백 전략
| 상황 | 동작 |
|------|------|
| gosso_cookie 있고 유효 | 사용자 본인 세션으로 API 호출 |
| gosso_cookie 있지만 만료 | 401 → 서비스 계정(wg0403) 폴백 |
| gosso_cookie 없음 | 기존과 동일하게 서비스 계정 사용 |

### GOSSOcookie 특성 (테스트 확인)
- API 호출 시 쿠키 값 로테이션 없음 (안정적)
- 재로그인해도 이전 쿠키 즉시 무효화되지 않음
- LFON 세션이 살아있는 동안 유효

## 결정 사항 및 주의점
- **JSP는 별도 배포 필요**: LFON 서버 관리자에게 `docs/jsp_sso_gosso_patch.md` 전달
- **Lucid 먼저 배포, JSP는 나중에**: gosso 파라미터 없이도 기존대로 동작하므로 순서 무관
- **위젯은 대상 아님**: 위젯은 간단 채팅용이라 캘린더 기능 미포함
- **reservation, works_it MCP는 미적용**: 캘린더만 우선 적용, 필요 시 동일 패턴으로 확장 가능
