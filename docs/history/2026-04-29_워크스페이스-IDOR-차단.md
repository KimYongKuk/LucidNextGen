# 2026-04-29 워크스페이스 IDOR 차단 (1·2단계)

## 개요
보안기술팀(A2310009)이 `POST /v1/workspaces`에서 클라이언트가 보낸 `user_id`를 검증 없이 owner로 INSERT하는 취약점을 이용해 운영자(A2304013) 사이드바에 위조 워크스페이스를 주입하고, `chat.py`의 워크스페이스 컨텍스트 진입 시 owner 검증 부재를 함께 이용해 채팅 흐름 진입까지 성공시킨 사건의 후속 패치.

- **1단계**: 워크스페이스/채팅 세션 owner 검증을 인증된 JWT 사번 기반으로 일원화 + 공통 헬퍼 도입(직접 차단 대상)
- **2단계**: 동일 패턴이 잔존하던 board/feedback/report/upload 라우트 일괄 정리 + 운영자(admin) 라우트 권한 검증 도입(잠재 IDOR 근절)

## 상세 내용

### 공통 헬퍼 (`authz.py`)
- `_assert_owner(resource, empno, *, resource_type, resource_id_field, owner_field, public_field)` — 코어
- `assert_workspace_owner(workspace, empno, *, allow_public=True)` — 조회/사용용은 `allow_public=True`, 수정/삭제/업로드는 `allow_public=False`
- `assert_chat_session_owner(session, empno)` — 세션은 public 개념 없음
- 거부 시 `[SECURITY] {resource_type}_access_denied` 마커로 stdout 로깅 → 향후 보안 이벤트 적재 시 단일 지점에서 추가 가능

### `workspace.py` 8개 라우트 변경
1. `GET /v1/workspaces` — `user_id` Query 제거, JWT 인증 사번으로 본인 목록 반환
2. `POST /v1/workspaces` — body `user_id` 필드 제거(클라이언트 신뢰 금지), owner는 `current_user["empno"]`로 강제, `_is_operator()` 검사도 인증 사번 기준
3. `GET /v1/workspaces/{uuid}` — `assert_workspace_owner(allow_public=True)`
4. `PUT /v1/workspaces/{uuid}` — `assert_workspace_owner(allow_public=False)` + 운영자 검사
5. `DELETE /v1/workspaces/{uuid}` — `assert_workspace_owner(allow_public=False)`
6. `POST /v1/workspaces/{uuid}/upload` — `assert_workspace_owner(allow_public=False)` (업로드는 owner만)
7. `GET /v1/workspaces/{uuid}/files` — `assert_workspace_owner(allow_public=True)`
8. `DELETE /v1/workspaces/{uuid}/files/{file_id}` — `assert_workspace_owner(allow_public=False)`

### `chat.py` 변경
- `chat.py:639` (A2A 경로) / `chat.py:765` (Legacy 경로) — `workspace_data = service.get_workspace_by_uuid(...)` 직후 `assert_workspace_owner(workspace_data, authenticated_empno)` 호출
- `chat.py:1105` (`GET /v1/chat/sessions`) — `workspace_id` Query 주어진 경우 워크스페이스 owner 검증 추가
- `chat.py:1191` (PATCH 세션) / `chat.py:1264` (DELETE 세션) / `chat.py:1300` (POST pin) — 기존 inline owner check를 `assert_chat_session_owner`로 통합

### Pydantic 모델
- `WorkspaceCreate`에서 `user_id` 필드 삭제 → 클라이언트가 보내도 무시(호환성 유지). 추후 `extra="forbid"` 추가는 별도 PR.

## 결정 사항 및 주의점

- **위젯 호환성**: 워크스페이스 API는 위젯에서 호출하지 않음을 확인하고 `get_current_user`(JWT 쿠키) 단일 인증만 적용. 채팅 라우트는 기존 `get_chat_user`(쿠키 + X-Widget-Auth dual-auth) 유지.
- **프론트엔드 정리(`?user_id=...` 쿼리/`user_id` body 제거)는 별도 PR**: 백엔드가 user_id Query/Body를 무시하도록 변경되어 호환성 문제는 없음. 깔끔한 정리는 2단계 라우터 글로벌 dependency 적용 시 일괄 처리.
- **이후 작업으로 이연 (3단계 — 현재 홀딩)**:
  - `workspaces` 테이블에 `created_by_authenticated_empno`/`created_by_ip` 컬럼 추가(추적성).
  - `[SECURITY]` 로그 라인을 `user_security_events` 테이블에 적재 + 임계치 초과 시 메일 알림.
  - 다운로드 라우트(`/v1/{pdf,image,ppt,docx,xlsx}/download/...`) path traversal 및 사번 일치 검증 점검.
  - CI 가드: `user_id.*Query`/`user_id.*Form`/Pydantic body의 `user_id: str` 패턴 차단 스크립트.
- **프론트엔드 호출 사이트 정리(별도 PR)**: `frontend/lib/api/workspaces.ts`, `frontend/lib/api/notifications.ts`, 그리고 `?user_id=...` 직접 fetch 호출 18곳 — 백엔드가 user_id Query/Body를 무시하므로 호환성 OK이나 정합성 위해 정리.
- **삭제는 hard delete 유지**: `chat_sessions`/`chat_log_new`/`workspaces` 모두 물리 삭제. 사후 추적 필요 시 별도 audit 테이블/트리거 도입(별도 작업).
