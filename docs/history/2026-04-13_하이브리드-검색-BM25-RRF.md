# 2026-04-13 하이브리드 검색 (BM25 + Semantic RRF)

## 개요
시멘틱 서치만으로는 Lot 번호, 코드명 등 정확한 키워드 검색이 실패하는 문제를 BM25 키워드 검색을 병행하여 해결. Reciprocal Rank Fusion (RRF)으로 두 결과를 합산하는 하이브리드 검색 도입.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| `backend/app/services/hybrid_search_service.py` | 신규 | BM25 인덱스, 캐시 매니저, RRF 합산, 토크나이저 |
| `backend/app/services/chromadb_service.py` | 수정 | `_sync_search_collection()`에 hybrid/semantic/bm25 모드 추가, `search()`에 search_mode 파라미터 추가, 업로드/삭제 시 BM25 캐시 무효화 |
| `backend/requirements.txt` | 수정 | `rank-bm25==0.2.2` 추가 |

## 상세 내용

### 문제 상황
- "PP2509-138" Lot 번호로 검색 시, 시멘틱 서치 top 5 결과에 해당 키워드가 전혀 포함되지 않음
- 원인: 임베딩 모델이 `PP2509-129`, `PP2509-138` 등 유사 코드를 의미적으로 구별하지 못함
- fulltext에는 4개 청크에 해당 데이터가 존재하나, 시멘틱 유사도 순위에서 밀려남

### 해결 아키텍처
```
쿼리 ─┬─→ ChromaDB 시멘틱 검색 (top limit*3)  ─┐
      └─→ BM25 키워드 검색 (top limit*3)       ─┤
                                                 ↓
                                    RRF 점수 합산 → top limit 반환
```

### hybrid_search_service.py 구성
- **tokenize()**: 정규식 기반 토크나이저. `PP2509-138` 같은 코드를 통째로 토큰으로 보존
- **BM25Index**: rank_bm25 (BM25Okapi) 래퍼. 문서 토큰화 + 검색
- **BM25CacheManager**: 스레드 안전한 싱글턴 캐시. `collection_name + count` 기반 자동 무효화, TTL 5분
- **reciprocal_rank_fusion()**: 시멘틱 + BM25 결과를 RRF로 합산. min_relevance 필터 호환

### RRF 가중치
- 기본값: semantic 0.5 / BM25 0.5
- 환경변수로 오버라이드 가능: `HYBRID_SEMANTIC_WEIGHT`, `HYBRID_BM25_WEIGHT`
- RRF 상수 k=60 (환경변수: `HYBRID_RRF_K`)

### search_mode 파라미터
- `"hybrid"` (기본): RRF 합산 — 기존 호출 코드 변경 불필요
- `"semantic"`: 기존과 동일한 시멘틱 전용 검색
- `"bm25"`: 키워드 전용 검색

### min_relevance 호환
- 시멘틱에서 온 결과: 기존처럼 cosine similarity로 필터링
- BM25에서만 온 결과: 키워드 정확 매칭이므로 필터 통과 (핵심 설계)

### BM25 캐시 무효화
- `upload_file()` 완료 후: `get_bm25_cache().invalidate(collection_obj.name)`
- `delete_session_files()`: `get_bm25_cache().invalidate(f"session_{session_id}")`

## 결정 사항 및 주의점
- 한국어 형태소 분석기(mecab) 대신 정규식 토크나이저 사용: Windows 호환성 우선
- 자연어 검색 품질 저하 없음 확인 ("잔류 리튬 영향 분석" 쿼리에서 기존과 동일한 결과)
- 향후 가중치 튜닝은 운영 로그 기반으로 진행
- `rag_server.py` 등 기존 호출 코드 변경 없이 자동 적용
