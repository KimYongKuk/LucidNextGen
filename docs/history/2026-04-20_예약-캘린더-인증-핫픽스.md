## 각 수정 상세

### 1. `81070e3` — 예약 도구 gosso_cookie 주입 제외 ✅

**증상**: A2304013 실사용자 예약 시도 중 `create_reservation` ToolException —
`unexpected_keyword_argument`. pydantic validation error.

**원인**: `calendar_worker.prepare_tools`가 `create_reservation`/`get_my_reservations`/
`cancel_reservation`에 보안 래핑으로 `gosso_cookie`를 주입. 하지만 **해당 예약 도구들의
MCP 스키마에는 gosso_cookie 필드가 없음** (캘린더 도구와 달리). 이전엔 context에
gosso_cookie가 없어 주입이 skip되었으나, `PLANNER_ENABLED=true` 후 gosso가 정상 전달되며
잠재 버그가 발현.

**수정**: `secured_tools`를 두 그룹으로 분리
- `calendar_gosso_tools`: gosso 주입 O (create_event, update_event 등)
- `reservation_no_gosso_tools`: gosso 주입 X (create_reservation 등)

LLM이 실수로 gosso_cookie 넘겨도 래퍼가 `input_data["args"].pop("gosso_cookie")`로 제거.

### 2. `3bc90cc` — 예약 취소 백엔드 소유자 검증 ✅

**취약점**: `cancel_reservation`이 기존엔 프롬프트 방어만 의존. 서비스 계정으로
DELETE 호출하므로 LLM이 프롬프트 지시를 무시하면 타인 예약도 취소 가능.

**수정**: `reservation_mcp_server.py`에 백엔드 레벨 검증 추가
- `_fetch_my_reservations_raw()`: get_my_reservations 로직 공통 헬퍼로 추출
- `_get_my_reservation_ids()`: 내 예약 ID 집합 반환 (60초 TTL 캐시)
- `cancel_reservation`: DELETE 전 `reservation_id in my_ids` 검증, 아니면 거부
- `create_reservation`/`cancel_reservation` 성공 시 캐시 invalidate

stderr audit log 추가: 거부 시 `[Reservation MCP] 🚫 소유자 검증 실패: ...`

### 3. `e29c857` — typing import 누락 ✅

**증상**: `3bc90cc` 배포 후 blue/green에서 reservation_server MCP 스폰 실패. 로그:
```
NameError: name 'Dict' is not defined. Did you mean: 'dict'?
```

**원인**: `_my_reservations_cache: Dict[str, tuple]` 추가했으나
`from typing import Optional`만 있고 Dict/List 빠짐.

**수정**: `from typing import Optional, Dict, List`로 보강.

추가로 blue venv에 `asyncpg` 누락되어 있어 수동 `pip install asyncpg==0.31.0` 실행
(배포 프로세스의 requirements 설치 step 누락 의심, 별도 조사 필요).

### 4. `fae8d78` → `b3c457a` — 쓰기 폴백 차단 시도 및 리버트 ⚠️

**배경**: 실사용자 최지원(A2604011)이 본인 캘린더(id=3991)에 일정 등록 시도 → 실패.
로그:
```
[Calendar MCP] 일정 등록: cal=3991, user=GO#2011, gosso=있음
[Calendar MCP] 사용자 GOSSOcookie 만료/실패 → 서비스 계정 폴백: POST
→ None (LFON 거부)
→ "서버에 연결할 수 없습니다" misleading 에러
```

**추측 기반 진단 (잘못됨)**: gosso 만료 → service account 폴백이 타인 캘린더 쓰기
거부 → 우리 코드는 네트워크 에러처럼 표시.

**시도한 수정 (fae8d78)**:
- `_api_request`에 `is_write` 검사 추가
- 쓰기(POST/PUT/DELETE)는 서비스 계정 폴백 금지
- `{"__auth_expired__": True}` sentinel로 "세션 만료" 메시지 반환

**발견된 문제**: 리버트 직전 확인 — 이전에 정상 작동하던 케이스에서도 차단됨.
**LFON이 서비스 계정의 본인 캘린더 쓰기를 어느 정도 허용**했던 것으로 보이며,
내 수정이 그 허용 경로까지 막음.

**리버트 (b3c457a)**: fae8d78 전체 원복. 에러 메시지는 여전히 "서버에 연결할 수 없습니다"
(misleading)이지만 **정상 케이스 복원 우선**.

**교훈**: HTTP status code / response body 미확인 상태에서 추측 기반 수정은 금지.
재발 시 `_api_request_with_cookies`의 `(401, 403)` 분기에 response body 로깅 추가
후 근본 원인 확정하는 게 우선.

### 5. `2585ca9` — 위젯-gw GOSSOcookie URL 전달 △

**의도**: 그룹웨어 임베드 위젯에서 LFON `GOSSOcookie`를 추출해 iframe URL에
`&gosso=...` 파라미터로 전달. 임베드 사용자도 쓰기 인증 받도록.

**미해결 이슈**: iframe이 로드하는 `/embed/gw/page.tsx`는 URL의 `gosso` 파라미터를
읽지 않음. middleware도 `/embed/*` 경로 제외. 결과적으로 **URL에 gosso 붙지만
아무도 소비하지 않음**.

**왜 위젯이 원래 작동했는가 (추정)**:
- 과거에 외부 SSO 링크(`?empno=X&gosso=Y`)로 한 번이라도 접속한 사용자 →
  middleware가 `lucid.landf.co.kr` 도메인에 `gosso` 쿠키 24h 저장 →
  동일 도메인의 위젯 iframe이 그 쿠키 재사용 → 정상 작동.
- 이 흐름이 깨지는 순간 (한 번도 외부 링크 안 거친 사용자) → 위젯에선 쓰기 불가.

**유지 이유**: 무해함 (URL 파라미터만 추가). 추후 `embed/gw/page.tsx`에서
URL `gosso` 읽어 쿠키 세팅하면 완성될 수 있음.

### 6. `86650fb` — use-simple-chat GOSSOcookie regex 추가 △

**의도**: 현재 프론트 regex `/gosso=([^;]+)/`는 case-sensitive라
LFON 원본 `GOSSOcookie`(대문자)를 매치 못 함. middleware가 세팅한
소문자 `gosso` 쿠키에만 의존.

**수정**: `GOSSOcookie` 우선 매치 → 없으면 `gosso` 폴백.

**효과 조건**: LFON이 `GOSSOcookie`를 `.landf.co.kr` 도메인에 세팅해야 iframe에서
직접 읽힘. 조사 미완료 (도메인 설정 확인 필요).

**무해성**: fallback 체인으로 추가하므로 기존 흐름은 그대로.
