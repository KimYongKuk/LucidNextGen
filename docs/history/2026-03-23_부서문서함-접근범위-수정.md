# 2026-03-23 부서 문서함 접근 범위 수정

## 개요
ApprovalWorker가 부서 문서함(수신/참조/완료) 조회 시 사용자의 직속 dept_id 하나만 검색하던 문제를 수정하여, `v_appr_user_accessible_depts` JOIN을 통해 접근 가능한 모든 부서(소속+담당자 지정)를 검색하도록 변경. MCP 서버의 SQL 검증도 함께 수정.

## 변경 파일 요약
| 파일 | 변경 유형 | 설명 |
|------|-----------|------|
| backend/app/agents/workers/approval_worker.py | 수정 | 시스템 프롬프트 - 부서 뷰 조회 시 JOIN 패턴 강제 |
| backend/app/mcp_servers/approval_mcp_server.py | 수정 | ALLOWED_VIEWS에 accessible_depts 추가, auth 검증에 employee_number 추가 |

## 상세 내용

### 문제 현상
- "내 수신문서함의 '[IT운영팀 신원철] 3/23~ 기타시스템 신규의 건' (문서번호: 1171249)" 조회 시
- DA파트(사용자 직속 부서)에서만 검색 → 문서 없음 → 다른 뷰 탐색 → 최종 실패 (8회 도구 호출)
- 실제로는 IT운영팀 수신문서함에 존재하며, 사용자가 접근 가능

### 원인 (2가지)

#### 원인 1: MCP 서버 SQL 검증 거부
- **ALLOWED_VIEWS**에 `V_APPR_USER_ACCESSIBLE_DEPTS` 미포함 → 단독 쿼리 거부
- **Auth 검증(검증 3)**: `employee_number`를 인증 식별자로 인식 못함
  - 기존: login_id, user_id, dept_id만 체크
  - JOIN 쿼리 `WHERE a.employee_number = 'A2304013'`에서 'wg0403'(login_id)도 없고, 1367(user_id)도 없고, 507(dept_id)도 없음 → **검증 실패**
  - → 첫 번째 올바른 JOIN 쿼리가 auth 에러로 거부됨

#### 원인 2: 시스템 프롬프트 지시 오류
- WORKFLOW 섹션: "부서 뷰 (v_appr_dept_*): dept_id 사용" → 단일 dept_id만 사용 유도
- CRITICAL RULES 2번: "login_id/user_id/dept_id를 그대로 사용하세요"
- auth 검증 실패 후 LLM이 `dept_id = 507`을 추가 → DA파트에서만 검색

### 수정 내용

#### MCP 서버 (approval_mcp_server.py)
1. **ALLOWED_VIEWS**: `V_APPR_USER_ACCESSIBLE_DEPTS` 추가
2. **Auth 검증 (검증 3)**: `employee_number`를 4번째 인증 식별자로 추가
   ```python
   has_auth_id = (
       f"'{auth_login_id}'" in query_str
       or f"= {auth_user_id}" in query_str
       or f"= {auth_dept_id}" in query_str
       or f"'{employee_number}'" in query_str  # NEW
   )
   ```
3. **검증 실패 메시지**: employee_number 안내 및 부서 뷰 JOIN 예시 추가

#### 시스템 프롬프트 (approval_worker.py)
1. **WORKFLOW**: 부서 뷰 조회 시 JOIN 패턴 필수 지정 + SQL 예시
2. **CRITICAL RULES 2번**: `dept_id` → `employee_number` + JOIN 접근 안내
3. **SQL QUERY RULES 3번**: 필터링 옵션에 `employee_number` 추가
4. **TOOL SELECTION GUIDE**: 부서 뷰 5개 항목 모두 JOIN 패턴으로 변경
5. **수신문서함 단건 조회**: `WHERE doc_id = N` → `JOIN ... WHERE a.employee_number AND r.doc_id = N`

## 결정 사항 및 주의점
- employee_number는 `prepare_tools()` 보안 래핑으로 강제 주입되므로 사용자 변조 불가
- 부서 뷰 조회 시 항상 JOIN이 추가되어 쿼리 복잡도 약간 증가하나, 인덱스 있어 성능 영향 미미
- `v_appr_user_accessible_depts`는 DBA가 이미 생성한 VIEW로 추가 작업 불필요
