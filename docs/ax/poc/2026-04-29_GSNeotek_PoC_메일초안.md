제목: [LF / Lucid AI] 오픈소스 LLM PoC 협의 — 시나리오 및 사용량 수준 공유

안녕하세요. LF AX팀 김용국입니다.

지난 미팅에서 말씀 주신 오픈소스 모델 PoC 지원 건 관련하여,
저희 쪽 테스트 시나리오와 사용량 수준을 러프하게 정리해 드립니다.

────────────────────────────────────────
1. PoC 목표
────────────────────────────────────────
현재 운영 중인 사내 챗봇 "Lucid AI"는 AWS Bedrock 기반으로
- 빠른 분류·요약 작업: Claude Haiku
- 본문 응답·도구 호출: Claude Sonnet
의 2-tier 구조로 동작 중입니다.

이번 PoC에서는 **Haiku 트래픽을 오픈소스 모델로 우선 대체**하는 것을 1차 목표로,
가능하다면 일부 시나리오에서 **Sonnet 대체(고급 추론 영역)** 도 함께 검토하려 합니다.

검토 후보 모델(예시):
- DeepSeek V4
- GPT-OSS 120B
- Qwen3 / Llama 3.3 등 한국어 성능이 검증된 모델

저희 쪽에서 AWS EC2 GPU 인스턴스를 직접 띄워 테스트할 계획이며,
GS네오텍에서 모델 선정·서빙(vLLM/TGI 등) 구성·튜닝 측면에서 도움을 주시면 좋겠습니다.

────────────────────────────────────────
2. 테스트 시나리오 (대분류)
────────────────────────────────────────
챗봇은 사용자 발화를 분류하여 적절한 워커가 처리하는 구조이며,
PoC 평가 대상 시나리오는 크게 아래와 같습니다.

(A) 단순·고빈도 작업 — Haiku 대체 검증
   - 사용자 의도 분류 (한국어 + 시스템명 혼재, JSON 출력)
   - 대화 요약 / 핵심 정보 추출
   - 짧은 제목 생성 등

(B) 일반 응답·검색 — Sonnet 비교 검토
   - 일반 대화 / 코딩 / 번역 / 한국어 비즈니스 작문
   - 웹 검색 결과 종합

(C) 도구 호출 워커 — Sonnet 안정성 비교
   - 사내 문서 RAG 응답
   - 메일 조회·요약·답장 초안
   - 엑셀/PPT 생성 (도구 호출 루프 다수)
   - 사내 시스템 SQL 생성·조회

평가 시 보고자 하는 지표:
   - 정확도 (정답셋 매칭 / 인간 평가)
   - 한국어 자연성
   - TTFT(첫 토큰 응답 시간) / 출력 속도
   - 도구 호출 / JSON 포맷 준수율
   - GPU VRAM 점유 / 동시 처리량

────────────────────────────────────────
3. 사용량 수준 (러프 가이드)
────────────────────────────────────────
- 월간 활성 사용자(MAU): **약 400명대**
- 영업일 기준 일평균 메시지: **500~850건**
- 피크일 메시지: **800건대**
- 평균 사용자 입력 길이: **200자 내외** (긴 케이스는 첨부 컨텍스트 포함 수만 자)
- 평균 응답 토큰: **1.5K 토큰 내외** (PPT/엑셀 생성은 더 길어짐)
- 분당 최대 동시 호출: **20건 미만** / 동시 활성 사용자 **10명 미만**

요약하면 **"분당 30 RPM·동시 10 user 처리 가능하면 운영 충분"** 수준이며,
추후 사용자 확장을 고려해 **2배 여유(60 RPM)** 정도 보고 있습니다.

컨텍스트 윈도우는 **128K 이상** 지원되는 모델을 선호합니다
(긴 메일/문서 분석 시나리오에서 일부 100K를 넘기는 경우 발생).

────────────────────────────────────────

────────────────────────────────────────
